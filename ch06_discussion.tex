\chapter{Discussion} \label{chapter:ch6_discussion}

This chapter returns to the question posed in the introduction: whether frequently changing collaborative-robot tasks can be programmed with less friction by specifying spatial intent directly in the real workspace. The goal here is not to repeat the implementation chapter, but to discuss what the built system actually shows, where it is useful, and where its current limits lie.

\section{What the thesis built}

The main result of this thesis is an end-to-end system for \emph{in-workspace spatial authoring} of selected robot tasks. The system combines workspace sensing, spatial input, task-specific interpretation, and robot execution behind a shared modular runtime with explicit interfaces. In practical terms, this means that the thesis did not only explore a single pen-based interaction technique. It produced a complete prototype workflow that covers graph construction, activation, calibration, authoring, confirmation, execution, visualization, and persistence, as described in \autoref{sec:ch3_ui_orchestration} and \autoref{chapter:ch5_user_workflow}.

At the same time, the thesis deliberately stayed within a narrower scope than a full robot programming environment. As stated already in the introduction, the goal was not to design a general-purpose language with branching, loops, exception recovery, or rich sensor-driven logic. Instead, the system focuses on \emph{task-oriented use cases} implemented as plugins on top of shared sensing and execution infrastructure. This choice is important for the interpretation of the results: the thesis is not evaluated by how well it replaces every existing robot-programming workflow, but by whether it provides a useful way of entering selected robot tasks through spatial input.

The two main use cases, pick-and-place and seam welding, are therefore more than simple demonstrations. They show that the same runtime can support different task structures while reusing the same architectural backbone. Pick-and-place depends on scene refresh at execution time and therefore tests whether the system can remain responsive to the current state of the workspace. Seam welding resolves the task during authoring and tests whether rough spatial input can be turned into a usable trajectory-centric procedure. Together, these use cases show that the thesis result is both a concrete prototype and a reusable framework for further task-oriented development.

\section{What the results mean}

In many existing deployments, a collaborative robot is integrated for one specific task and that task then remains largely fixed. The presented system explores a different approach. Here, the use case is prepared once, but the operator can redefine the concrete task instance quickly within that use case through spatial input. This shifts reprogramming away from repeated low-level editing through teach pendant programming, offline programming, or hand-guiding, and toward fast re-authoring inside a narrow task-specific interface.

A separate result of the thesis is the modular architecture, which follows from the use-case plugin requirement and from the analysis choice to keep component alternatives open. The shared runtime separates sensing, the operator-facing control interface, 3D scene presentation, robot execution, and use-case logic through explicit module boundaries. As a result, the two use cases are not two unrelated demos, but two concrete task definitions built on the same infrastructure. This is important for interpreting the result: the thesis does not only show that two example tasks can be made to work, but that one system can support different task structures while keeping the task-specific logic inside the use case.

The two use cases reveal different strengths. Pick-and-place is important because it shows that the system is not limited to fully fixed commands. The command is authored once, but part of its execution is recomputed from the current scene. This is a positive result for the overall architecture: it demonstrates that the system can combine authored intent with runtime scene information instead of only replaying frozen trajectories. At the same time, pick-and-place does not appear to be the strongest practical target for the current prototype. In many industrial settings, material-handling tasks are highly specific and are usually integrated together with conveyors, grippers, and dedicated scene-sensing components. In such cases, a specialized integration may remain the more appropriate solution.

Seam welding is the clearest practical fit for the thesis approach. The task is naturally trajectory-centric, the user intent is spatially meaningful, and the resulting program structure is simple enough that the use case can infer the final robot procedure from the authored trajectory and the sensed scene. More importantly, this type of task matches the situation that motivated the thesis in the first place: work in which the programming effort can become significant because the welds have to be re-taught repeatedly. For fully repetitive welding in a fixed production setup, the advantage of this approach is limited because the task can be integrated once and then reused. In small-batch production, the advantage becomes more meaningful because the same type of operation must be reprogrammed repeatedly. The most interesting case is one-off or highly variable welding, where a human could still perform the task manually, but a sufficiently capable version of this approach could make robot execution practical by lowering the cost of programming. In that setting, the workflow of \emph{prepare the workpiece, indicate the intended weld, confirm the result, and run the robot} is not only more direct, but points toward a form of human-robot cooperation in which the human defines the task in context and the robot performs the consistent physical execution.

These results also clarify where the approach is useful. It is most relevant when the physical operation is repetitive enough that robot execution is worthwhile, but the concrete task changes often enough that repeated low-level reprogramming becomes the dominant cost. This is why the thesis is more convincing for frequently re-taught welding-like tasks than for highly specific material-handling cells. The value lies less in replacing expert robot programming everywhere than in giving the user a fast way to redefine a known task inside a prepared use case.

The results also raise a question about how narrowly or broadly a use case should be defined. The strongest case is a highly specialized use case prepared for one customer or one recurring workflow. In that case, the interface can closely match the task and ask the operator only for the spatial inputs that vary from one task instance to the next. A more generic use case may be shared across multiple deployments, which reduces the need to develop a separate task definition for each installation and may reduce the need for a skilled integrator to prepare a specialized use case for each one. The trade-off is that such a use case must expose more parameters and rely on a less direct mapping from user intent to execution, so the interface is less closely matched to the specific task. The thesis does not resolve where the best balance lies, but it does show that both directions are compatible with the same underlying idea: the one-time effort goes into preparing the use case, while the concrete task is later redefined through spatial input.

This in turn raises the question of how much programming structure such a system should expose. A one-off task definition is the strongest fit when the workflow is narrow and well understood. For more generic use cases, a simple linear sequence of commands may be a practical middle ground, because it allows somewhat broader tasks, such as defining multiple welds in sequence, without turning the interface into a full robot programming language. Beyond a simple linear command structure, however, the main advantage of the approach begins to weaken. If the operator must build programs with general logic, branching, loops, and low-level motion structure, it becomes less clear why this should be done through the present interface rather than through the manufacturer's programming environment. Spatial input may still remain useful there, especially for entering poses and trajectories in the real workspace, but at that point it would function better as an aid to robot programming than as a complete alternative to it.

\section{Current limitations and validity boundary}

The most important limitation of the current prototype is scene understanding. The scene-detection pipeline is intentionally simple: it operates on registry-defined fiducial boxes and exposes only a narrow object model. This was sufficient for the prototype demonstrations and for validating the architectural flow from scene sensing to use-case interpretation, but it is also the point where the present system is furthest from broader practical use. The issue is not only that the current implementation is box-based. More fundamentally, many tasks depend on richer object understanding than the prototype can currently provide, so this limitation affects how far the demonstrated workflow can be transferred beyond the present use cases.

A second group of limitations concerns sensing robustness and calibration. Marker-based tracking is sensitive to visibility, blur, lighting conditions, and calibration quality, and the current pen tracking does not fuse IMU data even though the device provides it. The robot-mounted stereo configuration also depends on a rigid mount and on successful calibration. These choices were acceptable for a prototype intended to test the workflow, but they leave open how robust the same setup would be in a less controlled environment. In that sense, the thesis shows that the workflow can be realized on real hardware, but not yet that the present sensing stack is mature enough for wider deployment without further refinement.

Other limitations are better understood as engineering incompleteness or deliberate scope boundaries than as threats to the central idea itself. On the execution side, the current system mainly covers robot motion. Generalized tool control is not implemented, and the demonstrated use cases do not perform collision checking, path planning, or simulation. These are real limitations, but they concern execution completeness more than the basic feasibility of the authoring workflow itself. Authored tasks are also still executed through the thesis runtime, which is partly a strength because it allows use cases such as pick-and-place to adapt execution to the current scene instead of freezing everything at authoring time. At the same time, this means that authored tasks are not yet exported into a native robot-side program representation. For fixed tasks such export may be possible as a later extension, whereas for runtime-adaptive tasks it becomes a deeper interface question rather than a simple serialization step.

The current frontend should be read in the same way. It is a prototype control plane rather than a finished operator interface, and its main role in the thesis is to make the workflow executable and inspectable. This matters for usability and workflow polish, but it is not the main factor that determines the validity of the thesis result. The more consequential questions concern scene understanding, sensing robustness, execution semantics, and the range of tasks that can be expressed through the present approach.

The evaluation boundary of the chapter is therefore qualitative and workflow-oriented. The system was demonstrated on representative use cases and on real hardware, but the thesis does not include a formal user study, a controlled comparison against pendant-based programming, or a quantitative characterization of global tracking accuracy. The contribution is therefore best read as a prototype-level demonstration of what this programming approach can support today, together with a clearer picture of where it appears most promising and where the unresolved difficulties still lie.

Taken together, these limitations place the result on a specific level rather than reducing it to a negative one. The thesis shows that a modular system for task-oriented spatial authoring can be built, that it can be exercised on substantially different use cases, and that it identifies a practically interesting region between fixed task integration and repeated low-level robot reprogramming. The next chapter turns from this interpretation of the current result to the directions in which the work can be extended.
