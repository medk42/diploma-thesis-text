\chapter{Discussion} \label{chapter:ch6_discussion}

This chapter returns to the question posed in the introduction: whether frequently changing collaborative-robot tasks can be programmed with less friction by specifying spatial intent directly in the real workspace. The goal here is not to repeat the implementation chapter, but to discuss what the built system actually shows, where it is useful, and where its current limits lie.

\section{What the thesis built}

The main result of this thesis is an end-to-end system for \emph{in-workspace spatial authoring} of selected robot tasks. The system combines workspace sensing, spatial input, task-specific interpretation, and robot execution behind a shared modular runtime with explicit interfaces. In practical terms, this means that the thesis did not only explore a single pen-based interaction technique. It produced a complete prototype workflow that covers graph construction, activation, calibration, authoring, confirmation, execution, visualization, and persistence, as described in \autoref{sec:ch3_ui_orchestration} and \autoref{chapter:ch5_user_workflow}.

At the same time, the thesis deliberately stayed within a narrower scope than a full robot programming environment. As stated already in the introduction, the goal was not to design a general-purpose language with branching, loops, exception recovery, or rich sensor-driven logic. Instead, the system focuses on \emph{task-oriented use cases} implemented as plugins on top of shared sensing and execution infrastructure. This choice is important for the interpretation of the results: the thesis is not evaluated by how well it replaces every existing robot-programming workflow, but by whether it provides a useful way of entering selected robot tasks through spatial input.

The two main use cases, pick-and-place and seam welding, are therefore more than simple demonstrations. They show that the same runtime can support different task structures while reusing the same architectural backbone. Pick-and-place depends on scene refresh at execution time and therefore tests whether the system can remain responsive to the current state of the workspace. Seam welding resolves the task during authoring and tests whether rough spatial input can be turned into a usable trajectory-centric procedure. Together, these use cases show that the thesis result is both a concrete prototype and a reusable framework for further task-oriented development.

\section{What the results mean}

In many existing deployments, a collaborative robot is integrated for one specific task and that task then remains largely fixed. The presented system explores a different approach. Here, the use case is prepared once, but the operator can redefine the concrete task instance quickly within that use case through spatial input. This shifts reprogramming away from repeated low-level editing through teach pendant programming, offline programming, or hand-guiding, and toward fast re-authoring inside a narrow task-specific interface.

This is also why the modular architecture matters. It is not the thesis result \emph{instead of} the use cases; it is the reason why the use cases are meaningful as evidence. The explicit separation between sensing, UI, visualization, robot execution, and use-case logic makes it possible to evaluate one concrete prototype today while still treating the system as an extensible basis for other inputs, other scene-detection pipelines, and other task modules later on. In that sense, the architecture is a real thesis result, but it should be read together with the use cases that give it practical meaning.

The two use cases reveal different strengths. Pick-and-place is important because it shows that the system is not limited to fully fixed commands. The command is authored once, but part of its execution is recomputed from the current scene. This is a positive result for the overall architecture: it demonstrates that the system can combine authored intent with runtime scene information instead of only replaying frozen trajectories. At the same time, pick-and-place does not appear to be the strongest practical target for the current prototype. In many industrial settings, material-handling tasks are highly specific and are usually integrated together with conveyors, grippers, and dedicated scene-sensing components. In such cases, a specialized integration may remain the more appropriate solution.

Seam welding is the clearest practical fit for the thesis approach. The task is naturally trajectory-centric, the user intent is spatially meaningful, and the resulting program structure is simple enough that the use case can infer the final robot procedure from the authored trajectory and the sensed scene. More importantly, this type of task matches the situation that motivated the thesis in the first place: work that may need to be re-taught repeatedly, but where the repeated physical execution itself is better handled by the robot than by the user. In that setting, the workflow of \emph{prepare the workpiece, indicate the intended weld, confirm the result, and run the robot} appears well matched to the task and reduces how much low-level pose entry the user must perform.

Taken together, these results suggest that the thesis is most relevant as a tool for repeated, task-specific reprogramming. The strongest value is not in replacing expert robot programming everywhere. Rather, it is in task classes where the user should remain responsible for setup and intent, while the robot performs the repetitive execution. This interpretation also explains why the thesis is well aligned with collaborative robots: the human and the robot do not solve the same part of the problem. The human provides local task knowledge and decides \emph{what} should happen; the robot performs the repeatable physical procedure.

\section{Where the approach is useful}

The clearest useful region for the current approach is work that is spatially meaningful, repeated often enough to justify robot execution, but variable enough that reprogramming by conventional means becomes tedious. This includes one-off operations, short batches, frequent changeovers, or tasks that are repeatedly re-authored in slightly different spatial configurations. In such cases, the cost is often not the motion itself, but the repeated effort of defining poses and trajectories in a low-level workflow.

This is where the thesis makes its strongest contribution. It shows that a user can interact with the robot system at the level of task intent rather than at the level of individual robot instructions. The use case still has to be implemented, and the current system still requires setup, calibration, and task-specific assumptions, but once that infrastructure exists, the actual authoring step becomes much closer to saying \emph{do this here} than to writing a complete robot program from scratch.

In that sense, the architecture can also be read as a reusable layer on top of a prepared robot cell. A use case can be developed once for a specific task family, while the operator later reuses it on site by providing new spatial inputs. This is still far from a turnkey industrial product, and the thesis does not validate such a deployment model experimentally. However, the prototype does show that this direction is technically coherent and worth further exploration.

The linear `UsecaseTree` model should also be interpreted in this context. For this thesis, it is best understood as a practical scope choice rather than as a major limitation. It allows multiple commands to be authored, confirmed, visualized, and executed in a consistent way without turning the system into a full robot language. That is sufficient for the evaluated task-oriented workflows and for testing how spatial authoring behaves across several commands. For the present thesis, this middle ground was more useful than either a single one-off command model or a full programming language would have been.

\section{Current limitations and validity boundary}

The largest practical limitation of the current prototype is scene understanding. The scene-detection pipeline is intentionally simple: it operates on registry-defined fiducial boxes and exposes only a narrow object model. This was sufficient for the prototype demonstrations and useful for validating the architectural flow from scene sensing to use-case interpretation. However, it is also the first component that would need fundamental extension for broader use. The limitation is not only that the current implementation is box-based, but that the practical usefulness of many tasks depends on richer object understanding than the present prototype can provide.

The second major limitation is the sensing and calibration stack. Marker-based tracking is sensitive to visibility, blur, lighting conditions, and calibration quality, and the current pen tracking does not fuse IMU data even though the device provides it. The robot-mounted stereo configuration also depends on a rigid mount and on successful calibration. These issues do not invalidate the concept demonstrated in the thesis, but they limit confidence in how far the current implementation would transfer unchanged beyond the present setup. In other words, the thesis shows that the workflow is viable as a prototype, but not yet that the current sensing stack is sufficient for a broader deployment.

The execution layer also remains incomplete. The current robot interface is motion-centric, tool actions are not abstracted in a general way, and the demonstrated use cases do not perform collision checking or path planning. Simulation mode is also not supported for the thesis programs. These are important limitations because they affect the gap between a working prototype and a more robust deployment. The current system is strongest when the task can be kept simple enough that the lack of deeper planning does not dominate the workflow.

One further consequence of the current design is that program execution remains mediated by the thesis runtime. This is partly a strength, because it allows use cases such as pick-and-place to adapt execution to the current scene instead of freezing everything at authoring time. At the same time, it also means that the thesis does not yet provide export of authored tasks into a native robot-side program representation. For fixed tasks this may be possible as a future extension, but for runtime-adaptive tasks it becomes a deeper design question rather than a simple serialization step.

Compared with these issues, the current UI should be seen as a secondary limitation. The frontend is explicitly a prototype control plane rather than a production interface. This affects operator comfort and workflow polish, but it is not the main factor that determines whether the thesis result is meaningful. The more important questions concern scene understanding, tracking quality, execution planning, and task semantics. Those are the parts that define whether the approach can become broadly useful, whereas the UI can be refined later without changing the core concept.

The evaluation boundary of this chapter is therefore qualitative and workflow-oriented. The system was demonstrated on representative use cases and on real hardware, but the thesis does not include a formal user study, a controlled comparison against pendant-based programming, or a quantitative characterization of global tracking accuracy. For that reason, the claims of this chapter should stay at the level of \emph{what the prototype shows} and \emph{what that suggests}. The thesis can reasonably argue that spatial authoring appears promising for selected task-oriented workflows, especially trajectory-centric tasks such as welding. It cannot reasonably claim that the presented prototype is already a general replacement for established robot-programming methods.

These limits do not reduce the thesis to a negative result. On the contrary, they help define the scope of what has actually been shown. The thesis demonstrates that a modular system for task-oriented spatial authoring can be built, that it can be exercised on substantially different use cases, and that it reveals a practically interesting region between fixed task integration and repeated low-level robot reprogramming. The next chapter therefore turns from this interpretation of the current result to the concrete directions in which the work can be extended.
