\chapter{Problem Decomposition and Analysis} \label{chapter:ch2_analysis}

In this chapter we will analyze the goals of this thesis and derive requirements for our system. We will then analyze different approaches of meeting these requirements, compare them to each other, and select an approach best suited to our goals and requirements. 

\section{Target tasks and their requirements} \label{sec:ch2_analysis_tasks_and_requirements}

Let us first re-state the goals we want to achieve with our system for reference:
\begin{goals}
  \item \label{an_goal:1} Explore and validate programming of cooperative robots directly in the real workspace by defining robot motions and tool actions using spatial, AR-inspired inputs, as an alternative to traditional teach pendant, offline programming, and hand-guiding workflows.
  \item \label{an_goal:2} Provide an extensible system where task-oriented use cases can be implemented as plugins that reuse the same acquisition/execution infrastructure.
  \item \label{an_goal:3} Demonstrate the approach on a real robot with representative use cases for a material-handling task (pick-and-place) and a trajectory-centric task (seam welding), and evaluate the benefits and limitations of spatial authoring for these tasks.
\end{goals}

First, we will define the requirements for the system based on the use cases we want to implement. The two use cases mentioned in goal \ref{an_goal:3} are a material-handling task (\emph{pick-and-place}) and a trajectory-centric task (\emph{seam welding}). 

Our system should also support the addition of new use cases (goal~\ref{an_goal:2}), so we will outline some potential future use cases and see if the requirements derived from our main use cases are sufficient to support them as well. 

One example of a future use case is a \emph{painting task}, where the goal is to uniformly paint a selected face of an object using a spray gun. In addition, the system should be capable of expressing traditional robot motion primitives (e.g., moving to a point or along a path). While such primitives are not the main focus of our work, their compatibility with the proposed input architecture is relevant.

The ability to support both these additional tasks and the primary use cases does not constitute a formal proof of generality; rather, it serves as a \emph{plausibility check} and a concrete data point indicating that the proposed requirements and abstractions are sufficiently expressive to cover a broader class of tasks, including ones not explicitly targeted during system design.

\subsection{Pick-and-place} \label{subsec:ch2_analysis_pick_and_place}

One way to implement a \emph{pick-and-place} task is the following:
\begin{enumerate}[label=(\arabic*), itemsep=0.2em]
    \item \emph{scan the scene} to find the object to be picked (we also need to know which object to pick),
    \item \emph{calculate the grasp pose} (the detected position of the object is not necessarily the same as the grasp pose),
    \item \emph{move to the grasp pose},
    \item \emph{close the gripper},
    \item \emph{move to the release pose} without crashing into anything,
    \item \emph{open the gripper}, and
    \item \emph{move away}, for example, to a safe home position.
\end{enumerate}

This gives us several task-specific requirements:
\begin{enumerate}[label=(\roman*), itemsep=0.2em]
    \item the system needs to be able to capture the scene and detect objects in it, \label{req:scene_detection}
    \item the system needs to be able to calculate the grasp pose based on the detected object, \label{req:grasp_calculation}
    \item the user needs to define the object type to be picked, and \label{req:object_type}
    \item the user needs to define the release pose. \label{req:release_pose}
\end{enumerate}

Therefore the system will need to be able to provide at least \emph{scene detection} logic (covers requirement \ref{req:scene_detection}) and a spatial \emph{6D input interface} (requirement \ref{req:release_pose}). Requirement \ref{req:object_type} can be fulfilled either by a simple UI element (e.g. dropdown) or by spatial input as well --- given that we already plan to support a \emph{6D input interface}, we can also use it for object type selection too. Finally requirement \ref{req:grasp_calculation} can be fulfilled either on the use case plugin level (plugin will calculate the grasp pose based on the detected object) or even from the user input (user selects the object with a 6D input, we can also consider the selection pose as the grasp pose).

\subsection{Seam welding} \label{subsec:ch2_analysis_seam_welding}

One way to implement a \emph{seam welding} task is the following:
\begin{enumerate}[label=(\arabic*), itemsep=0.2em]
    \item \emph{approach the seam} to be welded (usually not the start of the seam, but a point close to it),
    \item \emph{approach the start of the seam},
    \item \emph{start welding},
    \item \emph{move along the seam} while welding,
    \item \emph{stop welding} at the end of the seam, and
    \item \emph{move away}, for example, to a safe home position.
\end{enumerate}

This gives us the following task-specific requirements:
\begin{enumerate}[label=(\roman*), itemsep=0.2em]
    \item the user needs to define the approach pose (a point close to the start of the seam), \label{req:approach_pose}
    \item the user needs to define the seam (either as a linear segment --- start and end points --- or as a trajectory), and \label{req:seam_definition}
    \item the user needs to define the welding parameters (e.g. welding speed, welding power, weaving settings). \label{req:welding_parameters}
\end{enumerate}
To fulfill these requirements, we could use the same \emph{6D input interface} as for the pick-and-place use case --- the user can select the approach pose (fulfilling requirement \ref{req:approach_pose}) and then define the seam by either selecting the start and end points of the seam or by defining a trajectory along the seam (requirement \ref{req:seam_definition}).

However this approach would depend heavily on the precision of the \emph{6D input interface}. Therefore we could instead first use the \emph{scene detection} logic to detect objects within the scene and then use use case specific logic to detect possible seams between the detected objects --- the user would then still use the \emph{6D input interface} to enter a trajectory representing the seam, however the system would align the user's input to the closest detected seam (requirement \ref{req:seam_definition}). The approach pose (requirement \ref{req:approach_pose}) could be automatically defined by the use case as a point at a certain distance from the start of the seam along the seam normal. Finally, the welding parameters (requirement \ref{req:welding_parameters}) could be defined by a simple UI element (with reasonable defaults).

\subsection{System requirements} \label{subsec:ch2_analysis_system_requirements}

Subsections \ref{subsec:ch2_analysis_pick_and_place} and \ref{subsec:ch2_analysis_seam_welding} give us the following system requirements:
\begin{requirements}
    \item \emph{6D input interface} for defining poses and trajectories (both in 6D; that is, with position and orientation) in the workspace. While extreme precision is not strictly required --- since use-case-specific logic can refine the user input with the help of the scene data --- higher input precision is nevertheless preferable, as it simplifies downstream processing, improves alignment robustness, and better supports future use cases with tighter accuracy requirements. \label{req:6D_input_interface}
    \item \emph{scene detection} logic that can detect objects in the scene and provide their poses. The precision requirements for the scene detection are higher than for the 6D input interface, because we use the detected objects to refine the user input. \label{req:scene_detection_logic}
    \item \emph{UI} for defining non-spatial parameters (e.g. object type for pick-and-place, welding parameters for seam welding). \label{req:ui}
\end{requirements}

Beyond the requirements directly derived from the use cases, we also have some more general requirements:
\begin{requirements}[resume]
    \item \emph{UI} also has to support selection of the use case to be programmed, visualization of the created program, and confirmation of the programming. This requirement is not directly derived from the use cases, but it is necessary for a complete system that can be used by end-users. \label{req:ui_use_case_selection}
    \item \emph{robot communication interface} that allows the use cases to move the robot and control its tool, and that can support multiple robot brands/models to ensure broader applicability. \label{req:robot_communication_interface}
    \item \emph{use case plugin support} that allows the addition of new use cases as plugins, reusing the same input and execution infrastructure outlined in the previous requirements. \label{req:use_case_plugin_support}
\end{requirements}

\subsection{Coverage of Additional Use Cases}

The requirements outlined in \autoref{subsec:ch2_analysis_system_requirements} are sufficient to cover the additional use cases mentioned in \autoref{sec:ch2_analysis_tasks_and_requirements}. 

For example, the \emph{painting task} can be implemented using the \emph{6D input interface} to select the face to be painted as a 6D pose or a trajectory circling the the face. The requested face can then be chosen by the use case logic as the face closest to the user input. The \emph{UI} can be used to define painting parameters (e.g. paint flow rate, spray pattern) and the use case can then generate the appropriate robot trajectory.

Similarly, traditional robot motion primitives can also be implemented using the same \emph{6D input interface} for defining target poses or paths. This use would require higher precision and lower noise of the input, particularly for defining paths, which further emphasizes the importance of a high-quality \emph{6D input interface}. At the same time, a use case may apply smoothing or refinement to the user-provided path, which can partially relax the precision requirements on the input.

\subsection{6D input interface} \label{subsec:ch2_analysis_6D_input_interface}

As stated in requirement \ref{req:6D_input_interface}, our system needs to provide \emph{6D input} (position and orientation) for defining poses and trajectories in the real workspace. High precision is not a strict requirement for our two use cases, because use case logic can refine user input using scene data, but higher stability and lower jitter are still desirable --- they simplify downstream processing and keep the interface usable for potential future tasks.

When selecting an input approach, we consider:
\begin{enumerate*}[label=(\roman*)]
    \item required hardware and calibration effort,
    \item tracking quality (noise/jitter, occlusion sensitivity, and latency),
    \item integration complexity, and
    \item intent signalling (how the user indicates when to capture a pose or start/stop sampling a trajectory).
\end{enumerate*}

We consider three candidate approaches:
\begin{enumerate}[label=(\roman*), itemsep=0.2em]
    \item \emph{hand tracking}, where the user's hand is the input device,
    \item \emph{VR/AR controllers}, using existing VR/AR hardware, and
    \item \emph{camera-based marker tracking}, where the user holds a tool with a known marker tracked by an external camera.
\end{enumerate}

\paragraph{Hand tracking.}
Hand tracking is an attractive option because it is intuitive and does not require a dedicated held tool. It can be implemented using open-source solutions such as MediaPipe \cite{MediaPipeRoot} (also used by Vysocký et al.\ \cite{Vysocky2023}) or via commercial ecosystems such as the Meta Quest headset or Leap Motion. Meta Reality Labs also reports several recent research systems for hand tracking on headset devices \cite{FB_research, FB__UmeTrack, FB__MEgATrack, FB__ElasticHandTrack}.
However, robust 6D interaction in our setting requires not only a plausible hand pose, but also sufficiently stable spatial position and orientation for selecting poses and drawing trajectories. With a single RGB camera (e.g., MediaPipe), depth can only be estimated, which is typically not accurate enough for precise workspace authoring. Multi-camera setups or RGB-D sensors can mitigate this (as in \cite{Vysocky2023}), but at the cost of additional calibration effort and system complexity. Additionally, reported accuracy limitations for gesture-based robot path definition \cite{Vysocky2023} make hand tracking a risky default for our use case.
A related idea by Zhang et al.\ \cite{Zhang2023} improves robustness for surface-following by using an RGB-D-T camera and thermal cues to infer contact on the object surface, but this does not directly generalize to trajectories in free space.

Hand tracking also complicates \emph{intent signalling}, because there is no explicit hardware trigger for pose capture or trajectory start/stop. This can be implemented using gestures, but gesture recognition can be error-prone and may conflict with reliable orientation estimation (e.g., pointing provides a clear direction, while pinch-like gestures can be less stable and can be triggered accidentally). Tracking quality can also degrade under occlusions and changing lighting conditions.

One could attempt to restrict hand tracking to 2D interaction and use scene data to infer intent (e.g., selecting a seam on a surface). While such a design could work for some surface-bound tasks, it does not generalize to cases where target poses are not located on a visible surface (e.g., a free-space release pose in pick-and-place). Therefore, we only consider fully 3D hand tracking as a candidate here.


\paragraph{VR/AR controllers.} 
VR/AR controllers offer accurate, low-latency 6D tracking and typically provide explicit buttons, which makes intent signalling (capture pose or start/stop sampling a trajectory) straightforward. The downside is that this approach requires dedicated hardware and depends strongly on the tracking stack and APIs provided by the manufacturer.

\paragraph{Camera-based marker tracking.}
Marker tracking is a widely used approach in AR pipelines: a known marker is detected in the camera image and its 6D pose is estimated. Compared to hand tracking, it can be simpler to implement, works with a single camera, and can provide sufficiently stable input for our authoring tasks. Since the user already holds a tool with the marker, we can also add an explicit trigger (e.g., a button), making intent signalling straightforward. The main limitation is that precision depends strongly on practical factors such as camera quality, marker size, viewing distance, lighting, and occlusion. In general, camera-only pipelines may introduce latency depending on camera exposure, frame rate, and processing, but they avoid some of the ambiguity inherent in single-camera hand tracking.

The D-Point pen \cite{dpoint_github} illustrates this category (camera-based marker tracking combined with an IMU), although replication reports \cite{dpoint_issues} indicate that achieved precision can be sensitive to implementation and hardware choices.

In our implementation, we instantiate camera-based marker tracking using \emph{ArUco} markers and the corresponding open-source detection pipeline. The specific choice of marker family is not fundamental to the interface: other fiducial systems (e.g., AprilTag) could be substituted with minimal changes to the surrounding architecture, as the downstream components consume only the estimated 6DoF pose and marker ID.

\subsection{Scene detection} \label{subsec:ch2_analysis_scene_detection}

Based on requirement \ref{req:scene_detection_logic}, our system needs to provide \emph{scene detection} logic that can detect objects in the scene and provide their poses. The accuracy requirements for scene detection are higher than for the 6D input interface, since the detected object poses are used to refine and constrain user-defined trajectories. For example, in the seam welding use case, we rely on the detected objects to find the seam between them and align the user-defined trajectory to it. If the estimated object poses are inaccurate, the derived seam geometry and aligned trajectory will also be affected, directly degrading execution quality.

When selecting an approach for scene detection, we consider:
\begin{enumerate*}[label=(\roman*)]
    \item accuracy,
    \item implementation effort,
    \item generalizability,
    \item cost, and
    \item runtime complexity.
\end{enumerate*}

We consider six candidate approaches:
\begin{enumerate}[label=(\roman*)]
    \item \emph{fiducial-based detection}, where known fiducial markers are placed on the objects to be detected,
    \item \emph{single-view model-based pose estimation}, where the system detects objects from a single camera based on known 3D models, \label{req:single_view_model_based_pose_estimation}
    \item \emph{multi-view model-based pose estimation}, where the system detects objects from multiple cameras based on known 3D models, \label{req:multi_view_model_based_pose_estimation}
    \item \emph{3D scanning and reconstruction}, where the system reconstructs a 3D model of the scene from many cameras and detects objects based on it, \label{req:3D_scanning_and_reconstruction}
    \item \emph{depth-based 3D scanning and reconstruction}, where the system merges multiple depth images of the scene into a 3D model and detects objects based on it, and \label{req:depth_based_3D_scanning_and_reconstruction}
    \item \emph{commercial systems} that handle scene detection.
\end{enumerate}

Approaches \ref{req:single_view_model_based_pose_estimation}-\ref{req:multi_view_model_based_pose_estimation} are discussed together as \emph{Model-based pose estimation} (single- vs multi-view variants), and \ref{req:3D_scanning_and_reconstruction}-\ref{req:depth_based_3D_scanning_and_reconstruction} are discussed together as \emph{Reconstruction-based approaches} (RGB vs depth-based variants).

\paragraph{Fiducial-based detection.}
Fiducial-based detection relies on artificial markers placed on the objects to be detected. The system detects the marker in the camera image, estimates its 6D pose, and applies a known transformation between the marker coordinate frame and the object coordinate frame to obtain the object pose. When markers such as \emph{ArUco} or \emph{AprilTag} are used, the object can also be identified through the marker ID.

In this thesis, we implement fiducial-based detection using \emph{ArUco} markers, which provide both object identification (via marker ID) and a direct 6DoF pose estimate using widely available open-source libraries. As with the pen input, the choice of ArUco is primarily pragmatic; alternative fiducial systems could be integrated later while keeping the same object-pose abstraction at the output of the scene detection component.

This approach can operate with a single calibrated camera and is relatively simple to implement. However, it requires physically instrumenting each object with a marker and ensuring precise marker placement. Errors in marker placement directly translate into systematic pose errors, and rotational inaccuracies can be amplified for points further away from the marker location.

The accuracy depends on camera resolution, calibration quality, marker size, viewing angle, and lighting conditions. It can be improved by using larger markers, optimizing camera placement, and reducing viewing angles that cause perspective distortion.

The implementation effort is low, as it primarily involves camera calibration and integration of an existing marker detection library. The hardware cost is also minimal, requiring only printed markers and a camera.

The runtime complexity is low, since marker detection and pose estimation can typically be performed in real time on a single camera stream.

While robust and efficient, this approach does not generalize well to scenarios where placing markers on objects is impractical or undesirable.


\paragraph{Model-based pose estimation.}
Model-based pose estimation relies on known 3D models of the objects to be detected. The goal is to estimate the 6D pose of each object instance in the scene relative to the camera or world coordinate system. Multiple approaches exist, using single- or multi-view RGB, grayscale, or depth cameras. The yearly BOP challenge \cite{BopChallenge} benchmarks the state of the art in 6D object pose estimation. An example of a multi-view multi-object 6D pose estimation system is \emph{CosyPose} \cite{labbe2020,CosyPoseWebsite}.

Compared to fiducial-based detection, this approach does not require instrumenting objects with markers. However, unlike fiducial-based detection, it requires known 3D object models in order to estimate object poses. In return, it directly provides object-level detection and pose estimation, inherently separating the scene into individual object instances without requiring explicit scene segmentation.

The accuracy of model-based pose estimation can be high, but depends on several factors, including the quality of the object models, the camera setup, and the specific algorithm used. Multi-view approaches typically provide higher robustness and accuracy than single-view ones, as they reduce depth ambiguity and improve handling of occlusions. Single-view RGB-based approaches must infer depth from visual cues and learned priors, which can introduce ambiguity in challenging scenarios.

The implementation effort can be significant, as it requires integration of the pose estimation pipeline, preparation of accurate object models, and potentially object-specific tuning to achieve reliable results in the target environment. In multi-camera setups, additional calibration and synchronization are required.

The approach generalizes well across different environments as long as object models are available. However, it does not naturally extend to previously unseen objects without first acquiring their 3D models.

The cost can remain relatively low when using open-source implementations and a limited camera setup. However, hardware costs increase if multiple synchronized cameras or depth sensors are required.

The runtime complexity can also be considerable, particularly for multi-view systems that require synchronization and processing of multiple camera streams.


\paragraph{Reconstruction-based approaches.}
3D scanning and reconstruction approaches aim to create a geometric 3D representation of the scene, either as a dense point cloud or directly as a mesh. This can be achieved by acquiring and aligning depth images of the scene from multiple viewpoints, or by capturing a sufficiently dense set of overlapping RGB/grayscale images from multiple viewpoints and applying structure-from-motion and multi-view stereo techniques.

This approach does not require known 3D models of the objects. However, it also does not inherently separate the scene into individual objects --- downstream processing is required to segment the scene into object-level representations, which can be a non-trivial problem. For example, seam detection for welding may still be feasible through geometric analysis of the reconstructed surfaces, whereas pick-and-place applications typically require explicit object-level segmentation and instance identification.

The accuracy of the reconstructed model can be high, but strongly depends on the properties of the captured scene (e.g., low-texture or reflective surfaces can cause difficulties for RGB-based reconstruction, while depth-based reconstruction can struggle with thin structures or small objects).

The implementation effort can be significant, as it requires integrating multiple views into a consistent geometric representation. The cost varies depending on the hardware setup, ranging from a single moving camera to a dense multi-camera configuration. RGB-based systems can be more affordable, but require good lighting and sufficient surface texture. Depth-based systems can be more robust to lighting and texture variation, but typically require more specialized hardware.

The runtime complexity can also be high, as many images must be captured and processed from multiple viewpoints (requiring physically moving the robot-mounted camera), especially for RGB-based approaches relying on structure-from-motion.

\paragraph{Commercial systems.}
Commercial machine vision systems integrate sensing hardware and proprietary software into turnkey solutions for industrial scene detection and pose estimation. Examples include industrial 3D vision platforms such as Photoneo PhoXi \cite{PhotoneoWebsite} and Cognex In-Sight systems \cite{CognexWebsite}. These systems are typically designed for applications such as bin picking, part localization, and automated inspection. They combine calibrated sensing hardware (often structured-light, laser, or stereo-based) with optimized segmentation and model-matching pipelines to provide object-level detection and 6D pose estimation directly to industrial robots.

The primary advantage of commercial systems is their robustness and industrial maturity. They are optimized for reliability in controlled industrial environments, provide vendor support, and can achieve high accuracy when properly configured. Integration effort can be moderate when using supported robot platforms, as standardized communication interfaces and deployment tools are typically available.

However, these systems are usually associated with high acquisition costs. They are also less flexible than custom research pipelines, since internal algorithms are proprietary and access to intermediate representations is limited. This can make adaptation to experimental setups or rapidly evolving research requirements more difficult. Furthermore, extending the system to new object categories or task types may require additional configuration, licensing, or vendor-specific retraining procedures.

Since commercial systems are typically installed as fixed sensing units and operate without requiring sensor movement during runtime, their online computational requirements are generally moderate. Nevertheless, they often operate in a request–response mode rather than streaming every frame, which can increase effective end-to-end latency depending on the deployment.

\paragraph{Summary}

Given the scope of this thesis and the available hardware, fiducial-based detection provides the most favorable balance between implementation effort, cost, and sufficient pose accuracy for our target use cases. The approach can be implemented using the same camera setup as the 6D input interface, and the detection pipeline can reuse significant parts of the existing marker-based pose estimation logic.

While marker-based detection does not provide perfect accuracy and may exhibit pose noise due to suboptimal lighting conditions and limited camera resolution, the obtained precision is sufficient for the seam welding and pick-and-place demonstrations presented in this thesis. Accuracy could be further improved by aggregating multiple observations over time or increasing image resolution.

Therefore, we select a camera-based fiducial marker detection approach for scene detection in this thesis. At the same time, we will keep the system architecture modular, allowing replacement of the detection component with model-based or reconstruction-based approaches in future work if higher robustness or model-free operation is required.


\subsection{Camera configuration} \label{subsec:ch2_analysis_camera_tradeoffs}

In the previous subsections (\autoref{subsec:ch2_analysis_6D_input_interface} and \autoref{subsec:ch2_analysis_scene_detection}) we selected camera-based marker tracking for both the 6D input interface and the scene detection. This shifts the main design question to the \emph{camera configuration}: the same camera setup determines the achievable pose stability for the pen input and the pose accuracy for detected scene objects.

We evaluate alternative camera configurations along two axes:
\begin{enumerate*}[label=(\roman*)]
  \item expected tracking quality for both pipelines (pen tracking and scene detection), and
  \item practical deployment cost (mounting, calibration, synchronization, and robustness in a real workspace).
\end{enumerate*}

Although planar fiducial pose can be estimated from a single calibrated RGB camera, our early prototyping revealed two practical issues: (i) depth (translation along the viewing direction) was noticeably noisier than the in-plane components, and (ii) we observed small systematic depth offsets that varied with detection conditions (e.g., changes in lighting affecting the perceived marker scale). This motivated switching to a stereo camera, where depth is recovered primarily by triangulation rather than relying on apparent marker size.

We evaluate the candidate configurations using:
\begin{enumerate*}[label=(\roman*)]
    \item setup/maintenance complexity (calibration effort, synchronization),
    \item pen 6D tracking quality (depth stability, jitter),
    \item scene pose quality (achievable accuracy, robustness), and
    \item mount robustness (likelihood the calibrated camera-to-world transform stays valid: bumps, vibration, cable strain, and incidental contact).
\end{enumerate*}

We consider six candidate camera configurations:
\begin{enumerate}[label=(\roman*), itemsep=0.2em]
    \item \emph{single robot-mounted camera}, where a single camera is rigidly mounted on the robot end-effector,
    \item \emph{single fixed camera}, where a single camera is rigidly mounted in the workspace,
    \item \emph{stereo robot-mounted camera}, where a stereo camera rig is rigidly mounted on the robot end-effector,
    \item \emph{stereo fixed camera}, where a stereo camera rig is rigidly mounted in the workspace,
    \item \emph{multiple fixed cameras}, where multiple cameras are rigidly mounted in the workspace, and
    \item \emph{robot-mounted camera, multi-view capture}, where a robot-mounted camera is used for pen tracking and the robot is moved through multiple viewpoints for scene capture.
\end{enumerate}

In all robot-mounted configurations we assume the robot is stationary during pen input and scene capture (or the motion is fully controlled by the capture procedure). Therefore, we do not consider the delay between reported and actual robot position as a significant factor for tracking quality. Instead, the main practical factor is whether the calibrated extrinsic relations remain valid over time: small mechanical shifts can easily translate into millimeter-level errors in the workspace at typical camera-to-scene distances.

\paragraph{Single robot-mounted camera.}
A single camera mounted on the robot end-effector is one of the simplest configurations to deploy. It requires intrinsic calibration and a one-time extrinsic calibration of the camera relative to the robot flange/end-effector. During runtime, the camera pose in the robot/world frame can be derived from robot forward kinematics combined with this fixed extrinsic transform. Since there is only one camera stream, no synchronization is required.

However, in monocular planar marker detection, pose uncertainty is not uniform across degrees of freedom. Translation parallel to the image plane is driven mainly by pixel localization of marker corners, while translation along the viewing direction (depth) is more sensitive to projective geometry and the known physical marker size. In practice this often produces noticeably noisier depth, especially at larger viewing distances. Rotational stability also depends on viewing angle: orientations that produce strong image-plane changes are usually estimated more reliably than out-of-plane rotations that only induce subtle perspective distortions.

Additionally, planar pose estimation can exhibit ambiguities in challenging views; practical libraries typically apply disambiguation heuristics, but failure cases exist. The estimate is also less well constrained under near-frontal views due to limited perspective distortion \cite{Rijlaarsdam2022}. For pen tracking, this matters because orientation noise can be amplified at the tool tip when the marker is located further from the tip (as in pen-like designs) \cite{dpoint_github}. For scene detection, similar issues arise when markers are placed on top faces and are frequently observed close to fronto-parallel.

Regarding mount robustness, a robot-mounted camera can be mechanically very repeatable if it is attached through a rigid, well-defined interface (e.g., a bracket keyed to the robot flange). In our setup, once the camera is mounted and calibrated, the relative camera-to-robot transform tends to remain stable, and even removal/re-mounting can produce only small errors when the interface constrains the pose well. The main risks are direct collisions with the environment and gradual shifts due to cable strain; the latter of which can be mitigated by careful routing and strain relief.

\paragraph{Single fixed camera.}
A single camera rigidly mounted in the workspace avoids collisions caused by robot motion and simplifies cabling, but it shifts the burden to the external mounting structure. Calibration requires intrinsic calibration and an extrinsic calibration of the camera relative to the robot/world frame. In principle this is comparable in complexity to the robot-mounted case, but in practice the calibrated transform is more likely to drift: a tripod, clamp, or wall mount can flex, be bumped, or settle over time.

The key practical disadvantage is that the camera-to-robot/world relation depends on the integrity of a longer external chain (mounting hardware, attachment points, surrounding structure). Small mechanical shifts at the mount can translate into significant pose errors in the workspace. Therefore, the limiting factor is often not the calibration procedure itself, but maintaining a rigid, disturbance-resistant mounting solution.

\paragraph{Stereo robot-mounted camera.}
A stereo configuration directly targets the main weakness of monocular pose estimation: unstable depth. Stereo recovers depth from disparity between two viewpoints, which reduces sensitivity to small changes in apparent marker scale and improves depth stability at typical working distances. In addition, some rotations that are weakly constrained in monocular views benefit indirectly from improved depth consistency across the marker surface.

Compared to a single robot-mounted camera, a stereo rig requires additional calibration (stereo extrinsics) and some form of synchronization between the two streams (often supported by the camera hardware). The rig is also larger, which can increase collision risk and complicate cable management. Nevertheless, the deployment workflow can remain simple, while tracking quality for both pen input and scene detection improves substantially.

\paragraph{Stereo fixed camera.}
A fixed stereo rig provides the same geometric depth advantages as the robot-mounted stereo case, while keeping the sensor physically separate from robot motion. The trade-offs are similar to the single fixed camera case: the critical factor becomes maintaining a rigid, stable mount so that the calibrated camera-to-world transform stays valid. The setup effort is higher than mono due to stereo calibration and synchronization, but the achievable tracking quality is typically much better than with a single fixed camera.

\paragraph{Multiple fixed cameras.}
Multiple fixed cameras can further improve robustness by providing additional viewpoints. Compared to stereo, extra cameras can reduce occlusions and provide more diverse viewing angles, making near-frontal and ambiguous views less likely. Both pen tracking and scene detection can benefit from multi-view constraints, especially in cluttered scenes or when the marker is frequently partially occluded.

This comes at the cost of substantially increased complexity: each camera must be calibrated, the system must maintain consistent extrinsics across the set, and synchronization becomes more challenging (multi-camera rigs typically do not provide the same out-of-the-box synchronization as stereo devices). Mount robustness also becomes harder to guarantee, because the overall quality depends on \emph{all} cameras remaining rigidly aligned. While multi-camera setups can yield excellent results in controlled installations, they are less attractive under rapid prototyping and frequent workspace changes.

\paragraph{Robot-mounted camera, multi-view capture.}
In this configuration, a robot-mounted camera is used for pen tracking, and the robot is moved through multiple viewpoints to capture the scene for reconstruction or multi-view pose refinement. This combines the mechanical repeatability of a robot-mounted sensor with the geometric benefits of multiple viewpoints.

The main trade-off is workflow complexity and runtime: scene capture becomes an active procedure that requires moving the robot, acquiring multiple frames, and integrating them. This increases scene acquisition time and introduces additional safety and coordination requirements, but can significantly improve scene pose quality (e.g., by reducing occlusion and depth ambiguity) without installing multiple fixed cameras.

Overall, monocular configurations are the simplest to deploy but suffer from depth instability and viewing-angle sensitivities that directly affect both pen tracking and fiducial scene pose estimation. Stereo addresses these weaknesses with moderate additional calibration and integration effort. Multi-camera setups can further improve robustness (occlusion handling and view diversity), but at the cost of substantially higher calibration/synchronization complexity and lower flexibility for rapid prototyping.

Robot-mounted configurations can provide stronger \emph{mount robustness} in practice because the camera pose remains mechanically tied to the robot, but they increase collision risk and require careful cable management. Fixed cameras avoid collision risks from robot motion, but maintaining calibration integrity depends heavily on the stiffness and disturbance-resistance of the external mount.

Based on our prototyping results and the practical requirement for stable depth together with reliable mount robustness, we implement the system using a \emph{robot-mounted stereo camera} in this thesis. We do not use robot-mounted multi-view capture, since the single stereo setup is sufficient as a proof of concept for the overall system; nevertheless, multi-view capture remains an interesting option for future work if higher scene pose quality is required.

Finally, the analysis above reflects our ArUco-based implementation and common planar-marker behavior. In our implementation we use OpenCV ArUco markers for both the pen and scene objects; alternative families (e.g., AprilTag) are compatible with the same overall pipeline. In particular, \emph{STag} markers \cite{BENLIGIRAY2019158} are a promising option for improving pose stability by combining square and circular features, and they were evaluated against several common marker systems.
