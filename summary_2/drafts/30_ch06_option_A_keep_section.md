\section{What the results mean}

In many existing deployments, a collaborative robot is integrated for one specific task and that task then remains largely fixed. The presented system explores a different approach. Here, the use case is prepared once, but the operator can redefine the concrete task instance quickly within that use case through spatial input. This shifts reprogramming away from repeated low-level editing through teach pendant programming, offline programming, or hand-guiding, and toward fast re-authoring inside a narrow task-specific interface.

This difference depends on the architectural choice made in the thesis. The shared runtime separates sensing, the operator-facing control interface, 3D scene presentation, robot execution, and use-case logic through explicit module boundaries. As a result, pick-and-place and seam welding are not two unrelated demos, but two concrete task definitions built on the same infrastructure. This is important for interpreting the result: the thesis does not only show that two example tasks can be made to work, but that one system can support different task structures while keeping the task-specific logic inside the use case.

The two use cases reveal different strengths. Pick-and-place is important because it shows that the system is not limited to fully fixed commands. The command is authored once, but part of its execution is recomputed from the current scene. This is a positive result for the overall architecture: it demonstrates that the system can combine authored intent with runtime scene information instead of only replaying frozen trajectories. At the same time, pick-and-place does not appear to be the strongest practical target for the current prototype. In many industrial settings, material-handling tasks are highly specific and are usually integrated together with conveyors, grippers, and dedicated scene-sensing components. In such cases, a specialized integration may remain the more appropriate solution.

Seam welding is the clearest practical fit for the thesis approach. The task is naturally trajectory-centric, the user intent is spatially meaningful, and the resulting program structure is simple enough that the use case can infer the final robot procedure from the authored trajectory and the sensed scene. More importantly, this type of task matches the situation that motivated the thesis in the first place: work in which the programming effort can become significant because the welds have to be re-taught repeatedly. For fully repetitive welding in a fixed production setup, the advantage of this approach is limited because the task can be integrated once and then reused. In small-batch production, the advantage becomes more meaningful because the same type of operation must be reprogrammed repeatedly. The most interesting case is one-off or highly variable welding, where a human could still perform the task manually, but a sufficiently capable version of this approach could make robot execution practical by lowering the cost of programming. In that setting, the workflow of \emph{prepare the workpiece, indicate the intended weld, confirm the result, and run the robot} is not only more direct, but points toward a form of human-robot cooperation in which the human defines the task in context and the robot performs the consistent physical execution.

Taken together, these results suggest that the thesis is most relevant not for static high-volume automation, but for tasks that fall between manual work and conventional robot integration. The strongest value is not in replacing expert robot programming everywhere. Rather, it is in task classes where the user still prepares the setup, understands the local situation, and decides \emph{what} should happen, while the robot performs the repeatable part of the physical procedure. In the current prototype, this idea is most convincing for welding-like tasks and less convincing for highly specific pick-and-place cells.

\section{Where the approach is useful}

The useful region of the approach is not defined only by how often a task is repeated, but by the balance between programming effort and execution benefit. If a task is integrated once and then reused unchanged, the presented workflow offers little advantage. If the task changes often enough that the operator would otherwise need to keep redefining poses and trajectories, then the cost of programming itself becomes the main issue. This is where the approach becomes interesting.

That is why one-off work, short batches, and frequent changeovers are especially relevant. In such cases, the physical operation may still be simple, but the concrete spatial definition changes from run to run. The main contribution of the thesis is therefore not only that it lets the user enter robot motions differently, but that it moves the user interaction upward from individual robot instructions to task intent inside a prepared use case. Once the use case exists, the user no longer needs to rebuild the full procedure each time.

This also helps explain how much programming structure such a system should expose. The strongest form of the approach is a task-specific use case that already covers the intended workflow and only asks for the concrete spatial inputs. For more generic situations, a simple linear sequence of commands can still be useful, because it extends the range of supported tasks without forcing the user into a full robot programming language. Beyond that point, however, the advantage of the approach begins to weaken. If the operator has to assemble a complex program with general logic, loops, and low-level motion structure, it becomes less clear why this should be done through the present interface rather than through the manufacturer-provided programming environment.

A related distinction is whether the use case is prepared specifically for one customer or provided in a more generic form. A customer-specific use case can match one recurring workflow closely and keep the operator interface very narrow. A more generic use case can cover a broader class of tasks, but usually at the cost of more parameters and a more demanding authoring process. The thesis does not resolve which of these approaches is preferable. It does, however, show that both fit naturally into the same architectural idea: the one-time effort goes into the use case, while the concrete task is redefined later through spatial input.
