# Style Do / Don't

## Do

- Start sections with the section purpose.
- Explain why a choice matters before evaluating it.
- Keep the main subject concrete: system, interface, module, workflow, task.
- Use cautious comparison language.
- Keep the thesis vocabulary stable across chapters.
- Mention the implication of a result, not only the result itself.
- Use direct transitions between observation and interpretation.
- Preserve the balanced, technical tone of Chapters 2 to 5.

## Don't

- Don't write promotional prose.
- Don't turn Chapter 6 into a generic summary of the thesis.
- Don't use broad claims like "best", "perfect", or "game-changing" without evidence.
- Don't invent new high-level vocabulary unless it is clearly defined.
- Don't drift into a product pitch.
- Don't overfocus on implementation defects that do not matter to the chapter argument.
- Don't replace concrete reasoning with abstract praise.

## Specific anti-patterns for Chapters 6 and 7

- "This is an awesome system that can do everything."
- "This proves the system is better than existing workflows."
- "The results are obviously commercial-grade."
- "The discussion is just a recap of what each chapter said."
- "Future work is a list of remaining bugs."

## Preferred pattern

- State the built artifact.
- State the observed result.
- State what that result means.
- State where it is useful.
- State where it stops being useful.
- State what should happen next.
