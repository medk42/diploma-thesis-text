# Feedback Map

## Directives With Highest Priority
- Tie discussion back to the introduction and analysis.
- Keep the use cases central.
- Treat architecture as an important result, but not the only result.
- Explain what the results mean, not only what the system does.
- Keep the writing style concrete and mechanistic.
- Keep the conclusion modest and short.
- Split future work into engineering completion and longer-term research.
- Keep unsupported commercial or benchmark claims out.

## User Ideas To Integrate Cautiously
- The thesis can be framed as a reusable task-specific integration layer for repeated robot tasks.
- The system suggests a human-in-the-loop workflow where the human supplies spatial intent and the robot handles repetition.
- Export or compilation toward robot-side execution is a meaningful future direction.
- One-off welding and similar repetitive tasks are the strongest value proposition.
- Pick-and-place is useful as a flexibility demonstration, but it is not the strongest practical sweet spot.
- The architecture should be presented as experimental and extensible, not polished or final.

## Ideas That Need Softening
- `faster than teach pendant` should stay qualitative unless measured.
- `ideal for one-off welding` should become `particularly well suited to repetitive, frequently re-taught welding-like tasks`.
- `different kind of integrator` should become `deployment model closer to a reusable task-specific integration layer`.
- `human-robot integration` should become `human-in-the-loop workflow` or `shared task execution`.

## Ideas To Keep Out Unless Verified
- Any exact timing claim.
- Any claim about global tracking precision.
- Any claim about industrial-grade readiness.
- Any claim that the system replaces existing robot programming workflows in general.
- Any claim that UI bugs or Linux issues are thesis findings unless tied to source text.
