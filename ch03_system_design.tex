\chapter{System Design} \label{chapter:ch3_system_design}

\autoref{chapter:ch2_analysis} motivated a modular architecture with explicit interfaces, so that sensing, UI, robot backends, and task logic can be exchanged without rewriting the whole system. This chapter describes the design of this modular system and the main interfaces that enable communication between its components.

The system is implemented in C++, with all components built as modules that communicate exclusively through the runtime core. Most modules and the core are portable across Windows and Linux; platform-specific dependencies are isolated to a small set of hardware-facing components (notably camera acquisition).

\section{System at a glance} \label{sec:ch3_overview}

}
  \label{fig:ch3_four_parts}

The system is organized into four conceptual parts, shown in \autoref{fig:ch3_four_parts}. This separation helps to keep the design modular: stable contracts are defined once, while concrete implementations can be replaced as the system evolves.

\begin{enumerate}[label=(\arabic*), itemsep=0.3em]
    \item \textbf{Module contract.}
    This part defines \emph{what a module is} and how modules interact, without assuming a specific runtime implementation. It does so by specifying the module and core contracts.
    It specifies the communication primitives (publish/subscribe and request/response), message structure (small headers with optional shared blobs for large data), channel declaration, channel mapping, module lifecycle and abstract core API.
    All module code depends only on this contract.

    \item \textbf{Core implementation.}
    The core is the concrete runtime that \emph{implements} the module contract.
    It is responsible for loading modules, instantiating them, applying the channel mapping, routing messages, scheduling message processing, and managing shared memory for large payloads.
    Different core implementations are possible in principle (threads, processes, distributed), while still keeping the same module contract.

    \item \textbf{Interface contracts.}
    On top of the generic module communication, we define standard interfaces for our application domain (e.g., robot control, scene data, camera image) and for UI interaction (e.g., activation, use case execution, 3D visualization). These interfaces define \emph{what} messages mean and what behavior other modules can rely on, independent of a specific module's implementation of the required functionality.

    \item \textbf{Modules.}
    Finally, the concrete functionality is implemented as modules: camera acquisition, calibration, 6DoF pen input, scene detection, use cases, and robot execution. UI itself is also implemented as a module, since it is just another component that interacts with others through the defined interfaces.
    These modules use the standard interfaces and are hosted by the runtime core.
    As long as the interfaces are respected, modules can be added, removed, or replaced with minimal impact on the rest of the system.
\end{enumerate}

In the remainder of this chapter, we describe these parts in the same order: first the module contract, then the concrete core implementation, and finally the interface contracts. Implementation details of the actual modules are covered in \autoref{chapter:ch4_implementation}.

\section{Module contract} \label{sec:ch3_module_contract}

This section defines the \emph{module contract}: the core abstractions and communication rules that all modules and the runtime core must follow. The contract is independent of a concrete core implementation and provides a stable interface for developing modules.

\subsection{Roles and separation of responsibilities} \label{subsec:ch3_roles}

The contract separates responsibilities between modules and the core through two interfaces: \type{IModule} and \type{ICore}. Each module implements \type{IModule}, which defines its lifecycle, message handling entry points, serialization of module state, and other required methods. The core exposes \type{ICore}, which provides services to modules, including message routing, shared-memory allocation for large payloads, and system management operations (module discovery, instantiation, save/load, etc.).

As shown in \autoref{fig:ch3_four_parts}, modules do not communicate directly. All interaction goes through the core, which routes messages between modules according to an explicit channel mapping. When a module publishes data or sends a request/response, it calls the corresponding method on \type{ICore}; the core then delivers the message to the mapped recipient(s).

On the receiving side, the core enqueues the incoming message and later invokes the appropriate \type{IModule} callback from a core-managed worker thread. The contract therefore assumes concurrent execution: multiple worker threads may call into the same module instance at the same time (same or different channels, or overlapping message and request handling). Modules must treat these callbacks as multi-threaded entry points and synchronize access to their internal state accordingly. The concrete queueing and worker configuration is part of the module declaration and is described next.

\subsection{Module description contract}

  \label{fig:ch3_channel_mapping}
\end{figure}

To support different communication patterns, the contract provides two complementary primitives: publish/subscribe for streams and request/response for on-demand queries. Publish/subscribe is used for continuously produced data such as camera frames, while request/response is used when work should happen only when requested (e.g., a scene module performs detection only after receiving a request and returns detected objects). Modules therefore declare their inputs (consumers: subscribe or request) and outputs (producers: publish or response). Each channel is tagged by a \emph{channel type identifier} (string) that specifies the message type. An example wiring is shown in \autoref{fig:ch3_channel_mapping}.

Each module describes itself using a \type{ModuleInfo} structure, which includes:
\begin{itemize}
    \item \textbf{Module identifiers and UI text:} module type identifier, display name, and description,
    \item \textbf{Channel declarations:} lists of publish producers, response producers, subscribe consumers, and request consumers,
    \item \textbf{Instantiation policy:} auto-create flag,
    \item \textbf{Concurrency configuration:} number of worker threads (described below).
\end{itemize}

Channel declarations specify what the module can produce and what it requires as input. Importantly, communication is not a broadcast by default: connections are \emph{directed} between specific producer and consumer channels. The channel type identifier defines compatibility (message type), but the actual wiring is provided at module instantiation time via a channel mapping. The mapping selects, for each consumer channel, which producer channel(s) it is connected to (by specifying producer module and channel IDs). This enables flexible configurations---for example, in \autoref{fig:ch3_channel_mapping} the \type{Pen} module subscribes only to \type{Camera A} even though multiple cameras publish \type{camera\_stream}.

As noted in \autoref{subsec:ch3_roles}, message processing is executed by core-managed worker threads. Each module specifies how many worker threads it requires, allowing it to process multiple messages, requests or responses concurrently. Worker threads are created per module and are divided into two classes: \emph{prioritized} and \emph{regular}. Prioritized workers handle channels marked as prioritized; regular workers handle the remaining channels. This mechanism is crucial for UI responsiveness, because it allows latency-sensitive UI traffic to be processed in parallel with heavy computation (e.g., camera processing for pen tracking).

Finally, the auto-create flag marks modules that are instantiated automatically when the system starts (e.g., the UI module). Auto-created modules cannot depend on a manually provided channel mapping. In this thesis, we therefore restrict auto-created modules to \type{AUTO\_ALL} consumer channels: they may declare inputs, but only in a form that can be wired automatically to all currently available producers of the given type (see \autoref{subsec:ch3_channel_contract}).

\subsection{Channel contract} \label{subsec:ch3_channel_contract}

In the previous section, we described how modules declare their inputs and outputs using channels. In this section, we define the channel structures and the wiring constraints in more detail.

Channel definitions are provided \emph{per module} as part of its \type{ModuleInfo}: they describe the module's channel \emph{endpoints} (what it can produce and what it can consume). Concrete \emph{connections} between endpoints are created by the core when a module is instantiated, by binding consumer endpoints to selected producer endpoints according to the provided channel mapping.

There are two kinds of channels: producer channels (described by \type{Producer} structure) and consumer channels (described by \type{Consumer} structure).

A \type{Producer} contains:
\begin{itemize}
    \item channel type identifier (e.g., \type{camera\_stream}, \type{scene\_req\_resp}),
    \item user-facing channel name and description (used for UI visualization),
    \item prioritized flag,
    \item message queue capacity (maximum number of pending items on the receiver side).
\end{itemize}

The priority setting affects scheduling only for channel endpoints that \emph{receive inbound work} (i.e., where the core can enqueue something and later invoke an \type{IModule} callback). Therefore, for producer endpoints it matters only for \emph{response producers}, since they receive incoming requests; publish producers never receive inbound traffic, so their priority setting has no effect.

We define message queue capacity to bound the amount of pending work at the receiver. If messages or requests arrive faster than the module can process them, the queue capacity prevents unbounded buffering; the module can then decide how to handle overload (drop, replace, etc.), described in \autoref{subsec:ch3_message_lifecycle}.

A \type{Consumer} contains:
\begin{itemize}
    \item channel type identifier, name, description, prioritized flag, and message queue capacity (same meaning as in \type{Producer}),
    \item count policy (\type{SINGLE}, \type{RANGE} or \type{AUTO\_ALL}); specifies how many producer endpoints of the same type must be bound to this consumer endpoint when the module is instantiated,
    \item min/max bounds (used only for the \type{RANGE} policy).
\end{itemize}

When \type{ICore::addModule} is called, the core validates that the provided channel mapping satisfies these constraints (type match and required cardinality bounds); otherwise the module instantiation fails.

Consumer endpoints always receive inbound work: subscribe consumers receive published messages, and request consumers receive responses. If marked as prioritized, their inbound items are processed by prioritized worker threads; otherwise they use regular workers.

The count policy expresses wiring requirements. With \type{SINGLE}, exactly one producer endpoint of the given type must be connected. With \type{RANGE}, the consumer requires between \type{min} and \type{max} connected producer endpoints (inclusive). With \type{AUTO\_ALL}, the consumer is wired automatically to all currently available producer endpoints of the given type, which is primarily used for auto-created modules that cannot receive an explicit mapping. For example, the UI module can subscribe to all available \type{camera\_stream} producers via an \type{AUTO\_ALL} subscribe consumer, combining the convenience of broadcast with the explicit wiring model used elsewhere.

Although the same \type{SINGLE}/\type{RANGE}/\type{AUTO\_ALL} policy is used for both subscribe and request consumers, its effect differs by communication primitive:
\begin{itemize}
    \item \textbf{Subscribe consumers (publish/subscribe):} the policy directly defines the stream fan-in --- i.e., how many publishers feed this input (one, several, or all). Published messages are effectively broadcast to all wired subscribers, and each delivered message carries the sender endpoint identifier so the receiver can distinguish origins if needed.
    \item \textbf{Request consumers (request/response):} the policy defines how many responders are available as potential request targets, but requests themselves are still \emph{directed}: each request names exactly one concrete responder (there is no request broadcast). For \type{SINGLE}/\type{RANGE}, the eligible targets are fixed when the module is instantiated and they are provided to the module; with \type{AUTO\_ALL}, the module is created without an explicit target list and must discover candidate responders at runtime via the core reflection API (see \autoref{subsec:ch3_reflection_orchestration}).
\end{itemize}

In request/response, the endpoint names reflect the \emph{inbound} direction:
a \emph{request consumer} is the caller-side endpoint that \emph{sends} requests and later \emph{consumes} the matching response,
while a \emph{response producer} is the callee-side endpoint that \emph{receives} requests and then \emph{produces} the response.


\subsection{Message structure: header and optional shared blobs}

The message model is designed to efficiently handle both small control messages and large data payloads. All messages are represented by the \type{MessageHeader} structure, which is used uniformly for publish/subscribe as well as request/response communication.

The header contains:
\begin{itemize}
    \item \textbf{Small copied payload} given by a data pointer and length. This part is intended for small, trivially copyable control data (e.g., pen intent, state updates) and is copied from the producer to each consumer.
    \item \textbf{Optional blob array} (pointer + count), representing one or more shared payloads used for large data (e.g., images, point clouds, serialized structures) that should not be copied repeatedly.
    \item \textbf{Request/response metadata:} \type{id\_} to match requests with responses and a \type{success\_} flag indicating whether the request was processed successfully at the communication layer.
    \item \textbf{Timestamp} (\type{timestamp\_ns\_}) for tracing / profiling.
\end{itemize}

In practice, most messages consist only of the small copied payload. When large data needs to be transmitted, the message includes one or more blobs. A producer allocates a blob, writes the payload into it, and sends a message that references the blob. Consumers can then read the payload directly and may forward the same blob further (e.g., camera $\rightarrow$ calibration $\rightarrow$ pen tracking) without copying the underlying data.

\subsection{Shared payload ownership and allocator contract} \label{subsec:ch3_allocator_contract}

Large payloads are represented by \type{SharedDataBlob}, which acts as an RAII handle to core-managed shared data (\type{ISharedData}). Modules never free raw memory directly. Instead, they only copy, store, and pass \type{SharedDataBlob} handles.

A \type{SharedDataBlob} stores a pair \type{(ISharedData*, IAllocator*)}. On construction and on copy, it calls \type{IAllocator::addOwner(data)}; on destruction (and when overwritten by a different assignment), it calls \type{IAllocator::removeOwner(data)}.
This makes shared payload ownership effectively reference-counted at the contract level: the underlying shared memory is released by the core allocator once the last \type{SharedDataBlob} handle is destroyed.

Blobs are allocated through core-provided allocators (\type{IAllocator}). A module can request an allocator via \type{ICore::createDynamicAllocator()} or \type{ICore::createBufferAllocator(slot\_size, slot\_count)} and then allocate blobs using \type{IAllocator::allocate()} (or helper methods such as \type{allocateFromData}). We distinguish two allocator types at the contract level:
\begin{itemize}
    \item \textbf{Dynamic allocator:} supports variable allocation sizes (suitable for irregular payloads such as scene snapshots) at the cost of relying on system allocations for each request. Dynamic allocator returns invalid blobs when the underlying system allocation fails.
    \item \textbf{Buffer allocator:} preallocates a fixed number of equally-sized slots (suitable for streaming payloads with known bounds, such as camera frames), requiring a known maximum size in advance and offering only a limited number of slots. Buffer allocator returns invalid blobs when it is out of free slots.
\end{itemize}

Allocator and blob lifetime are owned by the core to ensure consistent memory management across modules and to allow safe sharing between independently developed components. Modules are expected to delete allocators they no longer need using \type{ICore::deleteAllocator}. In this thesis, allocator cleanup is handled by the common \type{BaseModule} helper (\autoref{subsec:ch3_basemodule}) so that module implementations do not need to manage allocator lifetimes manually.


\subsection{Communication primitives} \label{subsec:ch3_comm_primitives}

This section describes the runtime semantics of the two communication primitives. Channel declaration and mapping rules are defined by \type{ModuleInfo} and the channel contract (\autoref{subsec:ch3_channel_contract}). Throughout the system, a channel endpoint is identified by \type{(module\_id,local\_channel\_id)} (represented by \type{ChannelIdentifier}): the module id selects the running module instance, and the local id selects one of its declared channel endpoints (from the appropriate publish/subscribe/request/response list for the given call).

\minihead{Publish/subscribe} is used for streams and events. A module publishes a message by calling \type{ICore::sendMessage(source\_channel, message)} with \type{source\_channel} identifying its producer channel. The core routes the message to all subscribe consumers that are connected to this producer by the current mapping. Delivery is best-effort and may be affected by the receiver's ingress and queueing policy (\autoref{subsec:ch3_message_lifecycle}).

\minihead{Request/response} is used for directed on-demand interactions. The requester calls \type{ICore::sendRequest(source\_channel, target\_channel, message)} and specifies both its local request consumer channel (\type{source\_channel}) and the \emph{concrete} target response producer channel (\type{target\_channel}) it is requesting from. The core delivers the request to the receiver by invoking \type{IModule::processRequest(response\_producer\_id, source\_channel, message)}. The receiver returns the response payload as \type{ResponseData}; the core (or the module hosting wrapper) then sends exactly one response back to the requester via \type{ICore::sendResponse(source\_channel, target\_channel, message)} with the response producer as \type{source\_channel} and the original request consumer as \type{target\_channel}. This design prevents modules from silently ``forgetting'' a request and ensures a well-defined one-request/one-response flow at the transport level.

\minihead{Synchronous vs asynchronous aspects.} On the receiver side, \type{IModule::processRequest} produces the response data synchronously (as a return value). From the requester perspective, request/response is still asynchronous: the response is delivered later via \type{IModule::processResponse} and must not be assumed to arrive immediately or on the same thread.

\minihead{Success and failure.} The field \type{MessageHeader.success\_} represents transport-level outcome (e.g., request dropped or rejected, invalid routing). Domain-level success (e.g., detection failed) is encoded in the payload and is specific to each channel type.

The ingress decision point \type{IModule::onIngress} is a required part of the contract and must be \emph{non-blocking}. It is executed on the core's delivery path and is therefore intended only for a fast admission-control decision under load. The core passes a \type{QueueStatus} describing whether the destination queue has available capacity. The module may then return an \type{IngressDecision} to either accept, drop, or (when allowed) drop/replace queued work (e.g., keep only the latest frame for a video stream). \subsection{Reflection and orchestration contract} \label{subsec:ch3_reflection_orchestration}

Besides message passing, the contract exposes a small \emph{reflection and orchestration} API intended mainly for UI and other controller-style modules. Its purpose is to allow a module to (i) discover what module types exist, (ii) inspect which module instances are currently running and how they are wired, and (iii) create, remove, save, and restore module graphs. This functionality is provided through the \type{ICoreControl} part of \type{ICore}.

\minihead{Loaded module catalog (module types).}
The core exposes a catalog of all module \emph{types} available in the runtime (e.g., discovered plugins). The pair \type{getLoadedModulesCount()} and \type{getLoadedModulesInfo(loaded\_module\_id)} allows querying each type's immutable \type{ModuleInfo} (declared channels, display metadata, auto-create flag, worker configuration, etc.). UI uses this catalog to present the user with a ``palette'' of modules that can be instantiated.

\minihead{Running module instances (created modules).}
While the loaded catalog lists available types, the runtime also maintains a set of created \emph{instances}. Each instance is assigned a \emph{monotonically increasing} running module identifier when created; identifiers are never reused. If an instance is later removed, its identifier becomes a tombstone that permanently refers to a non-existing module.

This contract is reflected in \type{RunningModuleInfo}, which contains an \type{exists\_} flag and a reference to the instance's \type{ModuleInfo} (\type{module\_info\_}). The method \type{getRunningModulesCount()} returns the total number of identifiers issued over the lifetime of the core object (not the number of currently alive instances). Callers therefore iterate \type{running\_module\_id} from \type{0} to \type{getRunningModulesCount()-1} and query each entry via \type{getRunningModulesInfo(running\_module\_id)}, filtering out non-existing modules using \type{exists\_}. Running module identifiers must not be assumed stable across program sessions or save/load.

\minihead{Wiring query (who is connected to whom).}
For each running module instance, \type{getRunningModulesChannelMap(running\_module\_id)} returns its current input wiring: for every subscribe and request consumer channel, the core provides the list of bound producer endpoints (identified by \type{ChannelIdentifier}, that is a pair of the producer module ID and the local producer channel ID). This is the core reflection primitive needed to visualize or debug the module graph and to implement higher-level orchestration logic (e.g., configuration panels that depend on which sources are currently connected).

\minihead{Existing producers by channel type (discovery).}
For reflection and UI/orchestration, the core provides
\type{getExistingPublishChannelsByName(channel\_type\_identifier)} and
\type{getExistingResponseChannelsByName(channel\_type\_identifier)}.
These functions return the set of currently known producer endpoints (\type{ChannelIdentifier}) for the given channel type identifier (\type{string}).

For \type{SINGLE} and \type{RANGE} request consumers, the requester typically does \emph{not} need discovery: the core provides the concrete mapped endpoints as part of \type{InputChannelMapInfo} during module creation, and the module uses those \type{ChannelIdentifier}s as request targets.
Discovery is primarily needed for \type{AUTO\_ALL} request consumers, where the target set is not known at instantiation time (and therefore is not enumerated in \type{InputChannelMapInfo}).

\minihead{Mapping state change indicator.}
Because the running set and its wiring may change over time (modules created/removed), the core exposes \type{getModulesMappingStateId()}. The returned value is a monotonically increasing identifier that changes whenever the mapping topology changes (modules are added or removed). UI modules can use it as a cheap change detector to refresh cached views.

\minihead{Orchestration entry points (create/remove/dependencies).}
The contract allows a controller to create and remove modules dynamically. A new instance is created using \type{addModule(loaded\_module\_id, channel\_map\_info)}. The core validates the provided mapping against the declared channel contract (channel type identifiers and the consumer count policy such as single/range/auto-all) and rejects invalid wiring. Existing modules may be removed via \type{removeModuleById(running\_module\_id, recursive)}; when recursive removal is requested, the core may also remove dependent modules (as defined by the connection types), and \type{collectDependencies(running\_module\_id)} can be used to inspect this dependency set in advance.

\minihead{Persistence entry points (save/load).}
Finally, the contract provides \type{save()} and \type{load(data, size)} to persist and restore the \emph{program state} consisting of created module instances, their wiring, and optional per-module serialized state (as returned by module \type{save()} / \type{load()}). The binary format is owned by the core. Loading reconstructs the module graph from the loaded catalog; running and loaded module identifiers may change, and modules must not rely on ID stability across save/load. Allocators are not persisted; modules are expected to recreate any allocators they need after loading.



\subsection{Module lifecycle and implementer rules} \label{subsec:ch3_module_rules}

This section summarizes (i) what runtime inputs the core provides to a newly created module instance, (ii) the expected lifecycle steps, and (iii) the key rules that module implementations must follow to be safely hostable by any compliant core.

\minihead{Construction inputs.}
A module instance is created through the core's hosting mechanism (e.g., a factory provided by the module package). The module-facing contract assumes the following runtime inputs are provided at creation time:
\begin{itemize}
    \item \textbf{Data path (\type{const char* data\_path}).} Optional filesystem location for module resources; may be \type{nullptr} if no user-provided data directory is available for the module. If the module needs to access this path after construction, it must copy the string into its own storage.
    \item \textbf{Core interface (\type{ICore* core}).} Used to send messages/requests, allocate shared payloads, and for reflection and orchestration services.
    \item \textbf{Input mapping (\type{InputChannelMapInfo channel\_map\_info}).} Mapping of this module's input channels (subscribe/request) to concrete producer endpoints. The module may use it to learn what it is wired to (especially request targets). The passed arrays are not owned by the module; if the module needs this information after construction, it must copy it into its own storage.
    \item \textbf{Logger (\type{logging::ILogger* logger}).} Core-provided logging sink for diagnostics.
    \item \textbf{Module identifier (\type{uint64\_t module\_id}).} Runtime ID assigned by the core. It is required to form \type{ChannelIdentifier} values (module + channel) when sending via \type{ICore}. Helper layers (e.g., \type{BaseModule}) may encapsulate this, but the ID is still part of the API.
\end{itemize}
Each module must expose its static description via \type{IModule::getModuleInfo()}, which must not return \type{nullptr}. In the typical implementation used in this thesis, the description is defined as a static \type{ModuleInfo} in the module package and a \type{const ModuleInfo*} is passed to the module instance during construction so that \type{getModuleInfo()} can return it. A common helper (\type{BaseModule}, \autoref{subsec:ch3_basemodule}) implements this boilerplate for module authors.

\minihead{Lifecycle.}
A compliant core follows these steps when creating and running a module instance:
\begin{enumerate}
    \item \textbf{Construct.} The host constructs the module instance (via the hosting mechanism) and provides the runtime inputs listed above.
    \item \textbf{Validate.} The core checks \type{IModule::valid()}. If it returns false, the module does not enter the runtime and is destroyed.
    \item \textbf{Start.} If valid, the core calls \type{IModule::threadStart(timeout)}. If start fails, the core may call \type{threadStop(timeout)} for cleanup and then destroys the instance.
    \item \textbf{Run.} After a successful start, the core may deliver messages according to \autoref{subsec:ch3_message_lifecycle}: ingress decisions via \type{onIngress} followed by processing callbacks \type{processMessage}, \type{processRequest}, and \type{processResponse}.
    \item \textbf{Stop and destroy.} When removing a module (or shutting down), the core calls \type{IModule::threadStop(timeout)}. After it returns, the core will not invoke further callbacks on that instance and finally destroys it.
\end{enumerate}

\minihead{Rules for module implementers.}
\begin{itemize}
    \item \textbf{Restricted core usage during construction and teardown.}
    In the core implementation used in this thesis, module graph mutations (create/remove/load) are serialized under an exclusive \type{core\_mutex\_} lock.
    Because message sending and \type{ICoreControl} queries acquire the same mutex (shared), calling
    \type{ICore::sendMessage}, \type{ICore::sendRequest}, \type{ICore::sendResponse} or any \type{ICoreControl} method
    from the core thread during module creation/removal (\type{ctor}, \type{threadStart}, \type{threadStop}, \type{dtor})
    can deadlock and is therefore forbidden.

    The only core interaction explicitly allowed in these phases is allocator management
    (\type{createDynamicAllocator}, \type{createBufferAllocator}, \type{deleteAllocator}) and logging via the provided \type{logging::ILogger},
    because allocator operations are synchronized independently of the module graph.

    Module-owned background threads may still attempt to send during teardown; the current core mitigates this by marking the module as being destroyed
    (and rejecting sends) and by temporarily releasing the exclusive lock while \type{threadStop} executes.
    Module code must therefore not rely on successful message sending or core reflection during teardown and must tolerate rejected sends.


    \item \textbf{\type{threadStart}/\type{threadStop} bound the active lifetime.} Start any internal background work in \type{threadStart}. Stop and join all module-owned background threads in \type{threadStop} before returning. The \type{timeout} parameter is a cooperative shutdown/initialization budget used for diagnostics; modules should keep start/stop bounded and must not block indefinitely, but \type{threadStop} must not return while module-owned threads are still running.
    \item \textbf{\type{onIngress} must be non-blocking.} \type{IModule::onIngress} runs on the core delivery path and must return immediately. It may only make an overload decision using \type{QueueStatus} and return an \type{IngressDecision}. It must not perform heavy work, wait for other threads, or depend on external services.
    \item \textbf{Processing callbacks are concurrent.} The core may invoke processing callbacks from multiple worker threads concurrently on the same module instance. Implementations must treat \type{processMessage}, \type{processRequest}, and \type{processResponse} as multi-threaded entry points and provide the necessary synchronization for internal state.
    \item \textbf{Blocking is allowed, but do not exhaust the worker pool.}
    Some modules (e.g., use case orchestration) intentionally implement a synchronous flow inside \type{processRequest} and may wait for other modules (e.g., send a request and wait until the matching \type{processResponse} arrives). This is permitted, but the module must provision enough worker threads so that the awaited response can still be processed.
    If all available workers of the relevant priority class are blocked waiting, responses will remain queued and the system can deadlock (e.g., one regular worker blocks waiting for a response that is queued and requires a regular worker to run \type{processResponse}).
    As a practical rule: if a callback may block waiting for a response delivered through the same worker class, allocate at least two workers in that class, or ensure the response is handled by a different priority class with at least one unblocked worker.

    \item \textbf{Do not keep raw message pointers.} The copied payload (\type{MessageHeader.data\_}) and blob pointer array are only valid for the duration of the callback. If a module needs to retain either, it must copy the payload or retain the corresponding \type{SharedDataBlob} handles rather than storing raw pointers.
    \item \textbf{Be robust to transport-level failures.} Message delivery is best-effort; modules must tolerate drops/rejections and request failures reported via \type{MessageHeader.success\_ = false} (e.g., invalid routing or module removal).
    \item \textbf{Do not assume stable IDs.} Loaded module identifiers and running module identifiers (including \type{module\_id}) are runtime artifacts. They may change across program sessions and after \type{load()}, and removed modules may leave gaps. Persistent state must not rely on ID stability.
    \item \textbf{Allocator discipline.} Shared payloads must be allocated only through core-provided \type{IAllocator} instances, and allocator lifetimes must be managed to avoid leaks. In this thesis, common helper utilities (e.g., \type{BaseModule}, \autoref{subsec:ch3_basemodule}) encapsulate allocator creation/deletion and message metadata setup.
    \item \textbf{No exceptions across the boundary.} All contract callbacks are \type{noexcept}; module code must not let exceptions escape these entry points.
\end{itemize}

\minihead{Optional capability discovery.}
Some functionality is defined as an additional C++ interface on top of the base module contract (e.g., \type{IActivableModule} for activation, \type{IUsecaseModule} for use case execution). These interfaces are intentionally \emph{not} part of \type{IModule}, because they are higher-level and would make the base contract unnecessarily large and hard to implement. Instead, the system provides \emph{wrappers} that implement \type{IModule} and translate module-level messages (activation requests, status queries, etc.) into direct calls to the optional interface (e.g., \type{ActivationWrapper} $\rightarrow$ \type{IActivableModule}). Wrappers are stackable (e.g., activation wrapper on top of a use case wrapper), therefore higher-level code sometimes needs to access the underlying optional interface implemented by the wrapped module. For this purpose, modules expose optional capabilities via a query mechanism (e.g., \type{query\_capability(typeid(T))}), which allows wrappers to discover that the module they wrap implements \type{IActivableModule}/\type{IUsecaseModule}. Concrete wrapper and protocol contracts are described in \autoref{sec:ch3_interface_contracts}.

\subsection{BaseModule} \label{subsec:ch3_basemodule}

In this thesis, all concrete modules inherit from \type{BaseModule}, a small convenience layer on top of \type{IModule}. It is \emph{not} a part of the minimal module contract, but it standardizes the boilerplate that nearly every module needs (message sending, allocator lifetime, access to construction inputs) and keeps module implementations focused on domain logic.

\minihead{Construction inputs and persistence.}
\type{BaseModule} mirrors the core-provided construction inputs described in \autoref{subsec:ch3_module_rules}. It stores references to \type{ICore} and \type{logging::ILogger}, stores the runtime \type{module\_id}, stores a pointer to the module's static description \type{ModuleInfo}, and copies the optional \type{data\_path} into an internal \type{std::string}. Importantly, it also \emph{copies} the provided \type{InputChannelMapInfo} into internal vectors, because the input mapping passed at creation time is not guaranteed to outlive the constructor. The copied mapping can be queried later via \type{getSubscribeChannelInfo()} and \type{getRequestChannelInfo()}.

\minihead{Message sending helpers.}
\type{BaseModule} provides thin wrappers around the three core send functions:
\begin{itemize}
    \item \type{sendMessage(publish\_producer\_id, message)} fills \type{MessageHeader.timestamp\_ns\_} and sets \type{MessageHeader.success\_ = true} (the \type{success\_} flag is only meaningful for responses), then calls \type{ICore::sendMessage} with a \type{ChannelIdentifier} constructed from the stored \type{module\_id} and the provided producer channel id.
    \item \type{sendRequest(request\_consumer\_id, target\_channel, message)} generates a fresh request ID (\type{message.id\_ = request\_id\_++}), fills \type{timestamp\_ns\_} and \type{success\_ = true} (the \type{success\_} flag is only meaningful for responses), calls \type{ICore::sendRequest}, and returns the generated ID to the caller.
    \item \type{sendResponse(response\_producer\_id, target\_channel, request\_id, message)} fills \type{message.id\_} with the provided request ID, fills \type{timestamp\_ns\_}, and calls \type{ICore::sendResponse}. The \type{success\_} flag must be set by the caller to indicate transport-level success or failure of the request processing.
\end{itemize}
These helpers are intentionally simple: they only ensure consistent metadata and correct \type{ChannelIdentifier} construction; they do not change routing semantics defined by the core.

\minihead{Allocator RAII wrappers.}
To simplify shared payload allocation (\autoref{subsec:ch3_allocator_contract}), \type{BaseModule} exposes \type{createDynamicAllocator()} and \type{createBufferAllocator(slot\_size\_bytes, number\_of\_slots)}. Both return an \type{AllocatorPtr} (\type{std::unique\_ptr<IAllocator, deleter>}) whose custom deleter calls \type{ICore::deleteAllocator}. This makes allocator ownership explicit and avoids leaks caused by forgotten release. Allocator operations are independent of message routing and may be used during creation/removal.

\minihead{Convenience utilities.}
\type{BaseModule} also provides:
\begin{itemize}
    \item \type{log(LogType, ...)} convenience overloads forwarding to the core-provided \type{ILogger},
    \item \type{getDataPath()} access to the resolved optional data path,
    \item name-to-index helpers \type{getPublishChannelByName}, \type{getResponseChannelByName}, \type{getSubscribeChannelByName}, \type{getRequestChannelByName}, to avoid hardcoding numeric channel indices that may change when the \type{ModuleInfo} arrays are edited.
\end{itemize}

\minihead{What it does not do.}
\type{BaseModule} does not implement queueing, worker scheduling, ingress decisions, or any threading policy; those remain the responsibility of the core (\autoref{sec:ch3_core}). It also does not guarantee deadlock-freedom or correctness of cross-module interactions. It only encapsulates repeated mechanical steps and provides a consistent module-author API.

Finally, \type{BaseModule} implements \type{IModule::getModuleInfo()} by returning the \type{ModuleInfo} pointer provided at construction. This ensures that higher-level infrastructure can always obtain the module description through the contract entry point, while concrete modules typically delegate the storage of that pointer to \type{BaseModule}.

The previous section defined the module contract in an implementation-independent way: what a module is, how it declares channels, how messages are represented, and what services the runtime must provide through \type{ICore}. \minihead{Reading note.}
This section is intentionally more implementation-oriented than the preceding contract: it documents one concrete in-process core that hosts modules and routes messages.
To keep the description traceable, we occasionally mention key C++ interfaces and internal structures; readers interested mainly in the higher-level design can skim such identifiers and focus on the described mechanisms.

\minihead{Scope and key design choices.}
The runtime is an \emph{in-process} host: modules run as dynamically loaded shared libraries (\type{.dll} on Windows, \type{.so} on Linux) and communicate only through the core.
Message delivery is asynchronous and uses per-module queues and worker threads.
Large payloads are passed via core-managed shared blobs allocated through \type{IAllocator} instances; the core owns allocator objects and shared payload lifetime so independently developed modules can share large data without copying and without cross-library memory ownership issues.

\minihead{Core split.}
The implementation is split into two cooperating parts:
(i) the \emph{central core}, which maintains global runtime state (loaded module catalog, running module instances, channel mappings/routing tables, reflection/orchestration API, and persistence), and
(ii) a \emph{per-module hosting wrapper}, which owns a single instantiated \type{IModule} and implements the concrete delivery machinery required by the contract (ingress decisions, payload/blob-handle copying into queues, worker threads, and invoking \type{processMessage}/\type{processRequest}/\type{processResponse}).

\subsection{Module packaging and hosting boundary} \label{subsec:ch3_core_dll_boundary}

Modules are packaged as dynamically loaded shared libraries: \type{.dll} on Windows and \type{.so} on Linux. The runtime core discovers available modules by scanning a user-provided \type{modules\_dir} directory and attempting to load each file as a shared library.

\minihead{Discovery and OS loading.}
On Windows the loader uses \type{LoadLibraryA}/\type{GetProcAddress}; on Linux it uses \type{dlopen}/\type{dlsym}. The filename stem is also used to derive the module's data folder. For \type{module\_a.dll} we look for \type{data\_dir/module\_a}. On Linux we additionally strip the conventional \type{lib} prefix, so \type{libmodule\_a.so} maps to \type{data\_dir/module\_a}. If the folder does not exist, the module receives \type{data\_path = nullptr}.

\minihead{Required C-style exports.}
Each module library must export a small fixed API with stable symbol names (i.e., exported with \type{extern "C"}). The core resolves these symbols and rejects the library if any are missing:
\begin{itemize}
  \item \type{readPluginApiVersion()} --- compatibility check against \type{CORE\_API\_VERSION}.
  \item \type{readModuleInfo()} --- returns a pointer to a statically allocated \type{ModuleInfo}.
  \item \type{createModule(data\_path, core, channel\_map\_info, logger, module\_id)} --- creates one running module instance.
  \item \type{destroyModule(instance)} --- destroys the instance created by \type{createModule}.
\end{itemize}
This API is the only thing the core assumes about a module library when loading it.

\minihead{\type{IDllModule} interface.}
\type{createModule(...)} does not return a pointer to \type{IModule}. Instead, it returns a pointer to an object implementing \type{dll::IDllModule}. This is the core-facing instance interface: it is intentionally small and matches exactly what the core needs to drive a running module:
\begin{itemize}
  \item three entry points for incoming traffic: \type{processMessage}/\type{processRequest}/\type{processResponse},
  \item lifecycle of module-owned worker threads: \type{threadStart}/\type{threadStop},
  \item persistence hooks: \type{save}/\type{load} (as byte blobs).
\end{itemize}
Inside the library, the returned object is typically a hosting wrapper (e.g., \type{DllModuleWrapper}) that owns an instantiated \type{IModule} and implements the queueing/worker-thread machinery required by the contract. The core never needs to know the concrete \type{IModule} type (nor link against it); it only stores and calls \type{IDllModule}.

\minihead{Ownership rules across the library boundary.}
The instance returned by \type{createModule} is allocated inside the module library and must also be destroyed inside that same library. The core therefore never calls \type{delete} on the returned pointer; it always destroys it via \type{destroyModule}. This avoids allocator/runtime mismatches across the boundary (especially on Windows) and keeps ownership explicit.

For the same reason, anything with non-trivial ownership (e.g., \type{std::string}, \type{std::vector}) is not transferred across the boundary as an owned C++ object. Variable-sized \emph{outputs} are returned as \type{message::SharedDataBlob} allocated by a core-owned \type{IAllocator}. This is why persistence is exposed on \type{IDllModule} as \type{save(allocator)\,$\rightarrow$\,SharedDataBlob} and \type{load(data,size)}: inside the library we can work with the higher-level \type{ISerializableModule::SaveData} representation, but the boundary itself stays in terms of raw bytes.

\minihead{What \emph{is} passed directly.}
Small POD-like structures (\type{ChannelIdentifier}, \type{MessageHeader}, \type{InputChannelMapInfo}) are passed directly. For variable-sized \emph{inputs}, a pointer and size is used; the pointer is borrowed for the duration of the call, so if the callee needs the data later, it must copy it.

\minihead{Where this leads next.}
With the packaging boundary defined (how module libraries are found, validated, and how instances are created/destroyed), the next section describes the startup sequence that builds the loaded module catalog and performs the initial auto-create pass.




\subsection{Startup sequence} \label{subsec:ch3_core_startup}

Core initialization is performed once via \type{Core::initialize(modules\_dir, data\_dir)}. The method builds the \emph{loaded module catalog} (all available module \emph{types}) and then performs an auto-create pass that instantiates selected module \emph{instances}.

\minihead{Step 1 --- Discover and load module libraries.}
The core iterates over \type{modules\_dir} and attempts to load each entry as a shared library. For each successfully loaded library it:
\begin{enumerate}
    \item builds the optional per-module data path (\type{data\_dir/\{module\_name\}}, see \autoref{subsec:ch3_core_dll_boundary}),
    \item resolves the required exported symbols (plugin API),
    \item checks \type{readPluginApiVersion() == CORE\_API\_VERSION} (otherwise the module is rejected),
    \item stores the loaded module in \type{loaded\_modules\_}: this catalog is constant for the lifetime of the core instance (modules are not unloaded).
\end{enumerate}

\minihead{Step 2 --- Auto-create pass.}
After all libraries are loaded, the core instantiates all modules that declare \type{ModuleInfo.auto\_create\_ = true} (single instance, created by the core directly).

If an auto-created module fails to create or start, \type{initialize} reports failure; library loading failures are logged and ignored.



\subsection{Creating a module instance} \label{subsec:ch3_core_create_instance}

Module instances are created in three places: during the startup auto-create pass, via \type{addModule} (e.g., from the UI module), and during core state load (persistence is covered later in \autoref{subsec:ch3_core_persistence}).
All three end up executing the same core procedure: choose a new runtime \type{module\_id}, call the module library's \type{createModule}, start the module threads, and then register the module into the core's routing tables.

\minihead{Inputs to creation.}
Creating an instance requires:
\begin{itemize}
    \item the module \emph{type} to instantiate (index into the loaded module catalog, \type{loaded\_module\_id}),
    \item a concrete input mapping \type{InputChannelMapInfo} that selects which existing channels the module consumes (subscribe/request).
\end{itemize}

\minihead{Mapping pre-check (\type{addModule} only).}
For \type{addModule}, the core first validates the provided \type{InputChannelMapInfo} against the module's \type{ModuleInfo} definition (see \autoref{subsec:ch3_core_mapping_validation}).
Invalid mappings are rejected before any instance is created.
For the auto-create pass this check is bypassed: auto-created modules are expected to use only \type{AUTO\_ALL} consumers and can therefore be instantiated with an empty explicit mapping.

\minihead{Create-and-start procedure.}
The actual creation is performed by \type{createAndStartModule(loaded\_module\_id, channel\_map\_info, timeout)} and follows these steps:
\begin{enumerate}
    \item Resolve \type{data\_path}: if the per-module folder exists under \type{data\_dir}, pass its path, otherwise pass \type{nullptr}.
    \item Allocate a fresh runtime \type{module\_id}. Runtime IDs are monotonic and are not reused (the ID is effectively the index into \type{running\_modules\_}).
    \item Create a per-instance logger (\type{ModuleLogger}) bound to the module type name and \type{module\_id}.
    \item Call the module library export \type{createModule(data\_path, this, channel\_map\_info, \&module\_logger, module\_id)}.
    The returned pointer is immediately wrapped into a smart pointer with a custom deleter that calls \type{destroyModule} (so the instance is always destroyed inside the same library).
    \item If creation succeeded, start the module-owned worker threads by calling \type{IDllModule::threadStart(timeout)}.
\end{enumerate}

\minihead{Failure behavior.}
If \type{createModule} returns \type{nullptr}, creation fails immediately.
If \type{threadStart} fails, the core attempts \type{threadStop(timeout)} and the instance is destroyed (via the wrapper/deleter), then creation fails.
In both cases the failure is logged and the core state remains unchanged.

\minihead{Success behavior.}
On success the instance is inserted into \type{running\_modules\_} and the core integrates it into its global view:
\begin{itemize}
    \item register the module's channels for discovery,
    \item update routing/dependency tables by registering the instance's input mapping.
\end{itemize}

For \type{addModule}, a successful creation also increments \type{module\_mapping\_state\_id\_} so the UI can detect that the module graph changed.

The next section describes the validation rules and the internal mapping tables that these registration steps populate.


\subsection{Mapping validation and internal mapping tables} \label{subsec:ch3_core_mapping_validation}

The core stores the current module graph explicitly. For each producer channel it keeps a list of connected consumers, and for each consumer channel it keeps the list of producers it is connected to.
These tables drive publish/subscribe routing and are also used to answer “who depends on whom?” when removing modules.

\minihead{Validation against \type{ModuleInfo}.}
When creating an instance via \type{addModule}, the core first checks that the provided \type{InputChannelMapInfo} matches the target module's \type{ModuleInfo} (\type{checkChannelMapValidity}):
\begin{itemize}
    \item \textbf{Consumer entry counts match:} the mapping must provide exactly as many subscribe/request consumer entries as the module declares.
    \item \textbf{Cardinality rules per consumer:}
    \begin{itemize}
        \item \type{SINGLE}: exactly one \type{ChannelIdentifier} must be provided,
        \item \type{RANGE(min,max)}: the number of provided identifiers must be within \type{[min,max]},
        \item \type{AUTO\_ALL}: no identifiers are provided (count must be zero).
    \end{itemize}
    \item \textbf{Referenced producers exist:} every provided \type{ChannelIdentifier} must point to an existing running module (not destroyed) and a valid producer channel index (publish producer for subscribe mappings, response producer for request mappings).
    \item \textbf{Type identifiers match:} for each referenced producer channel, the producer's \type{channel\_type\_identifier\_} must equal the consumer's expected \type{channel\_type\_identifier\_} from \type{ModuleInfo}.
\end{itemize}

\minihead{Per-instance mapping tables.}
For every running instance, the core keeps four per-module tables inside \type{structures::ModuleData}:
\begin{itemize}
    \item \type{mapping\_publish\_} and \type{mapping\_response\_}:
    for each producer channel ID, a list of connected consumer endpoints \type{(module\_id, consumer\_channel\_id)}. This is used for cascade removal (to find dependent modules) and for publish routing (method \type{sendMessage} iterates over the producer \type{mapping\_publish\_} entry)
    \item \type{mapping\_subscribe\_} and \type{mapping\_request\_}:
    for each consumer channel ID, the list of producer \type{ChannelIdentifier}s this consumer is currently connected to.
    This reverse view is used mainly for inspection, and for saving/restoring mappings.
\end{itemize}
Note that \type{ChannelIdentifier} is reused for both directions as a compact \type{(module\_id, channel\_id)} pair (producer in one context, consumer endpoint in the other).

\minihead{Global channel indexes.}
In addition to per-module tables, the core maintains global maps keyed by \type{channel\_type\_identifier\_}:
\begin{itemize}
    \item \type{existing\_publish\_channels\_}: all currently available publish producers (used for reflection and for wiring new \type{AUTO\_ALL} subscribe consumers),
    \item \type{existing\_response\_channels\_}: all currently available response producers (used similarly),
    \item \type{existing\_subscribe\_auto\_all\_channels\_}: all subscribe consumers declared as \type{AUTO\_ALL} (used only for wiring new producers to existing \type{AUTO\_ALL} inputs),
    \item \type{existing\_request\_auto\_all\_channels\_}: same for request consumers.
\end{itemize}
These are updated on creation in \type{registerModuleChannelNames} and on removal via \type{removeFromExistingMap}.

\minihead{Wiring a new instance.}
After a new instance is created, started, and inserted into \type{running\_modules\_}, the core makes it visible and wires it in two steps:

\begin{enumerate}
    \item \textbf{Register channel types for discovery (\type{registerModuleChannelNames}):}
    the instance's publish/response producers are inserted into \type{existing\_publish\_channels\_}/\type{existing\_response\_channels\_}, and any \type{AUTO\_ALL} subscribe/request consumers are recorded in \type{existing\_subscribe\_auto\_all\_channels\_}/\type{existing\_request\_auto\_all\_channels\_}.
    These maps are the core's global ``what exists right now'' indexes:
    they answer reflection queries (``list all channels of type X'') and they are also the input for incremental \type{AUTO\_ALL} wiring when modules are added later.
    On removal, the same indexes are updated (\type{removeFromExistingMap}) so discovery stays consistent.

    \item \textbf{Build routing and dependency edges (\type{registerModuleConnections}):}
    \begin{itemize}
        \item \textbf{Explicit mapping (\type{registerConsumers}):}
        for each subscribe/request consumer entry in \type{InputChannelMapInfo} (SINGLE/RANGE),
        the core writes the edge in both directions:
        into the producer's forward table \type{mapping\_publish\_}/\type{mapping\_response\_} (producer $\rightarrow$ list of consumers)
        and into this instance's reverse table \type{mapping\_subscribe\_}/\type{mapping\_request\_} (consumer $\rightarrow$ list of producers).
        \item \textbf{Catch up this instance's \type{AUTO\_ALL} consumers (\type{registerToProducersAutoAll}):}
        connect each \type{AUTO\_ALL} consumer to all already-existing matching producers.
        \item \textbf{Catch up existing \type{AUTO\_ALL} consumers (\type{registerToConsumersAutoAll}):}
        connect this instance's new producers to all already-existing \type{AUTO\_ALL} consumers of the same channel type.
    \end{itemize}
\end{enumerate}

This makes \type{AUTO\_ALL} wiring incremental: whenever a new instance appears, both sides are ``caught up'' with whatever already exists.

\minihead{\type{AUTO\_ALL} and directed requests.}
The core uses the same mapping tables for both publish/subscribe and request/response so registration, removal, and inspection stay uniform.
Requests are still \emph{directed}: \type{sendRequest} names one concrete target response channel, so \type{AUTO\_ALL} does not turn requests into a broadcast.
What \type{AUTO\_ALL} buys you is that the instance can be created without a pre-filled mapping; if it needs to send a request, it can first query the core for existing response channels of a given type and then choose a target.

Because the input mapping is fixed at instance creation time, removing a module can invalidate other modules that are explicitly wired to it.
The core therefore treats explicit edges as \emph{hard dependencies} and uses \type{mapping\_publish\_}/\type{mapping\_response\_} to find dependents for cascade removal (\type{collectDependentModules}).
\type{AUTO\_ALL} edges are treated as \emph{non-binding} and ignored there: they are runtime-expanded wiring that can shrink/grow as modules appear and disappear, so they do not create a ``must-remove'' dependency.
(In valid configurations we expect the explicit dependency graph to be acyclic; \type{AUTO\_ALL} can add extra runtime edges without making removals unpredictable.)

The global producer indexes (\type{existing\_publish\_channels\_}/\type{existing\_response\_channels\_}) are what backs the ``what channels exist?'' queries used by modules and the UI.
The per-instance consumer-side tables (\type{mapping\_subscribe\_}/\type{mapping\_request\_}) capture the instance's \emph{effective} input wiring, including \type{AUTO\_ALL} connections that were expanded at runtime.
They are used to expose the current mapping (\type{getRunningModulesChannelMap}) and to persist/restore the wiring as part of core state saving/loading.


\subsection{Message routing and failure behavior} \label{subsec:ch3_core_routing}

All message delivery enters the runtime through \type{ICore} (\type{sendMessage}, \type{sendRequest}, \type{sendResponse}).
Each call takes a shared lock on the core state, validates the source/target identifiers against \type{running\_modules\_} and the per-instance mapping tables, and then forwards the message to the destination module via the \type{dll::IDllModule} entry points.
The core does not implement queueing or worker threads itself; it only decides \emph{where} to forward, while \type{IDllModule} decides \emph{how} to host and process the work.

\minihead{Publish/subscribe: \type{sendMessage}.}
A module publishes by calling \type{ICore::sendMessage(source\_channel, message)}. The core performs:
\begin{enumerate}
    \item \textbf{Validate source:} check that \type{source\_channel.module\_id\_} exists, the module is not destroyed, and that \type{source\_channel.local\_channel\_id\_} is a valid publish producer index (bounds-check against \type{mapping\_publish\_}).
    \item \textbf{Resolve routing:} read the producer-side list \type{mapping\_publish\_[source\_producer\_id]}.
    This list already contains both explicit and \type{AUTO\_ALL} connections; routing does not distinguish them.
    \item \textbf{Forward to each consumer:} for every consumer endpoint \type{(target\_module\_id, target\_subscribe\_consumer\_id)}:
    validate that the target module still exists and that the consumer ID is in range (bounds-check against \type{mapping\_subscribe\_}),
    then call
    \begin{quote}
      \type{target->module\_->processMessage(target\_subscribe\_consumer\_id, source\_channel, message)}.
    \end{quote}
\end{enumerate}
This call is the hand-off into the per-module hosting wrapper: \type{processMessage} is the core-to-module boundary for delivery, and it is where queueing/threads happen (described in \autoref{subsec:ch3_core_per_module_wrapper}).

\minihead{Request/response: \type{sendRequest} and \type{sendResponse}.}
Requests are directed: \type{sendRequest} always names one explicit target response channel.
The core still uses mapping tables for validation and for dependency bookkeeping, but not for choosing a target.

\begin{itemize}
    \item \textbf{\type{sendRequest(source, target, msg)}:}
    validate that both modules exist and that \type{source.local\_channel\_id\_} is a valid \emph{request consumer} index (\type{mapping\_request\_} bounds-check),
    while \type{target.local\_channel\_id\_} is a valid \emph{response producer} index (\type{mapping\_response\_} bounds-check).
    On success, forward directly to the target:
    \begin{quote}
      \type{target->module\_->processRequest(target\_response\_producer\_id, source\_channel, msg)}.
    \end{quote}

    \item \textbf{\type{sendResponse(source, target, msg)}:}
    validate that both modules exist and that \type{source.local\_channel\_id\_} is a valid \emph{response producer} index (\type{mapping\_response\_}),
    while \type{target.local\_channel\_id\_} is a valid \emph{request consumer} index (\type{mapping\_request\_}).
    On success, forward directly to the target:
    \begin{quote}
      \type{target->module\_->processResponse(target\_request\_consumer\_id, source\_channel, msg)}.
    \end{quote}
\end{itemize}

\minihead{Failure behavior.}
If a referenced module/channel does not exist, or the instance is already being destroyed (\type{destruction\_in\_progress\_}), publish delivery is dropped:
\type{sendMessage} returns without forwarding to that endpoint (and continues with other endpoints if possible).

For requests, the core additionally synthesizes a failure response if it cannot deliver the request:
it immediately calls \type{sendResponse} back to the caller with the same request \type{id\_}, an empty payload, and \type{success\_=false} (timestamped at send time).
This guarantees that a failed request still produces a terminal response, instead of silently disappearing.

The next section describes the \type{dll::IDllModule} implementation returned by \type{createModule}: the per-instance \type{DllModuleWrapper} that performs ingress decisions, queueing, worker scheduling, and (for requests) response dispatch.




\subsection{Per-module hosting wrapper} \label{subsec:ch3_core_per_module_wrapper}

The previous section showed that the core only decides \emph{where} to forward a message: it resolves the destination endpoints and calls one of
\type{IDllModule::processMessage/processRequest/processResponse}.
Everything about \emph{how} the work is queued, scheduled, and executed lives on the module side of the DLL boundary in a per-instance hosting wrapper.

\minihead{Instance creation inside the module library.}
Each library implements \type{createModule(...)} by constructing the concrete \type{IModule} (usually via \type{std::make\_unique}), checking \type{valid()}, and then returning a heap-allocated wrapper that implements \type{dll::IDllModule}:
\begin{itemize}
    \item if the module (or an internal decorator) is not valid, \type{createModule} returns \type{nullptr} and the core treats creation as failed,
    \item otherwise the library returns \type{new DllModuleWrapper(std::move(module), core, module\_id, logger)}.
\end{itemize}
This is why the core never sees \type{IModule} directly: it only stores an \type{IDllModule*} and later destroys it through \type{destroyModule}, i.e., in the same library that allocated it.

\minihead{What \type{DllModuleWrapper} owns.}
A \type{DllModuleWrapper} owns exactly one \type{IModule} instance and uses its \type{ModuleInfo} to preallocate the delivery infrastructure:
\begin{itemize}
    \item it reads the inbound channel counts (subscribe consumers, response producers, request consumers) to size internal structures,
    \item for each inbound channel it allocates fixed-capacity ring buffers sized by \type{message\_queue\_capacity\_} and pre-fills a reusable \type{ProcessingData} pool (avoids allocating on every message); channels are marked as prioritized using \type{prioritized\_}.
\end{itemize}

\minihead{Thread lifecycle.}
When the core calls \type{IDllModule::threadStart(timeout)}, the wrapper spawns two worker pools:
\begin{itemize}
    \item \textbf{regular workers} for non-prioritized channels,
    \item \textbf{prioritized workers} for prioritized channels.
\end{itemize}
The wrapper waits until all its worker threads report as running (via counters) and then calls \type{IModule::threadStart(remaining\_timeout)} to let the module start any internal threads it may have.
Stopping is symmetric: \type{threadStop} marks the instance as stopping, wakes all workers, waits for them to exit, prints wrapper-side metrics, and then calls \type{IModule::threadStop(remaining\_timeout)}.

\minihead{Ingress: fast hand-off and copying into queues.}
All three delivery entry points (\type{processMessage/processRequest/processResponse}) are thin wrappers that call \type{pushProcessingData(type, local\_channel\_id, source, header)}.
The ingress logic is:
\begin{enumerate}
    \item \textbf{Select the per-channel queue.}
    The wrapper maps \type{(type, local\_channel\_id)} to one internal queue index (messages, then requests, then responses).
    If the ID is out of range, the message is dropped.
    \item \textbf{Ask the module what to do.}
    Under the wrapper mutex, it computes a \type{QueueStatus} (\type{NORMAL} if the queue has space, \type{QUEUE\_FULL\_CAN\_DROP} if the queue is full but there is a message that has not started processing yet and can be dropped, or \type{QUEUE\_FULL} if the queue is full and all messages are being processed.) and calls \type{IModule::onIngress(...)}.
    This gives the module a single place to express policies like “drop if overloaded” or “keep only the latest”.
    \item \textbf{Enqueue a self-contained work item.}
    If accepted, the wrapper takes one \type{ProcessingData} from the preallocated pool, copies:
    (i) the raw payload bytes into \type{ProcessingData::data\_},
    and (ii) the blob handles into \type{ProcessingData::blobs\_}.
    Then it patches \type{MessageHeader} to point to these owned buffers and pushes the work item to the channel's processing queue.
\end{enumerate}
Copying into \type{ProcessingData} makes the queued work self-contained. The core forwards \type{MessageHeader} with pointers that are only meant to be used during the delivery call, but the wrapper processes messages later on worker threads, so it must copy \type{data\_} and the blob-handle array into the enqueued \type{ProcessingData} before returning. The pool then reuses the same \type{ProcessingData} slots, so the internal vectors keep their capacity over time and most ingress operations avoid repeated allocations. The copied \type{SharedDataBlob} handles must be stored with the work item as well (to reconstruct \type{MessageHeader} for deferred processing); keeping these handles until processing finishes also keeps the referenced shared payloads alive for the duration of the call into \type{IModule}.

\minihead{Worker execution model.}
Workers sleep on condition variables and wake when there is work in any queue of their class (regular/prioritized).
When awakened, a worker picks a non-empty queue (round-robin across channels) and processes exactly one work item:
\begin{itemize}
    \item \textbf{Message:} calls \type{IModule::processMessage(local\_channel\_id, source, header)}.
    \item \textbf{Response:} calls \type{IModule::processResponse(local\_channel\_id, source, header)}.
    \item \textbf{Request:} calls \type{IModule::processRequest(local\_channel\_id, source, header)} and immediately sends exactly one response back via the core (see below).
\end{itemize}
After processing, the wrapper clears stored blob handles (releasing shared payload references) and returns the \type{ProcessingData} object back into the per-channel pool.

\minihead{Request/response completion.}
From the module's and core's perspectives, \type{sendRequest} is asynchronous: it forwards the request into the target \type{IDllModule} and returns immediately.
Inside the target module, request handling is synchronous \emph{at the processing point}: the wrapper executes \type{IModule::processRequest} on a worker thread, obtains a \type{ResponseData}, and sends a matching response via:
\begin{quote}
\type{core->sendResponse(\{module\_id\_, local\_channel\_id\_\}, source\_channel, response\_header)}.
\end{quote}
This guarantees a single completion path: every accepted request produces one response message (success or failure) with the same request \type{id\_}.

\minihead{Dropping and failure responses.}
If the wrapper is stopping, or ingress decides to drop a work item due to queue pressure, the item is not enqueued.
For requests, the wrapper additionally synthesizes a failure response (empty payload, \type{success\_=false}) so callers do not wait indefinitely.
During \type{threadStop}, the wrapper also clears any queued-but-unprocessed work and releases stored blob handles; for queued requests, it emits failure responses before discarding them.



\subsection{Module removal and cascade deletion} \label{subsec:ch3_core_removal}

Module instances can be removed explicitly via \type{removeModule} (e.g., from the UI) and implicitly during state load (to clear the current runtime before reconstructing a saved graph).
Removal is implemented by \type{removeModule/removeModuleImpl} and mirrors creation: it first stops per-module worker threads (wrapper side), then removes routing edges and discovery entries (core side).

\minihead{Locking and the ``stable graph'' requirement.}
Removal takes \type{add\_remove\_mutex\_} and then a \emph{unique} lock on \type{core\_mutex\_}.
This prevents concurrent graph edits (add/remove/load) and prevents concurrent sends/reflection from observing partially updated mapping tables while edges are being removed.

\minihead{Which modules must be removed (dependencies).}
Because each module's input mapping is fixed at creation time, removing a producer can invalidate modules that explicitly depend on it.
The core therefore computes a cascade set via \type{collectDependentModulesImpl(id)}:
it performs a graph walk starting from \type{id} using producer-side tables \type{mapping\_publish\_} (publish$\rightarrow$subscribe edges) and \type{mapping\_response\_} (response$\rightarrow$request edges).
For each reached consumer endpoint, the core checks the consumer declaration in the destination module's \type{ModuleInfo} and:
\begin{itemize}
    \item treats \type{SINGLE}/\type{RANGE} consumers as \emph{hard dependencies} (the destination module is added to the removal set),
    \item ignores \type{AUTO\_ALL} consumers for cascade removal (they are runtime-expanded, non-binding wiring).
\end{itemize}
If the computed set contains more than the requested module and \type{recursive=false}, removal fails with \type{HAS\_DEPENDENCIES}.

\minihead{Removal order (reverse dependency order).}
If recursive removal is allowed, the core removes all modules in the computed set \emph{from the back}.
Intuitively, this destroys the most dependent modules first, so modules are not left running while their explicit inputs disappear.

\minihead{Stopping threads: why the core temporarily unlocks.}
For each module being removed, the core first marks \type{destruction\_in\_progress\_ = true} and then temporarily releases \type{core\_mutex\_} to call \type{IDllModule::threadStop(timeout)}.
This unlock is necessary to avoid deadlocks: \type{threadStop} waits for the wrapper's and module's worker threads to exit, but those threads may currently be calling into the core via the \type{ICore} API (e.g., to send messages or query reflection) and would be blocked on \type{core\_mutex\_} while doing so.
The \type{destruction\_in\_progress\_} flag ensures that concurrent deliveries see the instance as non-routable and are dropped (or turned into failure responses for requests, as described in \autoref{subsec:ch3_core_routing}).
After \type{threadStop} returns, the core clears the instance pointer (\type{module\_ = nullptr}) thus calling the custom deleter and the API's \type{destroyModule} method and reacquires \type{core\_mutex\_} to continue with table updates. 

This temporary unlock is the reason for using the second \type{add\_remove\_mutex\_}: it prevents concurrent graph edits while the core is temporarily unlocked during thread stopping.

\minihead{Detaching routing edges.}
Once the instance is stopped, the core removes all edges involving it, in both directions:
\begin{itemize}
    \item \textbf{Remove outgoing edges (\type{removeMappingProducers}).}
    The core iterates this module's \type{mapping\_publish\_}/\type{mapping\_response\_} and, for each connected consumer endpoint, removes ``our channel'' from the destination module's consumer-side table (\type{mapping\_subscribe\_}/\type{mapping\_request\_}).

\end{itemize}
After these steps, the removed module is disconnected from the routing graph in both producer and consumer views.

\minihead{Removing discovery entries.}
Finally, the core removes the module from the global discovery maps via \type{removeFromExistingMap}:
\type{existing\_publish\_channels\_}, \type{existing\_response\_channels\_}, and (when applicable) \type{existing\_subscribe\_auto\_all\_channels\_}/\type{existing\_request\_auto\_all\_channels\_}.
This keeps reflection queries and future \type{AUTO\_ALL} wiring consistent with the current set of running modules.

\minihead{Graph-change notification.}
After the removal loop finishes, the core increments \type{module\_mapping\_state\_id\_}.
The UI can use \type{
    getModulesMappingStateId()} to detect that the module graph changed and refresh any cached views of running instances and their mappings.

\minihead{Failure reporting.}
If any \type{threadStop} call fails to stop within the timeout, the core still proceeds with removal (with a warning) and returns \type{FAILED\_TO\_STOP\_THREADS}.
This keeps the runtime from leaving partially removed modules wired into the graph, but may imply undefined behavior if the module truly failed to stop its threads.



\subsection{Locking model and core usage constraints} \label{subsec:ch3_core_locking}

The core is entered concurrently from multiple module threads (delivery, requests, reflection), so it uses three mutexes with fixed responsibilities.

\minihead{\type{core\_mutex\_}: protects the live graph and routing tables.}
\type{core\_mutex\_} (a \type{std::shared\_mutex}) guards the mutable runtime state:
\type{running\_modules\_}, all mapping tables (\type{mapping\_publish\_}, \type{mapping\_subscribe\_}, \type{mapping\_response\_}, \type{mapping\_request\_}),
the discovery maps (\type{existing\_*\_channels\_}), and \type{module\_mapping\_state\_id\_}.
Read-only entry points (routing and reflection) take a \emph{shared} lock, while graph edits take a \emph{unique} lock.
During removal, the core temporarily releases \type{core\_mutex\_} around \type{IDllModule::threadStop} (see \autoref{subsec:ch3_core_removal}); the instance is made non-routable via \type{destruction\_in\_progress\_} for that window.

\minihead{\type{add\_remove\_mutex\_}: serializes graph edits and fixes lock ordering.}
All entry points that can change the graph (\type{addModule}, \type{removeModule}, \type{load}) take \type{add\_remove\_mutex\_} first and only then take \type{core\_mutex\_} (unique).
In particular, removal keeps \type{add\_remove\_mutex\_} held even while \type{core\_mutex\_} is temporarily released for \type{threadStop}.
This implies a practical restriction: module threads must not attempt graph edits (\type{addModule}/\type{removeModule}/\type{load}), especially during shutdown/\type{threadStop}, because they would deadlock on \type{add\_remove\_mutex\_}.

\minihead{\type{allocator\_mutex\_}: allocator lifetime only.}
Allocator creation/deletion (\type{createDynamicAllocator}, \type{createBufferAllocator}, \type{deleteAllocator}) is protected by \type{allocator\_mutex\_} and is independent from \type{core\_mutex\_}.
Allocator bookkeeping therefore does not couple to routing or graph edits.

\minihead{\type{destruction\_in\_progress\_}: making instances non-routable.}
Each running module instance has a \type{destruction\_in\_progress\_} flag used as a fast “do not deliver” gate.
When set, the core treats the instance as non-existent for routing: messages/responses are dropped and requests are completed via synthesized failure responses (as described in \autoref{subsec:ch3_core_routing}).



\subsection{Allocation and shared payload management} \label{subsec:ch3_core_allocation}

The contract in \autoref{subsec:ch3_allocator_contract} exposes shared payload allocation through \type{IAllocator} and shared payload access through \type{ISharedData}.
In this core implementation (in-process DLL modules), both the allocator objects and the allocated shared payload objects live entirely on the core side.
When a module calls \type{createDynamicAllocator} or \type{createBufferAllocator}, the core constructs a concrete allocator instance, stores it in \type{allocators\_}, and returns a raw \type{IAllocator*} handle to the module.
The \type{allocators\_} container is guarded by \type{allocator\_mutex\_} (only for creating/removing allocators); \type{deleteAllocator} removes the corresponding allocator from \type{allocators\_}.

Allocation itself happens inside the selected allocator backend and returns core-owned \type{ISharedData} objects (concretely, \type{SharedDataCore}) that provide \type{data()} and \type{size()} for direct access.
The \textbf{dynamic allocator} performs real heap allocation for each request and frees the block again when the payload is released.
The \textbf{buffer allocator} preallocates a fixed pool of equally-sized slots upfront; allocating a blob means taking a free slot from the pool (or failing if none are available), and releasing returns the slot back to the free list for reuse.
A process-isolated design would keep the same message-level contract, but would replace this pointer-based \type{ISharedData} representation with an inter-process form (e.g., shared-memory handles/offsets).



\subsection{Persistence} \label{subsec:ch3_core_persistence}

The persistence API (\type{ICore::save}/\type{ICore::load}) stores and reconstructs the \emph{runtime module graph} together with \emph{optional per-module state}. Concretely, the save captures: 

\begin{enumerate}[label=(\roman*)]
    \item the set of created module instances (split into auto-created and manually created),
    \item the effective channel mapping for each instance (explicit producer lists for non-\type{AUTO\_ALL} consumers, and an \type{auto\_all} flag otherwise),
    \item a list of \type{expected\_modules} that were available in the core at save time, and
    \item optional module-owned state via \type{ISerializableModule}. Allocators and transient runtime resources (module IDs, queues, in-flight messages, request IDs, etc.) are not persisted.
\end{enumerate}

\minihead{Save file structure (JSON + binary blobs).}
The core serializes the state into a single container returned as \type{SharedDataBlob}.
It starts with a JSON header that describes the module graph on a high level (instances, expected modules, and mappings) and includes per-module JSON state returned by \type{ISerializableModule::save()}.
Any per-module \emph{named binary blobs} returned by \type{ISerializableModule::save()} are stored as separate binary sections alongside the JSON (identified by name), so the JSON stays purely descriptive and human-readable while modules can still persist arbitrary binary data as needed.

\minihead{Saving steps.}
\begin{enumerate}
    \item The core takes a shared lock on \type{core\_mutex\_} to snapshot the current graph.
    \item It assigns each running instance a stable textual name of the form \type{<loaded\_name>\#<index>} (per-module counter) and uses these names inside the JSON for mapping, instead of raw runtime IDs.
    \item For each running instance it stores the instance metadata and the consumer-side mapping:
    \begin{itemize}
        \item for \type{SINGLE}/\type{RANGE} consumers, the list of producer endpoints (\type{producer\_module} by instance name and \type{producer\_channel\_id}),
        \item for \type{AUTO\_ALL} consumers, only \type{auto\_all=true} (explicit producer lists are omitted because they are derived at runtime).
    \end{itemize}
    \item For per-module state, the core calls \type{IDllModule::save()} on each instance, passing a core allocator. The DLL wrapper serializes the returned \type{ISerializableModule::SaveData}: a small JSON state header and an optional list of named binary blobs. The core incorporates the JSON fragment into the overall JSON header of the save file and stores any blobs as separate named payload records in the same container, alongside the JSON header.
\end{enumerate}

\minihead{Loading steps.}
Loading is a destructive operation: it replaces the current manually-created graph with the one from the save.
\begin{enumerate}
    \item The core takes \type{add\_remove\_mutex\_} and a unique lock on \type{core\_mutex\_}, then removes all \emph{non-auto-created} running modules (recursively), keeping auto-created ones (created during \type{initialize}). The internal state is therefore reset to an ``only auto-created instances remain'' baseline.
    \item The container is unpacked into the JSON header and a list of per-module binary blobs. The core validates the save-file format version and the module API version (hard failure on mismatch). It also compares the saved core version against the current one, but only emits a warning, because newer builds may still be able to load older saves.
    \item The core verifies \type{expected\_modules}: every module listed in the save must exist among currently loaded modules (the current installation may contain additional modules, but it must not be missing any that the save expects).
    \item Auto-created instances are \emph{not recreated}. For each saved auto-created entry, the core finds the corresponding already-running instance by \type{loaded\_name}, then restores its module state by calling \type{IDllModule::load()} with the saved \type{SaveData} (JSON + any attached binary blobs).
    \item Manually created instances are recreated from the saved description. For each instance, the core resolves producer references using the saved instance names, builds \type{InputChannelMapInfo}, calls \type{addModule} (which creates the module and starts its threads), and then calls \type{load} to restore the module state.
\end{enumerate}

This procedure guarantees that the \emph{mapping topology and per-module configuration} are restored (including \type{AUTO\_ALL} behavior via re-registration at creation time). It does \emph{not} guarantee stable numeric runtime IDs: IDs are reassigned during reconstruction, and the save format therefore uses instance names for mapping instead of persisting raw IDs. The expected lifecycle for persisted modules is: \emph{create instance} $\rightarrow$ \emph{threadStart} $\rightarrow$ \emph{load state} (auto-created instances skip the creation step because they already exist).




\section{Interface contracts} \label{sec:ch3_interface_contracts}

The module contract from \autoref{sec:ch3_module_contract} defines how modules exchange messages through the core, but it intentionally stays application-agnostic: it does not define what a particular message means. As the concrete modules for this thesis were implemented, we therefore introduced a set of \emph{interface contracts}: small, shared definitions that assign structure and semantics to selected message channels (camera frames, pen poses, robot commands, activation control, \dots). Their role is pragmatic: to ensure that modules remain interchangeable and independent, as long as they implement the same externally defined communication protocol.

\subsection{Overview: protocols on top of core messaging} \label{subsec:ch3_interface_contracts_overview}

\minihead{Interface contracts as application-level protocols.}
An interface contract can be seen as an application-level protocol layered on top of the generic core transport:
it specifies (i) which channel types exist and who publishes/subscribes or requests/responds, (ii) the concrete payload layout (POD header and optional binary blobs), and (iii) the expected interaction pattern (ordering, acknowledgements, best-effort streams, cancellation rules, \dots).
This makes the communication boundary explicit and keeps modules replaceable as long as they implement the same contract.

\minihead{Header-only schemas and channel identifiers.}
Concretely, contracts are implemented as C++ header-only definitions shared by the participating modules. Each contract provides matching \type{communication\_channel::Producer} and \type{communication\_channel::Consumer} descriptors with a \type{channel\_type\_identifier\_} string. The identifier acts as a compatibility key for binding channels in the core and also includes a compact schema hint and a version tag (e.g., \type{/v1}). In this thesis, these definitions are located under \path{backend/module_helpers/} (e.g., camera and pen message headers, activation wrapper messages, robot interface messages). For details of the exact binary layouts, the corresponding header files are the source of truth.

\minihead{Wrappers for higher-level protocols.}
Some interface protocols involve a larger state machine than a single message exchange, typically because they need many related operations and well-defined ordering (e.g., parameter enumeration, value edits, list operations, progress reporting, cancellation, and only then activation). To keep such logic consistent and to reduce burden on individual modules, the system uses \emph{wrappers}: \type{IModule} implementations that encapsulate another module instance and implement the protocol on its behalf. For example, the activation wrapper owns the parameter-management protocol (including request/response handling and serialization of parameter values), collects and validates the effective parameter set, and only then invokes a small capability interface on the wrapped module (e.g., \type{IActivableModule::activate}/\type{deactivate}). Traffic outside the wrapper's scope is forwarded when normal ingress is allowed, while protocol-specific ingress is intercepted and handled by the wrapper.

\minihead{Capability queries for wrapper stacking.}
To support stacking multiple wrappers, modules expose optional capabilities via \type{IModule::query\_capability}. Each wrapper forwards the query to the wrapped object, and the bottom-most module returns a pointer to the requested interface if supported (or \type{nullptr} otherwise). This allows extending the system's functionality (e.g., module activation) using messaging and wrappers without changing the base module contract.

\subsection{Activation interface} \label{subsec:ch3_activation_interface}

\minihead{Purpose.}
Many modules cannot start producing meaningful outputs immediately: they first need user-provided configuration (e.g., selecting a concrete device instance) or they must derive internal state from incoming data (e.g., computing calibration from a short image sequence). These steps can take tens of seconds and must run as a cancellable task. The activation interface standardizes this interaction so the UI can configure and (de)activate modules without requiring module-specific UI logic.

All files mentioned in this section are located in \path{backend/module_helpers/activation_wrapper/} (protocol + \type{ActivationWrapper}) and \path{backend/module_helpers/parameter_description/} (parameter schema).

\minihead{What the contract provides.}
A module declares a \emph{fixed list} of activation parameters (see \path{parameter_description.h}). Each parameter specifies:
\begin{itemize}
    \item a type (\type{BOOL}, \type{LONG}, \type{DOUBLE}, \type{STRING}, \type{ENUM}, \type{CUSTOM}),
    \item an optional default value and optional limits / enum options,
    \item and a cardinality (single value or a list with min/max size).
\end{itemize}
This enables both lightweight configuration (e.g., numeric settings, selecting a mode via \type{ENUM}) and data-driven activation via \type{CUSTOM}: the parameter value is obtained from an existing input consumer channel (either \emph{subscribe} or \emph{request/response}). In practice, this is used to pass module-specific payloads required for activation (e.g., a pen pose, list of images, or robot configuration), while keeping the wrapper protocol generic.

Once parameters are filled, the UI triggers \type{ACTIVATE}. At that point, the wrapped module receives the complete parameter set and performs the module-specific activation procedure asynchronously (e.g., probing a device that matches the requested properties, or running calibration on the provided data). The same protocol also supports \type{DEACTIVATE} and cancellation of long-running operations.

\minihead{Implementation structure.}
The protocol is implemented by \type{ActivationWrapper}. The wrapper owns the parameter storage and implements the parameter editing/validation protocol, while the wrapped module exposes a small capability interface \type{IActivableModule} (\path{activable_module.h}).

\minihead{How modules implement the contract.}
To make a module activable through the standard UI:
\begin{itemize}
    \item Declare the activation parameter list using \type{ParameterList}.
    \item Implement \type{IActivableModule}. At minimum, provide \type{activate()}, \type{deactivate()}, and \type{isActivated()}. The remaining hooks are optional: if you do not use request-based \type{CUSTOM} inputs, \type{sendRequestFromActivation(...)} can return \type{false}; if you do not need filtering/normalization, \type{processCustomMessageOrResponse(...)} can return \type{ProcessingResult::ACCEPT}; and if you do not report progress, \type{getActivationProgress()} can return \type{ProgressData} with \type{progress\_type\_=ProgressType::NONE}.
    \item Expose the activation-wrapper response channel in the module's \type{ModuleInfo} (so that the wrapper can receive UI requests via its response-producer entry point).
    \item Wrap the module in \type{ActivationWrapper} in the module factory (\type{createModule}) and pass the parameter list to the wrapper.
\end{itemize}

\minihead{Channels.}
The contract itself is a prioritized request/response pair between the UI and the wrapper (definitions in \path{message_types.h}):
\begin{itemize}
    \item \textbf{UI $\rightarrow$ ActivationWrapper:} request channel for parameter edits and activation control.
    \item \textbf{ActivationWrapper $\rightarrow$ UI:} response channel with result, \type{activated} flag and progress.
\end{itemize}
All \emph{other} channels of the wrapped module are gated: while deactivated, non-activation ingress is rejected (not buffered). The only exception is an input channel that is temporarily allowed when the wrapper is waiting for a pending \type{CUSTOM} value.

\minihead{Message shapes.}
Requests carry a small fixed header (request kind + parameter addressing) and optionally attach a single blob:
\begin{itemize}
    \item \type{SET\_VALUE}: one blob containing the new value (for non-\type{CUSTOM} parameters) or a capture trigger (for \type{CUSTOM} parameters). The wrapper validates the value against the parameter schema before storing it in \type{parameter\_values}.
    \item \type{LIST\_ADD}/\type{LIST\_REMOVE}: edits list cardinality without providing a value (no blob). New list entries are created empty and must be filled using \type{SET\_VALUE}.
    \item \type{READ\_ACTIVATION\_PARAMETERS}: no blob; on success the response includes one blob with the textual schema description (generated by \type{ParameterList::toString()}).
    \item \type{READ\_VALUES}: no blob; on success the response includes one blob with a compact binary snapshot of all current values. \type{CUSTOM} items are represented only by a present/absent marker (no payload).
    \item \type{ACTIVATE}: no blob; starts an asynchronous activation task and returns \type{RUNNING} (or \type{FAIL}). Progress is reported via the \type{progress\_} field in responses and via \type{GET\_STATUS}.
    \item \type{DEACTIVATE}: no blob; symmetric to \type{ACTIVATE} (async task, \type{RUNNING}/\type{FAIL}, progress via status).
    \item \type{GET\_STATUS}: no blob; returns \type{RUNNING} while the wrapper is either (i) waiting for a \type{CUSTOM} capture or (ii) executing an activation/deactivation task, otherwise \type{SUCCESS}. The response always includes the current \type{activated} flag, and includes \type{progress\_} when an activation/deactivation task is active. 
    
    Polling \type{GET\_STATUS} therefore allows the UI to (a) detect when the wrapper is no longer busy, (b) render module-reported progress, and (c) observe whether the module ended up activated or deactivated once the task terminates.
    \item \type{CANCEL\_TASK}: no blob; cancellation semantics depend on what is currently running.
    \begin{itemize}
        \item If the wrapper is \emph{waiting for a \type{CUSTOM} value}, \type{CANCEL\_TASK} returns \type{SUCCESS} if and only if the wait was cancelled before any value was captured (the stored \type{CUSTOM} remains empty). If it returns \type{FAIL}, the wait was not active anymore --- typically because the value was captured just before the cancel request was processed --- so the UI must treat the \type{CUSTOM} as filled.
        \item If an \type{ACTIVATE}/\type{DEACTIVATE} task is running, \type{CANCEL\_TASK} always returns \type{FAIL} by design: cancellation is best-effort and asynchronous, and the wrapper does not block to wait for the task to observe the cancel flag. Callers must poll \type{GET\_STATUS} until it stops reporting \type{RUNNING}; at that point the response's \type{activated} flag reflects the post-task mode (unchanged if the operation did not complete, flipped only if it completed).
    \end{itemize}
\end{itemize}
For \type{CUSTOM} parameters, when a candidate message/response arrives on the configured input (subscribe channel, or a matching response identified by the stored request ID), the wrapper forwards it to \type{IActivableModule::processCustomMessageOrResponse()}.
The module can then (i) \type{DROP} the packet (e.g., unrelated message on a shared channel) and keep waiting, (ii) \type{ACCEPT} it, in which case the wrapper serializes the incoming \type{MessageHeader} (+ blobs) into a canonical opaque byte array, or (iii) \type{ACCEPT\_REPLACE}, where the module supplies its own serialized representation (e.g., reduced/normalized data) to be stored instead.
The resulting byte array is stored in \type{parameter\_values} and later passed unchanged to \type{IActivableModule::activate()}.

\minihead{Flow.}
A typical UI-driven interaction is:
\begin{enumerate}
    \item UI requests the parameter schema (\type{READ\_ACTIVATION\_PARAMETERS}) and shows it in a generic activation panel.
    \item UI edits values (\type{SET\_VALUE}, \type{LIST\_ADD}/\type{LIST\_REMOVE}); the wrapper stores the values and validates basic constraints (type, ranges, enum domain, list bounds).
    \item For \type{CUSTOM}, UI triggers a capture via \type{SET\_VALUE}; the wrapper either waits for an incoming message on a configured subscribe channel, or asks the module to issue a request (\type{sendRequestFromActivation()}) and then waits for the corresponding response. The module may filter/transform the captured payload via \type{processCustomMessageOrResponse()}.
    \item UI sends \type{ACTIVATE}; the wrapper starts an async activation task and calls \type{IActivableModule::activate(parameter\_values,\ cancel\_flag,\ cancelled)}.
    \item UI polls \type{GET\_STATUS} to display progress (\type{getActivationProgress()}); \type{CANCEL\_TASK} aborts waiting for \type{CUSTOM} or cancels a running activation/deactivation task.
    \item On successful completion, the wrapper flips the gating state and normal ingress is allowed. \type{DEACTIVATE} mirrors the same pattern.
\end{enumerate}

\minihead{Activation state semantics.}
Activation is primarily a \emph{module-level on/off switch} with explicit configuration: \type{activate()} receives the filled parameter set and is expected to bring the module into an operational state (e.g., connect to a device/robot, allocate resources, and begin producing outputs). Conversely, \type{deactivate()} is expected to shut the module down and stop producing outputs. The wrapper mirrors this operational state in an internal \type{activated} flag, reports it to the UI, and uses it to gate non-activation traffic.

A newly created module starts \emph{deactivated} (\type{activated=false}), so only the activation protocol is reachable until activation succeeds (and the module should treat itself as inactive accordingly).

\begin{itemize}
    \item \textbf{State changes are driven by task success.} \type{ACTIVATE} and \type{DEACTIVATE} start an asynchronous task, but the wrapper keeps its current \type{activated} value while the task is running. When the task finishes and the module reports success (callback returns \type{true}), the wrapper flips \type{activated}. On failure (returns \type{false}), \type{activated} remains unchanged.
    \item \textbf{Requests are state-guarded.} \type{ACTIVATE} fails if the module is already activated; \type{DEACTIVATE} fails if it is currently deactivated.
    \item \textbf{Cancellation keeps the previous state.} If a running activation/deactivation task is cancelled, the module should set the \type{cancelled} flag and leave its internal state consistent with the previous mode; the wrapper keeps the previous \type{activated} value (deactivated when cancelling activation, activated when cancelling deactivation).
    \item \textbf{After load, the module defines the activation state.} On load, the wrapper restores parameters and inner module data and then sets \type{activated} from \type{IActivableModule::isActivated()}. This is the only point where the module directly controls the wrapper's state without running \type{activate()/deactivate()}; the module is therefore responsible for persisting and restoring any activation artifacts needed to remain operational after load.
\end{itemize}

\minihead{Guarantees and limitations.}
\begin{itemize}
    \item The wrapper centralizes parameter handling: modules only implement \type{activate()/deactivate()}, progress/status checks, and two hooks for custom parameter processing.
    \item Basic parameter validation is enforced before activation (type checks, numeric bounds, enum options, list size constraints).
    \item Parameter edits are blocked while activated, while waiting for a \type{CUSTOM} value, or while an activation/deactivation task is running (to keep activation inputs consistent).
    \item \type{CUSTOM} payloads are treated as opaque; their semantics and internal structure are module-defined.
    \item Non-activation traffic is not buffered while deactivated; senders must tolerate dropped messages until activation completes.
    \item The wrapper gates \emph{ingress} (incoming traffic) while deactivated, but it does not suppress the wrapped module's own outgoing publishes/requests; therefore, modules must honor their activation state and avoid producing outputs or side effects while deactivated.
\end{itemize}

\minihead{Persistence boundary.}
On save/load, the wrapper persists \emph{parameter values} (including stored \type{CUSTOM} payloads as named binary blobs) together with the wrapped module state (the wrapper nests the inner module's save data). The wrapper itself cannot reconstruct the meaning of ``activated''; after load it restores parameters, loads the inner module, and queries \type{IActivableModule::isActivated()} to decide whether to open the ingress gate. This is done because calling \type{activate()} after load could result in tens of seconds of work (e.g., calibration); therefore, it is the responsibility of modules to save their activation state and any artifacts required to maintain it (e.g., requested camera parameters, calibration results).


\subsection{Use case interface} \label{sec:ch3_usecase_wrapper}

\minihead{Purpose.}
The next interface we cover is the \emph{use case interface}. It defines what a \emph{use case module} is and provides the plugin architecture required by \reqref{req:use_case_plugin_support}. Conceptually, a use case turns its inputs (poses, trajectories, detected objects, etc.) into an executable \emph{program command} and exposes a standardized command lifecycle (create $\rightarrow$ start $\rightarrow$ monitor $\rightarrow$ control $\rightarrow$ cleanup). This allows the frontend to run arbitrary use cases through a uniform protocol, without embedding module-specific UI logic.

All files mentioned in this section are located in \path{backend/module_helpers/usecase_wrapper/} (protocol + \type{UsecaseWrapper}), \path{backend/module_helpers/base_usecase/} (\type{BaseUsecase}), and \path{backend/module_helpers/parameter_description/} (shared parameter schema).

\minihead{What the contract provides.}
The use case contract exposes a \emph{single, uniform command API} that separates a use case into two explicit phases: \emph{command construction} and \emph{command execution}.

Through this API, another module (typically the UI) can:
(i) discover what inputs are needed to build a command (typed parameter schema, including \type{CUSTOM} inputs acquired from existing channels),
(ii) provide / trigger acquisition of those inputs,
(iii) request creation of a concrete \emph{command instance} (returned as an opaque JSON blob, optionally accompanied by lightweight visualization data for preview), and
(iv) start execution of that command (start in real/simulated mode) and manage it via a stable handle (\type{task\_id}) with standardized operations (query status, pause, resume, stop).

\minihead{Use case inputs.}
A use case exposes three \emph{fixed} parameter lists (all using the shared \type{ParameterDescription} format):
\begin{itemize}
    \item \textbf{Auto parameters}: must be \type{CUSTOM}. Values are acquired from existing input consumers (subscribe or request/response) during parameter acquisition. These parameters represent the primary data inputs for the use case (e.g., current scene state or pen-provided pose --- the spatial data).
    \item \textbf{Required parameters}: non-\type{CUSTOM} values that must be provided by the user before command creation. Their use should be kept minimal, since it shifts focus away from spatial input. However, some use cases require non-spatial input without a reasonable default (e.g., a pick-and-place command may require an object ID).
    \item \textbf{Advanced parameters}: non-\type{CUSTOM} values with defaults; optional for the user to tweak. Hidden by default, meant for secondary tuning (e.g., approach speed). These provide flexibility without undermining the spatial input paradigm.
\end{itemize}

\minihead{State ownership.}
Unlike activation, the wrapper does not own a persistent ``current parameter set'' for a use case. Parameter editing state lives in the caller (typically the UI): command creation submits a complete snapshot of all parameter values, and the created command is returned as an opaque JSON representation. This allows keeping multiple independent drafts/instances of the same use case externally (e.g., the user partially fills one instance, switches to another, and returns later), and later passing the stored JSON back unchanged when starting execution.

\minihead{Implementation structure.}
The contract is implemented in three layers:
\begin{itemize}
    \item \textbf{Messaging contract (\type{message\_types.h}, \path{usecase_wrapper}).} Defines the request/response protocol used by the UI.
    \item \textbf{\type{UsecaseWrapper} + \type{IUsecaseModule} (\path{usecase_wrapper}).} Similar in spirit to \type{ActivationWrapper}: the wrapper handles the messaging contract, while the wrapped module implements a small capability interface \type{IUsecaseModule}.
    \item \textbf{\type{BaseUsecase} (\path{base_usecase}).} Convenience base class for use case authors, implementing the program lifecycle (\type{programStart}/\type{programStatus}/\type{programPause}/\dots) and providing default \type{IModule} behavior. The remaining use case-specific logic stays in the derived module.
\end{itemize}

\minihead{How modules implement the contract.}
To register a new use case as a plugin module:
\begin{itemize}
    \item Define use case metadata (name and description) and the three parameter lists (auto/required/advanced).
    \item Implement your use case module by deriving from \type{BaseUsecase} (at minimum implement \type{createCommandFromParameters(...)}, \type{validateParameters(...)}, and \type{runProgram(...)}).
    \item Expose the use case wrapper response channel in the module's \type{ModuleInfo} (so \type{UsecaseWrapper} can receive UI requests via its response-producer entry point).
    \item In the module factory (\type{createModule}), construct the module instance and wrap it in \type{UsecaseWrapper} (passing the metadata and parameter lists). Return the wrapper as the exported module instance.
    \item Optionally implement \type{processCustomMessageOrResponse(...)} to filter/normalize \type{CUSTOM} inputs (and \type{sendRequestFromUsecase(...)} for request/response acquisition).
\end{itemize}

\minihead{Channels.}
The contract itself is a prioritized request/response pair between the UI and the wrapper:
\begin{itemize}
    \item \textbf{UI $\rightarrow$ UsecaseWrapper:} requests for parameter schema acquisition, \type{CUSTOM} input capture, command creation, and execution control.
    \item \textbf{UsecaseWrapper $\rightarrow$ UI:} responses with result codes and any associated payloads (e.g., schema description, captured \type{CUSTOM} bytes, command JSON, visualization blob, \type{task\_id}, status/error info).
\end{itemize}

\minihead{Message families.}
The UI--use case protocol can be viewed as four request groups; most requests optionally attach a single blob payload.
\begin{itemize}
    \item \textbf{Parameter schema acquisition.}
    \begin{itemize}
        \item \type{READ\_COMMAND\_PARAMETERS}: returns the declared parameter schema (auto/\allowbreak required/\allowbreak advanced), together with basic use case metadata (module identifier, use case name and description).
    \end{itemize}

    \item \textbf{Auto-input capture for \type{CUSTOM}.}
    \begin{itemize}
        \item \type{READ\_CUSTOM\_PARAMETER\_START}: starts acquisition for one auto parameter and returns a read handle.
        \item \type{READ\_CUSTOM\_PARAMETER\_CHECK}: polls completion (or cancels); on success returns an opaque captured value.
    \end{itemize}
    Captured messages/responses are not interpreted by the wrapper. Instead, each candidate packet is forwarded to \type{IUsecaseModule::processCustomMessageOrResponse()}, which can \type{DROP} it, \type{ACCEPT} it (stored as a canonical serialized payload), or \type{ACCEPT\_REPLACE} it with a module-provided normalized representation. If a packet is accepted, the stored bytes are then returned to the querying module in the next \type{READ\_CUSTOM\_PARAMETER\_CHECK}.

    \item \textbf{Command creation.}
    \begin{itemize}
        \item \type{CREATE\_COMMAND}: request attaches all filled parameter values; response returns an opaque command representation (JSON), optionally accompanied by lightweight preview/visualization data.
    \end{itemize}

    \item \textbf{Execution control + cleanup.}
    \begin{itemize}
        \item \type{PROGRAM\_START\_REAL} / \type{PROGRAM\_START\_SIMULATED}: starts execution from the command JSON and returns a fresh \type{task\_id}. All subsequent execution-related requests refer to that running instance by passing this \type{task\_id}.
        \item \type{PROGRAM\_STATUS}: queries current \type{ProgramStatus} (and optional error details).
        \item \type{PROGRAM\_PAUSE}, \type{PROGRAM\_RESUME}, \type{PROGRAM\_STOP}: optional controls (unsupported operations fail fast).
        \item \type{PROGRAM\_REMOVE}: explicit removal of finished tasks to release per-task resources.
    \end{itemize}
\end{itemize}

\minihead{Flow.}
A typical UI-driven interaction is:
\begin{enumerate}
    \item \type{READ\_COMMAND\_PARAMETERS}: fetch the parameter schema (auto/\allowbreak required/\allowbreak advanced) and basic metadata; the UI renders a generic editor.
    \item Fill out parameters via the UI. For each auto \type{CUSTOM} parameter: start acquisition with \type{READ\_CUSTOM\_PARAMETER\_START} and poll \type{READ\_CUSTOM\_PARAMETER\_CHECK} until \type{SUCCESS} (or cancel).
    \item \type{CREATE\_COMMAND}: submit all filled parameter values; the use case returns an opaque command representation (JSON) and optionally a lightweight preview/visualization blob.
    \item \type{PROGRAM\_START\_REAL} or \type{PROGRAM\_START\_SIMULATED}: provide back the command JSON to start execution and obtain a \type{task\_id}.
    \item While running: use \type{task\_id} to poll \type{PROGRAM\_STATUS} and optionally issue \type{PROGRAM\_PAUSE}/\type{PROGRAM\_RESUME}/\type{PROGRAM\_STOP} (if supported).
    \item After a terminal status: call \type{PROGRAM\_REMOVE} (once per \type{task\_id}) to deterministically release per-task resources.
\end{enumerate}

\minihead{Use case responsibilities.}
The contract intentionally splits use case logic into two distinct responsibilities:
(i) \emph{static construction} and (ii) \emph{dynamic execution}.

During construction, the caller submits a complete parameter snapshot and the use case performs a single conversion step: it resolves task-specific details and returns a self-contained command representation (JSON) and optional preview/visualization data.

During execution, that JSON is treated as the program to run: the wrapper only provides lifecycle management (\type{task\_id}, status, pause/resume/stop, remove), while the use case module itself must perform all runtime interaction with its environment via its own channels (requests/responses, optional subscriptions), e.g., query scene state, command robot motion, and wait for acknowledgements.

\minihead{BaseUsecase: what authors implement.}
\type{BaseUsecase} is a convenience base for use case modules. It mainly removes two categories of boilerplate:
(i) the \type{program*} lifecycle methods of \type{IUsecaseModule} (task IDs, per-task status tracking, thread ownership, pause/stop bookkeeping, and \type{PROGRAM\_REMOVE} cleanup), and
(ii) default implementations of \type{IModule} methods that many use cases do not need.

A use case author still implements the use case-specific parts, at minimum:
\begin{itemize}
    \item \type{createCommandFromParameters(...)}: builds the opaque command JSON (and optional preview/visualization data) from the filled parameter values.
    \item \type{validateParameters(command\_json)}: semantic validation of the created command before execution.
    \item \type{runProgram(command\_json, simulated)}: the blocking runtime logic executed in a worker thread.
\end{itemize}

For interactive use cases, it is common to also override:
\begin{itemize}
    \item \type{processCustomMessageOrResponse(...)} to filter/normalize captured inputs,
    \item optionally \type{sendRequestFromUsecase(...)} when \type{CUSTOM} acquisition uses request/response, and
    \item \type{valid()}, \type{onIngress()}, \type{processMessage()}, \type{processResponse()} to orchestrate other modules during execution (e.g., robot control, scene detection).
\end{itemize}

Pause/stop can be implemented cooperatively by periodically calling \type{handleControlRequests(...)} inside long-running loops. If a pause is requested, the call blocks until a resume arrives (via an internal condition variable); if a stop is requested, it aborts execution by throwing \type{BaseUsecase::StopException}.

If you need non-blocking control (e.g., to perform cleanup before exiting), \type{checkControlRequests()} can be used to poll whether pause/stop has been requested and handle it explicitly.

\minihead{Guarantees and limitations.}
\begin{itemize}
    \item \textbf{Uniform plugin protocol:} the UI never calls use case-specific channels; it only uses the wrapper protocol.
    \item \textbf{No activation state for use cases:} use case modules are not gated by the activation interface. They are expected to always respond to the wrapper protocol (at least \type{READ\_COMMAND\_PARAMETERS} and \type{CREATE\_COMMAND}); if prerequisites are missing (e.g., tracking or robot backend inactive), they fail fast with an appropriate error instead of being ``inactive''.
    \item \textbf{Validation boundary:} the wrapper enforces type/limits/list bounds according to \type{ParameterDescription}; the module focuses on semantic validation and execution.
    \item \textbf{Task identity and control:} all running programs are addressed by \type{task\_id}; invalid IDs return \type{ID\_INVALID}. Pause/resume/stop are optional capabilities (fail fast if unsupported).
    \item \textbf{Multiple concurrent use case instances:} the interface supports multiple concurrent tasks. Whether a specific use case allows this is module-defined (in \type{BaseUsecase} via \type{supports\_multi\_program\_}).
    \item \textbf{Explicit cleanup:} finished tasks should be removed via \type{PROGRAM\_REMOVE} to avoid accumulating per-task states and to release finished program threads. Calling \type{PROGRAM\_REMOVE} while a task is still running will fail; calling it again for an already removed \type{task\_id} returns an invalid-ID result. If the call is omitted, per-task bookkeeping and finished threads remain retained until module shutdown.
    \item \textbf{CUSTOM payload opacity:} captured messages/responses are stored as opaque byte arrays (after module-side filtering/serialization). Their internal meaning is strictly module-defined.
    \item \textbf{Stateless command construction:} parameter drafts and created command instances are external state (typically held by the UI), as described above in \emph{State ownership}. Neither \type{UsecaseWrapper} nor \type{BaseUsecase} persist drafts, created commands, or task history; if commands must be restored later (e.g., across sessions), the control module is responsible for storing the parameter snapshot and/or the returned command JSON and submitting it again when needed.
\end{itemize}





\subsection{3D visualization interface} \label{sec:ch3_visualization_interface}

\minihead{Purpose.}
\reqref{req:ui_use_case_selection} implies that the UI should visualize the programmed tasks, and \autoref{subsec:ch2_analysis_ui} suggests doing so using a simple shared 3D scene view (e.g., pose markers and trajectories in the workspace).
Instead of embedding module-specific rendering rules in the UI (e.g., special-casing a particular robot model or a concrete 6DoF input device), the 3D visualization interface lets \emph{producer modules} describe what should appear in the shared scene. This keeps the 3D visualization extensible and allows modules to be replaced without updating UI code: a new tracking module simply publishes its own visualization primitives.

All files mentioned in this section are located in \path{backend/module_helpers/visualization_3d_interface/}.


\minihead{Scene model.}
The contract represents the 3D scene as:
\begin{itemize}
    \item a static \textbf{resource registry} (shape descriptions), and
    \item dynamic \textbf{instances} (objects referencing a resource ID + current pose) and \textbf{trajectories} (poly-lines).
\end{itemize}
Resources are built from simple primitive shapes (box, sphere, cylinder) combined into a \type{ComplexShape} (see \path{scene_desc_api.h}). Instances refer to registered resources via \type{ResourceId} and are addressed by stable \type{ObjectId}s.

\minihead{Communication modes and channels.}
Unlike activation and use cases, this contract is not implemented as a wrapper. It exposes a small publish stream for best-effort updates plus a request/response path for full scene queries (see \path{message_types.h}):
\begin{itemize}
    \item \textbf{Publish:} \type{PubType::ANNOUNCE} (presence/readiness) and \type{PubType::UPDATE} (delta updates).
    \item \textbf{Request/response:} \type{ReqType::READ\_FULL} (resources + full current scene) and \type{ReqType::READ\_SCENE} (scene only).
\end{itemize}

\minihead{Message shapes.}
All messages use a tiny POD header and an optional blob payload:
\begin{itemize}
    \item \textbf{Publish:} data contains \type{PubType} (1 byte).
    \begin{itemize}
        \item \type{ANNOUNCE}: no blobs.
        \item \type{UPDATE}: one blob with a batch of ``add/update/remove'' commands for objects and trajectories.
    \end{itemize}
    \item \textbf{Request:} data contains \type{ReqType} (1 byte), no blobs.
    \item \textbf{Response:} data echoes \type{ReqType} and attaches one blob:
    \begin{itemize}
        \item \type{READ\_FULL}: resource registrations + current objects + current trajectories,
        \item \type{READ\_SCENE}: current objects + current trajectories.
    \end{itemize}
\end{itemize}
The exact binary layouts are implemented in \path{serialization_helper.h} and are treated as the source of truth; this section intentionally stays at the protocol level.

\minihead{Typical flows.}
\begin{itemize}
    \item \textbf{Bootstrap / (re)sync:} a producer publishes \type{ANNOUNCE}. A visualization consumer (typically the UI) reacts by issuing \type{READ\_FULL} to obtain all resources and the current full scene. From that point on, it applies \type{UPDATE} deltas.
    \item \textbf{Updates:} whenever the producer changes the scene (adds/updates/removes objects or trajectories), it publishes \type{UPDATE}. To avoid sending many small messages and to prevent partial intermediate states, changes are typically accumulated and then flushed as a single coalesced \type{UPDATE} at a logical update boundary (e.g., once per processing cycle).
\end{itemize}

\minihead{Helper implementation.}
To reduce boilerplate and keep updates compact, modules typically use \type{VisualizationHelper} (\path{visualization_helper.h}):
\begin{itemize}
    \item holds the \emph{current state} (registered resources, existing objects, existing trajectories),
    \item buffers edits in a \type{CommandCoalescer} / \type{CommandBuffer} so that add/\allowbreak update/\allowbreak remove operations are merged (e.g., add+remove cancels out; update+update merges),
    \item serves \type{READ\_FULL}/\type{READ\_SCENE} requests from the stored state, and
    \item publishes \type{ANNOUNCE} and \type{UPDATE} using a dynamic allocator for response/update blobs.
\end{itemize}
The helper is not internally thread-safe; most calls require external mutual exclusion (the only exception is retrieving the constant response-producer channel ID).

\minihead{How modules implement the contract.}
To make a module contribute to the shared 3D scene:
\begin{itemize}
    \item Add the contract channels to \type{ModuleInfo}: one publish producer (\type{visualization\_3d\_interface\_publish\_producer}) and one response producer (\type{visualization\_3d\_interface\_response\_producer}).
    \item Construct a \type{VisualizationHelper} in the module constructor, check \type{valid()}, and cache the helper's response-producer channel ID via \type{getResponseProducerChannel()}.
    \item Register all required resources via \type{registerResource(...)} \emph{before} announcing (resources are fixed after \type{announce()}). This is typically done directly from the module constructor after the \type{valid()} check.
    \item Call \type{announce()} once when the module is ready to interact with the system. It is typically triggered from runtime code (e.g. the consumer message processing methods or your own background thread), not the constructor, to avoid initialization deadlocks.
    \item Wire request handling: in \type{onIngress()}, accept requests on the cached response-producer channel; in \type{processRequest()}, forward those requests to \type{VisualizationHelper::processVisualizationRequest(...)}.
    \item During runtime, update the scene by calling \type{addObject}/\type{updateObject}/\type{removeObject} and \type{addTrajectory}/\type{updateTrajectory}/\type{removeTrajectory}, then call \type{sendUpdate()} once the logical update batch is complete to publish a coalesced delta. All helper calls except \type{getResponseProducerChannel()} must be done under a mutex.
\end{itemize}

\minihead{Guarantees and limitations.}
\begin{itemize}
    \item \textbf{State reconstructability:} a consumer can always reconstruct the producer's complete state by issuing \type{READ\_FULL} (or \type{READ\_SCENE} if resources are already known), even if it missed some \type{UPDATE}s.
    \item \textbf{Best-effort updates (no IDs):} \type{UPDATE} is a compact delta stream without update IDs. If messages are dropped, a consumer has no direct way to detect the gap other than requesting \type{READ\_SCENE}/\type{READ\_FULL} (a future extension could add monotonic update IDs).
    
    In particular, a dropped \emph{remove} leaves stale objects/trajectories visible indefinitely, and a dropped \emph{add} can cause later updates to reference an unknown ID until a re-sync is performed.
    \item \textbf{No global synchronization across producers:} updates from different producer modules are not globally serialized; the consumer simply merges per-producer contributions as they arrive (each producer's stream is self-consistent, but cross-producer timing is unconstrained).
    \item \textbf{Static resources:} resource registrations are intended to be declared once (before \type{announce()}) and then reused by instance updates; changing resource descriptions at runtime is not supported by the current protocol.
    \item \textbf{Thread safety:} \type{VisualizationHelper} is not internally thread-safe; all calls must be externally synchronized (e.g., by a mutex), except \type{getResponseProducerChannel()} as explicitly documented. This is intentional: a logical scene update typically consists of multiple \type{add/update/remove} calls followed by a single \type{sendUpdate()}, and these steps should not be interleaved with changes from other threads.
    \item \textbf{Simplified primitives:} the scene description supports only simple primitive-based shapes (no meshes, no boolean operations such as subtract/intersect; complex shapes are formed by composing primitives).
\end{itemize}



\subsection{Robot execution interface} \label{sec:ch3_robot_interface}

\minihead{Purpose.}
\reqref{req:robot_communication_interface} (see also \autoref{subsec:ch2_analysis_robot_interface}) motivates a robot-agnostic execution interface: use cases should be able to trigger motion primitives and observe execution state without hard-coding a vendor API. In this thesis, the interface is used primarily for robot motion and basic state feedback; tool control was considered in the interface design, but it is not implemented and is not required by the demonstration scenarios.

All files mentioned in this section are located in \path{backend/module_helpers/robot_interface/} (contract + feature messages) and \path{backend/module_helpers/robot_wrapper/} (client-side helper).

\minihead{Communication modes and channels.}
The contract uses a small, feature-agnostic envelope (request type / feature / action ID) and carries the \emph{feature-specific meaning} in an attached blob (see \path{message_types.h} and \path{message_types_definitions.h}). Concretely, it combines a request/response path for starting and controlling actions with two asynchronous subscribe streams for feedback:
\begin{itemize}
    \item \textbf{Request/response:} a client sends \type{START\_ACTION} or \type{UPDATE\_ACTION} for a specific \type{RobotFeature}. In practice, these requests almost always include a feature-specific blob that describes \emph{what} action to start/update and with which parameters. The robot module replies immediately with a standardized outcome and, for long-running actions, a fresh \type{action\_id}.
    \item \textbf{Subscribe (status):} periodic \type{STATUS} messages tagged by \type{RobotFeature}, typically with a feature-specific blob describing the current state (e.g., current pose/joints/execution state).
    \item \textbf{Subscribe (finished):} a \type{FINISHED} notification for long-running actions, identified by \type{action\_id} and a success flag. A feature-specific blob may additionally carry results or error details (e.g., an error message on failure).
\end{itemize}

\minihead{High-level request types.}
All robot actions share the same top-level request envelope:
\begin{itemize}
    \item \type{START\_ACTION}: starts a new action for a given \type{RobotFeature}. The concrete action kind and its parameters are encoded in the feature-specific blob.
    \item \type{UPDATE\_ACTION}: targets an already running action identified by \type{action\_id} (e.g., cancel or other feature-defined updates), again described by the feature-specific blob.
\end{itemize}

\minihead{Response outcomes.}
The response envelope standardizes common outcomes across all features (with any additional details carried in an optional feature-specific blob):
\begin{itemize}
    \item \type{FEATURE\_NOT\_SUPPORTED}: the requested feature is not implemented by the robot module.
    \item \type{DATA\_INVALID}: the feature-specific blob could not be parsed or validated.
    \item \type{SUCCESS}: the request was applied successfully (for \type{UPDATE\_ACTION}) or completed immediately (for \type{START\_ACTION}).
    \item \type{SUCCESS\_IN\_PROGRESS}: \type{START\_ACTION} succeeded and continues asynchronously; the response carries a fresh \type{action\_id}.
    \item \type{NOT\_IN\_PROGRESS}: \type{UPDATE\_ACTION} references an unknown or non-running \type{action\_id}.
    \item \type{FAILURE}: the request failed (optionally with feature-specific error detail).
\end{itemize}

\minihead{Motion primitives used in this thesis.}
Within \type{RobotFeature::ROBOT\_CONTROL}, the implemented messages cover:
\begin{itemize}
    \item \textbf{Start requests:} joint, linear, arc, and trajectory motions (long-running tasks), plus a \emph{robot specs} query (limits, joint count and ranges; immediate response).
    \item \textbf{Update requests:} an update command for cancelling an in-progress motion (by \type{action\_id}).
    \item \textbf{Asynchronous feedback:} periodic \type{STATUS} updates (joint positions, base/tool/flange poses in the world frame, and a coarse execution state), and a \type{FINISHED} notification for long-running motions (identified by \type{action\_id}, with success/failure and an optional error detail on failure).
\end{itemize}
The exact payload layouts are implemented in \path{features/robot_control/messages.h} and are treated as the source of truth; this section stays at the protocol level.

\xxx{Fig (OPTIONAL): Robot action lifecycle: START\_ACTION $\rightarrow$ (SUCCESS\_IN\_PROGRESS, action\_id) $\rightarrow$ STATUS stream + optional UPDATE\_ACTION $\rightarrow$ FINISHED(action\_id, success).}

\minihead{Typical flow (client perspective).}
A typical use case that commands motion proceeds as follows:
\begin{enumerate}
    \item Send \type{START\_ACTION} with \type{feature=ROBOT\_CONTROL} and a feature-specific motion payload.
    \item If the response is \type{SUCCESS\_IN\_PROGRESS}, store the returned \type{action\_id}. All subsequent control refers to this action by passing \type{action\_id} (either via \type{UPDATE\_ACTION} or by matching the asynchronous \type{FINISHED} notification).
    \item While the action runs, optionally consume \type{STATUS} messages for monitoring (pose/joints/state); if cancellation is needed, send \type{UPDATE\_ACTION} for the stored \type{action\_id} with the appropriate feature-specific update payload (e.g., a cancel request).
    \item When a \type{FINISHED(action\_id, success)} message arrives, treat the action as complete (and optionally inspect the feature-specific payload for error details on failure).
\end{enumerate}

\minihead{Helper implementation (recommended for clients).}
To reduce boilerplate for use cases, modules typically use \type{RobotWrapper} (\path{robot_wrapper.h}). It:
\begin{itemize}
    \item resolves the request and subscribe channels during construction and exposes \type{valid()},
    \item wraps robot control commands (move joint/linear/arc/trajectory, cancel, get specs),
    \item provides both \emph{non-blocking} calls (return \type{action\_id}) and a \emph{blocking} mode that waits for the corresponding \type{FINISHED},
    \item parses \type{STATUS} and \type{FINISHED} messages and exposes optional callbacks (\type{setStatusCallback}, \type{setFinishedCallback}), and
    \item wraps the low-level request/response + subscribe streams into a simple typed API.
\end{itemize}

\minihead{How modules implement the contract (client-side wiring).}
A module that \emph{controls} the robot (e.g., a use case) typically:
\begin{itemize}
    \item declares the three robot-interface \textbf{consumers} in \type{ModuleInfo}: one request consumer and two subscribe consumers (status + finished),
    \item constructs \type{RobotWrapper} in its constructor and fails module initialization if \type{valid()} is false,
    \item forwards core callbacks to the wrapper:
    \begin{itemize}
        \item in \type{onIngress()}, delegate to the wrapper when \type{handlesIngress(...)} is true,
        \item in \type{processResponse()}, delegate when \type{handlesResponse(...)} is true,
        \item in \type{processMessage()}, forward status/finished subscribe messages to \type{RobotWrapper::processMessage(...)}.
    \end{itemize}
    \item then issues motion/spec/cancel calls through the wrapper from its runtime logic.
\end{itemize}

\minihead{Notes for robot-module implementers.}
A module that \emph{implements} robot control is expected to:
\begin{itemize}
    \item respond to \type{START\_ACTION}/\type{UPDATE\_ACTION} immediately with standardized outcomes,
    \item generate and track \type{action\_id}s for long-running actions,
    \item publish periodic \type{STATUS} updates (feature-tagged), and
    \item publish a \type{FINISHED} message exactly once for each \type{SUCCESS\_IN\_PROGRESS} action when it reaches a terminal state.
\end{itemize}

\minihead{Guarantees and limitations.}
\begin{itemize}
    \item \textbf{Feature gating:} unsupported capabilities are reported explicitly via \type{FEATURE\_NOT\_SUPPORTED}, allowing clients to degrade gracefully.
    \item \textbf{Asynchronous completion:} long-running operations are identified by \type{action\_id} and terminate via \type{FINISHED}; request/response does not stream progress.
    \item \textbf{Best-effort feedback streams:} \type{STATUS} and \type{FINISHED} are asynchronous subscribe streams; if messages are dropped, a client may miss intermediate status samples, and missing \type{FINISHED} requires recovery logic (in this thesis, \type{RobotWrapper} includes practical heuristics for stale actions (actions for which no \type{FINISHED} arrived within a timeout), but robust gap detection would require explicit sequence IDs as a future extension).
    \item \textbf{Tool control is out of scope:} the feature list reserves identifiers for tool-oriented capabilities, but this thesis implements and uses only \type{RobotFeature::ROBOT\_CONTROL}.
\end{itemize}





\subsection{Scene detection interface} \label{sec:ch3_scene_detection_interface}

\minihead{Purpose.}
\reqref{req:scene_detection_logic} (see also \autoref{subsec:ch2_analysis_scene_detection}) calls for scene detection that provides object poses for downstream use cases (e.g., seam detection and trajectory alignment). The scene detection interface standardizes this output as a small request/response contract: a client can query which objects the detector knows about and request a fresh detection result on demand.

All files mentioned in this section are located in \path{backend/module_helpers/scene_detection_helper/}.

\minihead{Scene model.}
The contract assumes a \emph{static registry} of detectable object types and a \emph{dynamic scene} of detected instances:
\begin{itemize}
    \item \textbf{Registry objects:} a list of \type{RegisteredBox} items (stable object \type{id} + box dimensions).
    \item \textbf{Detected objects:} a list of \type{DetectedBox} items (object \type{id} + current 6DoF \type{Pose}).
\end{itemize}
In this thesis, objects are intentionally represented only as simple boxes (width/height/depth), and detection returns their full 6DoF pose (position + orientation).

\minihead{Communication modes and channels.}
The interface is a simple request/response pair (definitions in \path{message_types.h}):
\begin{itemize}
    \item \textbf{Request:} \type{ReqType::READ\_REGISTRY} or \type{ReqType::READ\_SCENE}.
    \item \textbf{Response:} echoes the request type and returns one blob with an array of objects (registry boxes or detected boxes).
\end{itemize}

\minihead{Message shapes.}
Requests and responses use small POD headers plus a single blob:
\begin{itemize}
    \item \textbf{Request header:} \type{version} + \type{req\_type}, no blobs.
    \item \textbf{Response header:} \type{version} + \type{req\_type} + \type{count}, plus one blob encoding \type{count} and the corresponding array.
\end{itemize}
The exact binary layouts and helper functions for building and parsing blobs are implemented in \path{message_types.h} and are treated as the source of truth.

\minihead{Typical flow (client perspective).}
A module that needs scene information typically:
\begin{enumerate}
    \item obtains the object registry once by sending \type{READ\_REGISTRY} and caching the returned \type{RegisteredBox} list.
    \item sends \type{READ\_SCENE} whenever it needs a fresh detection; parses the returned \type{DetectedBox} list (\type{id} + \type{Pose}).
\end{enumerate}

\minihead{RegistryRequestHandler (optional convenience).}
To avoid needing to handle the \type{READ\_REGISTRY} request manually, clients may use \type{RegistryRequestHandler} (\path{registry_request_handler.h}). It:
\begin{itemize}
    \item resolves the target request channel once,
    \item runs a small background thread that repeatedly issues \type{READ\_REGISTRY} until a valid response arrives (or shutdown), and
    \item exposes a thread-safe snapshot via \type{getRegisteredBoxesSnapshot()}.
\end{itemize}
This is a convenience only; the core contract remains plain request/response without wrappers.

\minihead{Notes for scene-detection module implementers.}
A scene detection module is expected to:
\begin{itemize}
    \item store the registry (registered boxes) internally and serve \type{READ\_REGISTRY} from that state (the registry should not change at runtime),
    \item perform detection on \type{READ\_SCENE} and respond with the detected boxes (\type{id} and \type{Pose}),
    \item validate \type{version} and \type{req\_type} and fail fast on unsupported requests.
\end{itemize}

\minihead{Guarantees and limitations.}
\begin{itemize}
    \item \textbf{Box-only object model:} the registry and detections are limited to \type{RegisteredBox} and \type{DetectedBox}; there is no support for meshes, richer geometry, or object-specific shape parameters beyond box extents.
    \item \textbf{Static registry assumption:} the registry is intended to remain constant over the module lifetime; clients typically request it once and reuse it.
    \item \textbf{No uncertainty model:} detections report only \type{id} and \type{Pose}; there is no standardized confidence, covariance, or multiple-hypothesis output in the current protocol.
\end{itemize}




\subsection{Supporting message formats} \label{sec:ch3_supporting_messages}

\minihead{Purpose.}
Besides the core interface contracts, the system uses several shared message formats for transporting camera, calibration, and pen data between modules. These are lightweight schemas (not wrappers) that keep the sensing pipeline modular: producers publish structured data, and downstream modules can be swapped as long as they keep consuming the same message types.

All files mentioned in this section are located in \path{backend/module_helpers/} under
\path{camera_messages/}, \path{camera_pose_helper/}, \path{calibrated_camera_world_messages/}, and \path{pen_messages/}.

\minihead{Camera images.}
\type{camera\_messages} defines a \type{CameraMessage} header plus one or more blobs that can pack \emph{multiple images per blob} (header + per-image descriptors + raw pixel data). In this thesis, it is used to send a stereo pair in a single blob, produced by \type{stereo\_camera\_module\_windows} and consumed by \type{web\_frontend\_module} (preview) and \type{camera\_pose\_injector\_module} (pose augmentation).

\minihead{Camera images with robot pose.}
\type{camera\_pose\_helper} extends the camera header with the latest robot flange pose (\type{CameraWithFlangePose}). This format is produced by \type{camera\_pose\_injector\_module} and consumed by \type{robot\_stereo\_camera\_calibration\_module}. The injector exists to attach robot pose at the message level (dropping camera frames until a pose is available), keeping calibration inputs self-contained.

\minihead{Calibrated camera set.}
\type{calibrated\_camera\_world\_messages} defines \type{CalibratedCameraSet}: camera images accompanied by per-camera world poses (extrinsics) and intrinsics (camera matrix \type{K} and distortion coefficients \type{D}, as used by OpenCV). It is produced by \type{robot\_stereo\_camera\_calibration\_module} and consumed by \type{pen\_tracking\_multicam\_module} and \type{scene\_detection\_stereocam\_module}.

\minihead{Pen messages.}
\type{pen\_messages} provides:
\begin{itemize}
    \item \type{PenMessageRaw}: a low-level stream (timestamp, tip pose, button states) published while the pen is being tracked.
    \item \type{PenMessageIntent}: discrete, higher-level intents emitted when the user completes a pen interaction: \type{POSE} (a single user-defined target pose), \type{TRAJECTORY} (a user-defined path; pose sequence stored in a blob), and \type{SPECIAL\_ACTION} (a generic auxiliary trigger interpreted by the consumer module).
\end{itemize}
In this thesis, \type{pen\_tracking\_multicam\_module} publishes both; \type{PenMessageIntent} is consumed by \type{web\_frontend\_module} (special action handling) and by use case modules that capture poses/trajectories as \type{CUSTOM} inputs.


\subsection{Implementation helper utilities} \label{sec:ch3_shared_helper_utilities}

\minihead{Purpose.}
Besides message schemas, the implementation relies on a few small reusable utilities used across multiple wrappers and message types (all under \path{backend/module_helpers/}).
\begin{itemize}
    \item \type{mixed\_buffer\_allocator}: combines a fixed-slot buffered allocator with a dynamic allocator to cheaply allocate typical small blobs while still supporting occasional large payloads.
    \item \type{pose\_utils}: a minimal \type{SE3} utility (composition, inversion, conversions) used in calibration and geometric processing.
    \item \type{synchronous\_request\_helper}: turns the core asynchronous request/response mechanism into a blocking call, which is convenient in modules that want a simple linear control flow (e.g., a use case issuing a scene-detection request and waiting for the response).
    \item \type{async\_helpers}: a small cancellable task wrapper used where the UI/wrappers need to run long work asynchronously (e.g., activation/deactivation).
    \item \type{base64} (third-party): base64 encode/decode used when converting \type{CUSTOM} parameter payloads to/from strings (in \type{parameter\_description}).
    \item \type{serialization\_helper}: lightweight binary (de)serialization utilities used pervasively for building and parsing blob payloads across message types and wrappers.
\end{itemize}



\section{Putting it together: UI orchestration and persistence} \label{sec:ch3_putting_it_together}

This chapter has so far introduced the system in terms of \emph{contracts}: the module contract (routing and lifecycle) and several interface contracts (activation, use cases, visualization, robot execution, scene detection, \dots). These contracts define \emph{what is compatible with what}, but they do not yet explain how a user actually \emph{operates} the system as a coherent tool: how modules are wired, configured, used to author commands, executed as tasks, and stored as reusable artifacts.

This section therefore describes one concrete \emph{control-plane integration} used in this thesis. The \type{web\_frontend\_module} provides the user-facing orchestration: it edits the module graph through core reflection/orchestration, drives activation and use case protocols, aggregates visualization streams, and exposes a persistence workflow. Additionally, it relies on a reusable helper, \type{UsecaseTree}, that turns the \emph{single use case} interface contract into the concrete programming model presented to the user: a linear list of commands that can be previewed, executed, and saved/loaded.

\subsection[UsecaseTree: from use cases to programs]{UsecaseTree: from use case modules to executable programs}
\label{sec:ch3_usecase_tree}

\minihead{Purpose.}
The use case interface contract (\autoref{sec:ch3_usecase_wrapper}) intentionally defines \emph{one} use case module as a black box that can (i) expose a parameter schema, (ii) construct a command (JSON) from concrete parameter values, and (iii) execute that command as a task with a status lifecycle. The contract itself does not prescribe a programming UI: a use case may be executed standalone (schema $\rightarrow$ values $\rightarrow$ execute), or embedded into richer authoring environments (e.g., as reusable blocks in a teach pendant). In this thesis, the \type{web\_frontend\_module} adopts \type{UsecaseTree} as a pragmatic integration that organizes use cases into a linear command list with explicit parameter editing, a per-command \emph{confirmation} step (command JSON generation), sequential execution, and program persistence.

All files mentioned in this section are located in \path{backend/module_helpers/usecase_tree/}. The helper interacts with use case modules exclusively via the use case wrapper protocol in \path{backend/module_helpers/usecase_wrapper/} (notably \path{message_types.h} and \path{usecase_module_interface.h}).

\minihead{What the helper provides.}
\type{UsecaseTree} is a reusable implementation component (not a system contract). Given a host module that can send requests and receive responses, it maintains a \emph{program-level} view of use case programming:
\begin{itemize}
    \item \textbf{Use case catalog:} a cache of discovered use cases (identifier, human-readable metadata, parameter schema groups, and the communication channel used for directed requests).
    \item \textbf{Editable program representation:} an ordered list of commands; each command binds a selected use case identifier to concrete parameter values (including list parameters) and may carry an optional preview visualization payload.
    \item \textbf{Command materialization:} support for acquiring missing \type{CUSTOM} inputs and for materializing a command into its executable representation (cached command JSON), including tracking whether the cache is in sync with the current parameters.
    \item \textbf{Execution management:} a sequential runner (\type{ProgramInstance}) that executes a snapshot of the command list, exposes program state/control (pause/resume/stop) and reports a final result.
    \item \textbf{Program persistence:} serialization/deserialization of the program (use case identifiers, parameter values, cached command JSON, and optional preview data) as a JSON document.
\end{itemize}

\minihead{Program model.}
\type{UsecaseTree} structures the programming workflow around three concepts (\path{structs.h}, \path{program_instance.h}):
\begin{itemize}
    \item \type{AvailableUsecase}: an entry in the discovered catalog, keyed by a stable \emph{use case identifier}, storing metadata, the three parameter lists, and the communication channel used for directed requests to the module.
    \item \type{ExistingCommand}: one command in the user program, referencing a chosen \type{AvailableUsecase} via a use case identifier and holding concrete parameter values. It may additionally cache a \type{command\_data\_json} string (generated by \type{CREATE\_COMMAND}) and optional visualization preview data returned by the module.
    \item \type{ProgramInstance}: the executing instance of a program run. It operates on a snapshot of the command list (list of \type{ExistingCommand}) and enforces a linear execution order (command $i$ must finish before command $i{+}1$ starts).
\end{itemize}
This separation is deliberate: the \emph{catalog} answers \emph{what can be done}, the \emph{command list} captures \emph{what the user authored}, and the \emph{program instance} defines \emph{how authored commands are executed} (sequentially, with explicit lifecycle control).

\minihead{Typical workflow (user-facing).}
Use case authoring and execution is \emph{orchestrator-driven}: the UI initiates each step via directed request/response on a selected use case channel, rather than reacting to a stream. A typical session built on \type{UsecaseTree} proceeds as:
\begin{enumerate}
    \item \textbf{Discover:} update the available use case catalog from connected modules.
    \item \textbf{Compose:} append/insert/remove commands to form a linear program; for each command, edit parameter values (including list cardinalities).
    \item \textbf{Acquire CUSTOM (optional):} for missing \type{CUSTOM} parameters, start a capture and poll until the value is produced (or cancel).
    \item \textbf{Confirm:} request \type{CREATE\_COMMAND} to obtain the command JSON (and optional preview visualization) and cache it as the executable representation of the command.
    \item \textbf{Execute:} start a program run (real or simulated). The runtime executes commands sequentially and exposes the program state and final result to the caller.
\end{enumerate}

\minihead{Execution control semantics.}
\type{UsecaseTree} delegates all execution to the use case wrapper’s task lifecycle. Concretely, a program run is realized by sending \type{PROGRAM\_START\_REAL} (or \type{PROGRAM\_START\_SIMULATED}) for each confirmed command JSON, polling \type{PROGRAM\_STATUS} until a terminal status is reported, and issuing \type{PROGRAM\_REMOVE} for explicit cleanup (\path{program_instance.h}, \path{program_instance.cpp}). 

The optional control operations \type{PROGRAM\_PAUSE}, \type{PROGRAM\_RESUME}, and \type{PROGRAM\_STOP} are treated as \emph{best-effort}:
\begin{itemize}
    \item if the active use case module implements these requests, the current command may pause/stop mid-execution and the polled status reflects that;
    \item if a module ignores or does not support the operation, \type{ProgramInstance} still provides a deterministic user-facing behavior at \emph{command boundaries}: a pause request takes effect after the current command completes (the next command is not started until resume), and a stop request terminates the program after the current command completes.
\end{itemize}
This yields a robust orchestration model without requiring every use case module to implement advanced control logic; modules that do support it simply provide a better interactive experience.


\minihead{Persistence boundary.}
\type{UsecaseTree} persistence is intentionally scoped to the \emph{user program}: it stores the ordered command list with parameter values and cached command JSON (plus optional per-command preview visualization), and it recreates commands on load. The cached command JSON is treated as a derived artifact; when parameters change, it may be invalidated and must be re-confirmed.

Internal identifiers used for bookkeeping are implementation details and are regenerated when loading; the stable anchor is the use case identifier and the parameter/value content. This persistence is separate from core-level persistence of the module graph and per-module state (discussed later in \autoref{sec:ch3_persistence_semantics}).

\minihead{Scope and limitations.}
\type{UsecaseTree} does not interpret the semantics of command JSON, nor does it define how use case modules implement their internals; it only standardizes the \emph{user-facing} composition and execution structure on top of the existing use case protocol. In this thesis, the chosen model is intentionally linear (sequential commands); parallel composition or branching program logic are out of scope.

\subsection{UI orchestration module} \label{sec:ch3_ui_orchestration}

\minihead{Purpose.}
While the contracts in this chapter define \emph{compatibility} (which channels can talk to which), the system still requires a control-plane component that \emph{drives} these interactions into an end-user workflow. In this thesis, this role is implemented by the web-based \type{web\_frontend\_module}: it is a normal module from the module contract’s perspective, but it orchestrates the module graph, activation, use case programming/execution (via \type{UsecaseTree}), visualization aggregation, and persistence.

\minihead{UI decomposition (three screens).}
The frontend exposes three user-facing views that mirror its orchestration responsibilities:
\begin{itemize}
    \item \type{\textbf{AddModuleUi}}: graph \emph{construction} by instantiating modules and selecting input channel wiring.
    \item \type{\textbf{ActivationUi}}: graph \emph{configuration} by editing activation parameters, activating/deactivating modules, and removing modules.
    \item \type{\textbf{MainVisualizationUi}}: \emph{operation} of the configured system (camera preview, 3D scene visualization, use case program authoring/execution, and utility actions).
\end{itemize}
This separation is conceptual: it reflects three phases of using the system rather than internal UI implementation details.

\minihead{Module graph wiring via core reflection/orchestration.}
Graph editing is implemented on top of \type{ICoreControl} (\autoref{subsec:ch3_reflection_orchestration}):
\begin{itemize}
    \item The frontend presents the \emph{loaded module catalog} as a palette of module types that can be instantiated, together with their declared subscribe/request consumers.
    \item For each consumer, it presents a \emph{current registry} of compatible producer endpoints (publish producers for subscribe wiring; response producers for request wiring), and the user \emph{explicitly selects} which endpoints to bind.
    \item On creation, the selected endpoints are collected into an \type{InputChannelMapInfo} and passed to \type{ICoreControl::addModule}; the core validates and materializes the wiring as part of the runtime graph.
    \item Removal is likewise delegated to the core (\type{removeModuleById}), with dependency-aware removal supported by \type{collectDependencies} when the UI needs to present the user with the impact of a removal.
\end{itemize}


\minihead{Activation driving (generic configuration panel).}
For each running module instance, the frontend determines whether it is activable by checking whether the module exposes the activation-wrapper protocol (activation response producer). Activable modules are configured and driven through the activation interface (\autoref{subsec:ch3_activation_interface}):
\begin{itemize}
    \item The UI requests the activation parameter schema (\type{READ\_ACTIVATION\_PARAMETERS}) and current values (\type{READ\_VALUES}) and renders a generic editor that follows the parameter-description definitions (types, ranges, enum domains, list bounds; see \path{backend/module_helpers/parameter_description/}).
    \item Users edit values and list cardinalities; changes are propagated to the activation wrapper using \type{SET\_VALUE} and \type{LIST\_ADD}/\type{LIST\_REMOVE}.
    \item Activation and deactivation are started explicitly by \type{ACTIVATE}/\type{DEACTIVATE} and monitored via \type{GET\_STATUS}; long-running tasks are presented as cancelable progress dialogs via \type{CANCEL\_TASK}.
    \item Modules that do not expose activation are treated as \emph{always ready} (no activate/deactivate controls), keeping the setup UI uniform even for modules that need no configuration.
\end{itemize}
Activation parameter values are owned by the activation wrapper/module side; the UI only edits them through the protocol.

\minihead{Use case programming and execution.}
The main visualization view embeds the programming UI (implemented in \path{program_tree.h}/\path{program_tree.cpp}) backed by \type{UsecaseTree} (\path{backend/module_helpers/usecase_tree/}; see also \autoref{sec:ch3_usecase_tree}):
\begin{itemize}
    \item The UI discovers available use cases by querying all connected use case modules for metadata and parameter schema (\type{READ\_COMMAND\_PARAMETERS}).
    \item The user composes a linear program by appending or reordering commands (each command binds a use case identifier to concrete parameter values).
    \item Missing \type{CUSTOM} inputs are acquired on demand through the use case wrapper’s capture protocol (\type{READ\_CUSTOM\_PARAMETER\_START} and repeated \type{READ\_CUSTOM\_PARAMETER\_CHECK}), either manually (``Click to set'') or via a pen-triggered auto-fill shortcut.
    \item Each command is explicitly \emph{confirmed} by requesting \type{CREATE\_COMMAND}, which materializes the command JSON (and optionally a preview visualization) and caches it as the executable representation.
    \item A program run is then started in real or simulated mode and controlled via start/pause/resume/stop; execution is sequential and its progress is surfaced to the user through the program state and completion dialogs.
\end{itemize}
This makes the control flow explicit and reproducible: execution is never pushed by modules; it is always initiated and advanced by the orchestrator via directed request/response.

Unlike activation, the linear program (the command list and each command's parameter values, and optionally cached command JSON and visualization) is stored and edited entirely within the UI (via \type{UsecaseTree}); only \type{CUSTOM} values are acquired externally from the selected use case module.

\minihead{Visualization: camera preview and 3D scene.}
The main view aggregates two visualization modalities:
\begin{itemize}
    \item \textbf{Camera preview (subscribe):} camera frames arrive as publish/subscribe messages and are displayed in the UI so the user can inspect what the camera sees (see \autoref{sec:ch3_supporting_messages}). The UI presents a single camera stream as the preview source.
    \item \textbf{3D scene visualization (subscribe + request):} incremental scene updates arrive as publish/subscribe messages (ANNOUNCE/UPDATE). When the UI first becomes aware of a visualization producer, it requests a full scene snapshot via the 3D visualization interface's request/response path (e.g., \type{READ\_FULL}; see \autoref{sec:ch3_visualization_interface}); afterwards it processes the delta updates as they arrive. The UI additionally exposes ``Reload Visualization'', which re-requests full snapshots to rebuild the local scene mirror.
\end{itemize}
The scene view additionally integrates per-command preview overlays returned by use case modules as part of \type{CREATE\_COMMAND}.

\minihead{On-demand utility actions (directed requests).}
Besides the generic contracts, the main view exposes a small set of practical operator actions implemented as explicit requests:
\begin{itemize}
    \item \textbf{Scan Scene:} triggers a fresh scene-detection request (\type{READ\_SCENE}) to an available scene-detection module; the resulting objects appear through the standard visualization channel (the UI does not interpret detection blobs directly).
    \item \textbf{Set Position / Move To Position:} persists a user-defined default robot joint configuration (stored in \path{main_visualization_data.json}) and commands a robot move to that configuration via the robot execution interface.
\end{itemize}

\minihead{Note.}
The UI module is described only through its externally observable behavior: how it exposes and drives the system interfaces to the user. Internal UI structure is intentionally omitted, unlike the core and selected modules, because it is not required to understand the system design. Consequently, the UI module is not decomposed further in \autoref{chapter:ch4_implementation}.

\subsection{User workflow state machine} \label{sec:ch3_workflow_state_machine}

\minihead{Purpose.}
The contracts and helper components above define individual interactions (mapping, activation, use case requests, visualization), but a user experiences the system as an end-to-end workflow. This section summarizes that workflow as a small state machine whose transitions are triggered by explicit user actions in the UI.

\minihead{Workflow states (user perspective).}
A typical session proceeds through the following states:
\begin{enumerate}
    \item \textbf{Graph setup:} instantiate modules and choose wiring for subscribe/request consumers (AddModuleUi; \type{ICoreControl::addModule}).
    \item \textbf{Module configuration:} edit activation parameters and activate required modules (ActivationUi; activation wrapper protocol).
    \item \textbf{Program authoring:} compose a linear list of use case commands and fill parameters, including on-demand acquisition of \type{CUSTOM} values (MainVisualizationUi; \type{UsecaseTree}).
    \item \textbf{Command confirmation:} materialize each command into its executable JSON representation (\type{CREATE\_COMMAND}); when all commands are confirmed, the program becomes eligible for execution.
    \item \textbf{Execution:} start a program run (real/simulated), monitor execution, and optionally pause/resume/stop; completion yields a final result (completed / stopped / error).
\end{enumerate}
At any point, the user may persist either the current system setup (module graph + optional per-module state) or the current program (command list and each command's authored state); the concrete semantics are described in \autoref{sec:ch3_persistence_semantics}.

\subsection{Persistence semantics} \label{sec:ch3_persistence_semantics}

\minihead{Two persisted artifacts.}
The UI exposes two save/load mechanisms that intentionally target different kinds of user work:
\begin{itemize}
    \item \textbf{System setup (\type{.aergo}, core-owned):} the current runtime module graph (which modules are instantiated and how they are wired) together with optional per-module state, saved/loaded via \type{ICore::save}/\type{ICore::load} (\autoref{subsec:ch3_core_persistence}).
    \item \textbf{Program (\type{.paergo}, UI-owned):} the user-authored linear program managed by \type{UsecaseTree} (command list, parameter values including \type{CUSTOM}, and cached derived artifacts), saved/loaded via \type{UsecaseTree::toJson()/fromJson()} (\autoref{sec:ch3_usecase_tree}).
\end{itemize}

\minihead{Rationale.}
The system setup is primarily an \emph{environment-specific configuration step}: the user instantiates the sensing/robot pipeline modules that match their hardware and workspace and activates them (e.g., camera selection and calibration, input device setup, scene detection). Once the graph works in a given environment, it can typically be reused many times without rebuilding it from scratch.
Programs are authored \emph{within} such a prepared setup: they describe the task sequence (use case commands and their parameters) that the user wants to run on top of the already-configured pipeline.

\minihead{Core state persistence (\type{.aergo}).}
Core persistence stores and reconstructs the running graph using \type{ICore::save}/\type{ICore::load} (\autoref{subsec:ch3_core_persistence}). The UI's role is orchestration and packaging:
\begin{itemize}
    \item \textbf{Save:} invoke \type{ICore::save()}, wrap the returned container into a user-facing \type{.aergo} archive with a human-readable \path{project.json} header and any accompanying binary blobs, and write it to disk.
    \item \textbf{Load:} read an \type{.aergo} archive from disk, unpack \path{project.json} and binary blobs, invoke \type{ICore::load(...)} to reconstruct the graph, then refresh UI views (module identifiers and wiring endpoints may change across load).
\end{itemize}
Per-module state is optional and module-owned: all modules expose \type{ISerializableModule}, but they may report \type{supports\_saving\_=false} and thus opt out of persistence. Regardless of whether a module supports persistence or chooses to serialize an ``activated'' state, its activation parameter values are persisted by the \type{ActivationWrapper}. 

Whether a module ends up activated after loading is module-defined: some modules may treat activation as a restorable internal state (e.g., calibrated), while others may require re-activation due to external dependencies such as connected hardware (e.g., robot communication).

\minihead{Program persistence (\type{.paergo}).}
Program persistence stores only the programming artifact managed by \type{UsecaseTree}:
\begin{itemize}
    \item each command is anchored by a stable \emph{use case identifier} and stores the current user-edited parameter values (including \type{CUSTOM} payloads);
    \item it additionally stores derived artifacts needed for execution and preview: cached command JSON (the executable representation) and optional preview visualization, together with the ``confirmed / out-of-sync'' flag that indicates whether the cached JSON matches current parameters (out-of-sync commands must be re-confirmed before execution);
    \item program save/load restores the authored program state (commands, parameters, cached artifacts), not an in-flight execution; a loaded program is ready to be confirmed (if necessary) and started.
\end{itemize}
To reconstruct the program during loading, the loader must resolve each stored use case identifier against 
currently connected use case modules. If an identifier is missing, the UI cancels the load and reports an error indicating which use case identifier is unavailable.

\minihead{Summary.}
Graph persistence is core-owned infrastructure that reconstructs the environment-specific runtime wiring and optional per-module configuration/state. Program persistence is a UI-level authoring artifact built on top of the use case interface; it stores the user-authored linear program (use case commands and their parameters) together with derived execution/preview artifacts (cached command JSON, optional preview visualization, and the confirmed/out-of-sync state). Loading a program requires the referenced use case identifiers to be available in the current setup; otherwise loading fails with a clear error.
