\chapter{Hardware Setup and Module Implementations}
\label{chapter:ch4_implementation}

This chapter describes the concrete prototype used in this thesis: the physical setup and the internal implementation of the modules that instantiate the end-to-end configuration introduced in \autoref{fig:runtime_full}.
Unlike \autoref{chapter:ch3_system_design}, which focused on contracts and interface semantics, this chapter focuses on the concrete implementation decisions inside the modules.

\minihead{Accompanying electronic attachment.}
The submitted electronic attachment contains (i) a pruned snapshot of the C++ source tree (\path{diploma-thesis-code/}) and (ii) the hardware and calibration artifacts required to reproduce the prototype setup (CAD, firmware, PDFs, configuration JSON).
A complete overview of the attachment structure, build notes, and artifact file locations is provided in \autoref{appendix:source_code} and \autoref{appendix:artifacts}.

\minihead{Implementation note.}
The system is implemented in C++ and targets Windows and Linux (build workflow and toolchain assumptions are summarized in \autoref{appendix:source_code}).
Most modules are platform-neutral; platform-specific code is currently concentrated in the Windows camera acquisition module (\autoref{subsec:ch4_mod_stereo_camera}).
The web frontend is based on Wt and was used on both Windows and Linux in development.

The pen tracking module uses the cross-platform BLE library SimpleBLE. The current discovery sequence assumes that the target service UUIDs are visible already during BLE scanning (as observed on Windows); on Linux, the module would need to connect first and query services after connecting. This platform-specific note is described again in \autoref{subsec:ch4_mod_pen_tracking}.

\minihead{Reading note.}
The UI module is described in Chapter~\ref{chapter:ch3_system_design} at the level of externally observable orchestration behavior (\autoref{sec:ch3_ui_orchestration}) and is not decomposed further here.
This chapter instead focuses on the sensing, authoring, execution, and use case modules that produce and consume the interface messages defined in Chapter~\ref{chapter:ch3_system_design}.

\minihead{Structure of this chapter.}
We first summarize the physical setup and calibration artifacts, then describe the module implementations in the same logical order as the runtime pipeline shown in \autoref{fig:runtime_full}.

\section{Physical setup and calibration artifacts}
\label{sec:ch4_physical_setup}

This section summarizes the physical components required to run the system and the calibration artifacts that the operator must provide.
The operational steps for setup and execution are deferred to \autoref{chapter:ch5_user_workflow}; the concrete artifact files shipped with the thesis attachment are listed in \autoref{appendix:artifacts}.

\subsection{Robot platform and execution backend}
\label{subsec:ch4_hw_robot}

The target robot platform is a Kassow collaborative robot.
During development, the prototype was tested primarily in the REMII offline environment, and validated on a physical robot of model KR1205.
As motivated in \autoref{subsec:ch2_analysis_robot_interface}, robot execution is abstracted behind the shared \type{robot\_interface} contract, implemented by the thesis-side module \type{robot\_module\_kassow}.

The prototype assumes a robot-mounted stereo camera rig: the camera is mechanically attached to the robot flange such that its pose is rigid with respect to the flange.
This enables the calibration module to combine robot kinematics (flange pose) with camera observations to place both cameras in a common workspace frame, as discussed in the camera trade-offs in \autoref{subsec:ch2_analysis_camera_tradeoffs}.
The mechanical mount used in the prototype is provided as a CAD artifact in \autoref{appendix:camera_mount_cad} and shown in \autoref{fig:ch4_camera_mount}.

Robot communication is realized through a small robot-side component (CBun) installed on the Kassow controller.
The CBun exposes the robot capabilities to the thesis runtime; the thesis-side \type{robot\_module\_kassow} consumes this endpoint and provides the \type{robot\_interface} protocol to the rest of the runtime.
CBun sources and build notes are provided in \autoref{appendix:cbun}; the operator procedure for installation/enabling is described in \autoref{chapter:ch5_user_workflow}.

\subsection{Stereo camera hardware}
\label{subsec:ch4_hw_camera}

The prototype uses a compact stereo camera module with two synchronized global-shutter sensors (OmniVision OV9281), providing $1280 \times 800$ grayscale images per sensor.
In the prototype configuration, the camera was operated as a single side-by-side stream ($2560 \times 800$) with MJPEG compression at 60\,FPS.

% ----------------------------
\subsection{Stereo camera acquisition module (Windows)}
\label{subsec:ch4_mod_stereo_camera}

\minihead{Role in graph.}
The \emph{Stereo Camera Input Module (Windows)} (\type{stereo\_camera\_module\_windows}) is the source of the stereo image stream in \autoref{fig:runtime_full}.
Its output is consumed by the \emph{Camera Pose Injector} (pose augmentation) and by the UI for camera preview.

\minihead{Interface-level I/O.}
The module publishes \type{camera\_messages} (\autoref{sec:ch3_supporting_messages}).
Each published message contains a \type{CameraMessage} header and a shared blob that packs the stereo pair (two image descriptors + pixel data) in one allocation.

\minihead{Activation and parameters.}
The module is activable via the activation wrapper (\autoref{subsec:ch3_activation_interface}); while deactivated, it does not publish camera frames.
The activation parameters are:
\begin{itemize}[nosep]
    \item \type{Resolution Width [px]} (default 2560): total width of the captured frame; in the stereo side-by-side mode each eye uses width/2,
    \item \type{Resolution Height [px]} (default 800): height of the captured frame,
    \item \type{Framerate [FPS]} (default 60): target capture framerate,
    \item \type{Framerate Tolerance [FPS]} (default 3): accepted deviation from the target framerate when matching device-reported modes,
    \item \type{Camera Index} (default 0): index into the filtered list of devices/modes that match resolution and framerate constraints,
    \item \type{Use Auto Exposure} (default true): enables camera auto exposure if supported,
    \item \type{Manual Exposure Value} (default -4): used only when auto exposure is disabled; activation fails if the value is outside the device-supported range.
\end{itemize}
The selected parameters are persisted in the module save state so that a loaded graph can restore the capture settings.

\minihead{Implementation.}
On activation, the module enumerates available capture devices and selects a matching MJPEG-capable video mode based on the requested resolution and framerate tolerance.
Captured frames are converted to an uncompressed 4-channel format and split into left/right images by treating the input as a side-by-side stereo frame.
Frames are published continuously from a dedicated capture thread; each message allocates its payload from a fixed-size buffer allocator to avoid per-frame heap allocations.

\minihead{Failure modes and known limitations.}
This is a Windows-only module (it uses Windows capture APIs and is not built on non-Windows platforms).
Activation can fail if no device/mode matches the requested parameters or if manual exposure cannot be applied; such failures are surfaced as activation failure (with diagnostic logging).

\subsection{Camera pose injector module}
\label{subsec:ch4_mod_pose_injector}

\minihead{Role in graph.}
The \emph{Camera Pose Injector} (\type{camera\_pose\_injector\_module}) joins the camera stream with the robot status stream in \autoref{fig:runtime_full}.
It consumes stereo camera images and robot status, and publishes camera frames augmented with the latest robot flange pose for calibration.

\minihead{Interface-level I/O.}
The module consumes \type{camera\_messages} (\autoref{sec:ch3_supporting_messages}) and the robot interface status stream (\autoref{sec:ch3_robot_interface}).
It publishes \type{camera\_pose\_helper} messages (\type{CameraWithFlangePose}) as described in \autoref{sec:ch3_supporting_messages}.

\minihead{Activation and configuration.}
The module is configuration-free and does not use the activation wrapper.
It operates continuously, but it drops camera frames until at least one robot status message has been received (so that a flange pose is available).

\minihead{Implementation.}
The module maintains a cached \emph{latest} flange pose extracted from incoming robot status messages (robot control feature).
For each incoming camera frame, it attaches the most recent cached pose and republishes the message with the original image blob.
Queueing is tuned for real-time behavior: camera input is handled as ``latest-only'' (older frames are replaced), while robot status messages are allowed to queue briefly and may drop the oldest entries if overloaded.

\minihead{Failure modes and known limitations.}
If robot status is missing or does not carry the required robot-control payload, camera frames are dropped and no augmented stream is produced.
Time association is implemented as ``latest sample'': each incoming camera frame is augmented with the most recently received flange pose from the robot status stream (no buffering, timestamp alignment, or interpolation).

As a consequence, the camera frames and poses are not synchronized in the general case.
This is sufficient for the workflow used in this thesis, where frames are collected only while the robot is stationary, so the flange pose is effectively constant over the capture window.

\subsection{Robot-mounted stereo camera calibration module}
\label{subsec:ch4_mod_camera_calibration}

\minihead{Role in graph.}
The \emph{Stereo Camera Calibration (Robot)} module (\type{robot\_stereo\_camera\_calibration\_module}) performs one-time calibration of the robot-mounted stereo rig and then publishes a calibrated stereo stream for downstream modules.

As shown in \autoref{fig:runtime_full}, the module consumes stereo frames augmented with the robot flange pose (from \emph{Camera Pose Injector}) and publishes a \emph{calibrated camera set} that is consumed by \emph{Pen Tracking} and \emph{Scene Detection}.

\minihead{Interface-level I/O.}
The input is a camera message together with a flange pose (see \autoref{sec:ch3_supporting_messages}). The output is a \type{CalibratedCameraSet} message carrying (i) the original camera header and image blobs and (ii) per-camera calibration data (world pose and intrinsics for the left and right camera).

\minihead{Activation and configuration.}
Calibration is executed during activation via the activation wrapper.
During activation, the calibration progress is reported as an integer progress bar.
On success, the calibrated parameters are stored in the module instance and can be serialized as part of the graph state (the module implements \type{save()/load()}).

\minihead{Calibration data requirements.}
The activation input is a list of stereo image pairs with an associated robot flange pose, all observing the same \emph{static} Charuco calibration target.
During collection, the Charuco board must remain fixed in the workspace and the robot should be moved between captures to obtain multiple, sufficiently different viewpoints.

Conceptually, each sample provides:
(i) a board observation in the camera frame (from Charuco detection) and
(ii) a flange pose in the world/base frame (from the robot status stream).
The hand--eye estimation step requires motion diversity: if the robot does not change its pose between captures (or changes only in a degenerate way), the relative motions are insufficient to constrain the rigid flange-to-camera transform.
Therefore, the user should collect a set of captures across distinct robot configurations (translations and rotations), while keeping the calibration target unmoved, so that the algorithm can relate changes in the robot pose to changes in the observed board pose.

In the implementation, the activation wrapper enforces only a minimum sample count (10 samples); the practical requirement is that the samples contain enough successful Charuco detections from varied viewpoints to support intrinsics, stereo, and hand--eye calibration (see \autoref{appendix:charuco} for the exact board used).

\minihead{Calibration problem statement.}
The calibration estimates:
(i) intrinsics for both cameras (matrix $K$ and distortion $D$),
(ii) stereo extrinsics (Right $\leftarrow$ Left),
(iii) the rigid hand--eye transform between the robot flange and the left camera.
This corresponds to the robot-mounted sensing configuration motivated in \autoref{subsec:ch2_analysis_camera_tradeoffs}. The Charuco board print and the exact board parameters used in the implementation are provided in \autoref{appendix:charuco}.

\minihead{Implementation approach.}
For each captured stereo pair, the module detects the Charuco board, calibrates intrinsics independently for the left and right camera (\type{calibrateCameraCharuco}), estimates stereo extrinsics with fixed intrinsics (\type{stereoCalibrate}), and computes the flange--camera transform using OpenCV hand--eye calibration (\type{calibrateHandEye}, Tsai method). Finally, a small nonlinear refinement step implemented using Ceres jointly refines the stereo and hand--eye parameters by minimizing reprojection residuals (Huber loss).
\minihead{Output and downstream usage.}
After a successful activation, every incoming camera+flange message is converted into a calibrated message by combining the current flange pose with the calibrated flange--camera transform, yielding the current world poses of the left and right camera. The image blobs are forwarded unchanged; downstream modules therefore consume the same image data, now accompanied by consistent world poses and intrinsics.

\minihead{Failure modes and limitations.}
Activation fails if the Charuco board cannot be detected reliably across enough views (too few corners/pairs) or if the captured viewpoints are degenerate. The calibration assumes a rigid robot-mounted stereo rig; if the mount is disturbed (e.g., collision), the extrinsics should be considered invalid and the module should be reactivated.

% ----------------------------
\subsection{Pen Tracking (Multi-Cam)}
\label{subsec:ch4_mod_pen_tracking}

\minihead{Role in graph.}
\emph{Pen Tracking (Multi-Cam)} (\type{pen\_tracking\_multicam\_module}) consumes the calibrated camera stream produced by the sensing pipeline (\autoref{fig:runtime_full}) and outputs
(i) a continuous pen pose stream with button states and (ii) discrete intent events used by the downstream authoring logic.
For operator feedback, it publishes visualization updates that show the current pen pose and, during interaction, a pose or trajectory preview that is removed after a fixed timeout.

\minihead{Interface-level I/O.}
Input is the \type{CalibratedCameraSet} stream (\autoref{sec:ch3_supporting_messages}), carrying per-camera intrinsics and world poses together with the corresponding image blobs.
The module publishes two pen outputs (\autoref{sec:ch3_supporting_messages}): a raw stream (pen tip pose and button states; \type{PenMessageRaw}) and an intent stream (user entered pose, trajectory, or a special action; \type{PenIntent}).
For visualization, it provides the 3D visualization response endpoint and publishes updates to show the current pen pose and recent intent previews (\autoref{sec:ch3_visualization_interface}).

\minihead{Configuration.}
The module is not activable; it starts processing immediately once loaded.
Configuration is compiled into the module (\path{pen_defaults.h}): the BLE service and characteristic UUIDs, the ArUco dictionary (\type{DICT\_4X4\_100}), the expected marker IDs (\type{\{17,38,47,56,58\}}) and their poses relative to a pen reference frame (defined at the marker cube center), the marker size (36\,mm), and a fixed transform from the pen reference frame to the pen tip
($T_{\text{pen}\leftarrow\text{tip}}$).
No per-pen calibration is currently performed; the tracking assumes the CAD-defined marker-to-pen geometry
(\autoref{subsec:ch4_hw_pen}, \autoref{appendix:pen_artifacts}).

\minihead{Marker configuration and pen geometry.}
The physical pen design and marker layout are described in \autoref{subsec:ch4_hw_pen} (see also \autoref{appendix:pen_artifacts}).
At the software level, the module assumes a fixed marker configuration: five known marker IDs with known poses in the pen reference frame, and a fixed transform from the reference frame to the tip ($T_{\text{pen}\leftarrow\text{tip}}$), all compiled into \path{pen_defaults.h}.

\minihead{Pose estimation pipeline.}
For each incoming calibrated frame, the module:

\begin{enumerate}[label=(\roman*)]
    \item unpacks camera intrinsics and per-camera poses from \type{CalibratedCameraSet} and converts image blobs to grayscale views;
    \item runs ArUco detection independently per camera with subpixel corner refinement;
    \item generates pose candidates using \type{solvePnP} (\type{IPPE\_SQUARE}) for each detected marker in each camera;
    \item scores candidates by reprojection consistency across all currently visible markers and cameras (inlier counting with a normalized reprojection threshold, plus a simple marker-shape quality score; the reprojection error is normalized by average side length in pixels to achieve scale invariance across viewing distances);
    \item optionally refines the initial estimate by a simple multi-camera depth/scale correction when the same marker is observed in multiple cameras; and finally
    \item runs a small (max 15 steps) joint nonlinear refinement in Ceres, optimizing the pen pose to minimize reprojection residuals over all observed marker corners across all cameras.
\end{enumerate}

The final pose is smoothed by a pose-specific one-euro filter. When tracking is temporarily lost, the filter holds the last output pose for a short period; after longer loss it resets its internal state and re-initializes on the next valid measurement.

\minihead{Intent extraction and events.}
The raw stream contains the current pen tip pose and the instantaneous primary/secondary button state.
Intent events are derived from button transitions:
a short primary press emits \type{POSE} (single pose),
a longer press with motion emits \type{TRAJECTORY} (a sequence of sampled poses serialized into a blob),
and a secondary press emits \type{SPECIAL\_ACTION}.

During interaction, the module visualizes the inferred intent either as
(i) an axes arrow at the selected pose (RGB axes for $x/y/z$) or
(ii) a preview of the captured trajectory.
Both visualizations are shown only temporarily and are automatically removed after a fixed timeout.

\minihead{Pen communication.}
Button state is received over BLE using SimpleBLE. The pen firmware transmits IMU samples as well, but the tracking implementation uses only the button flags (vision-only pose estimation).
On Linux, the BLE discovery/connection sequence would need minor adaptation, as the current implementation relies on Windows-style service exposure during scanning.

\minihead{Visualization outputs.}
The module registers simple geometric resources (a pen proxy model and an RGB axes arrow) and updates them each frame to show the pen body pose and the pen tip pose (RGB axes for $x/y/z$).

\minihead{Failure modes and known limitations.}
Tracking quality depends on marker visibility and stable corner detection (lighting, motion blur, partial occlusions).
When no valid pose can be estimated, the pen visualization is hidden until tracking resumes.
No IMU fusion is implemented in this module.

No per-device geometric calibration is performed; instead, the module assumes the CAD-defined marker-to-pen geometry. In the prototype, the marker plates are mechanically constrained (screw-mounted), which reduces assembly variability compared to adhesive mounting.

\subsection{Scene detection module (fiducial boxes)}
\label{subsec:ch4_mod_scene_detection}

\minihead{Role in graph.}
\emph{Scene Detection (Stereo Camera)} (\type{scene\_detection\_stereocam\_module}) consumes the calibrated stereo stream from the sensing pipeline (\autoref{fig:runtime_full}) and provides scene snapshots on request.
It is used to (i) expose the loaded object registry to downstream logic and (ii) return estimated object poses in the shared world frame.
For operator feedback, it also mirrors the most recently detected boxes into the shared 3D visualization scene.

\minihead{Interface-level I/O.}
Input is the \type{CalibratedCameraSet} stream (\autoref{sec:ch3_supporting_messages}); the implementation expects exactly two calibrated camera images (stereo pair) in each message.
The module provides the scene detection request/response endpoint (\autoref{sec:ch3_scene_detection_interface}) and handles two request types: \type{READ\_REGISTRY} and \type{READ\_SCENE}.

\minihead{Configuration and registry file.}
The module has no activation step; it loads its registry once during construction.
At runtime, the registry is read from \path{binaries/data/scene_detection_stereocam_module/registered_objects.json}.
This file is deployed by installing the module's packaged data from \path{scene_detection_stereocam_module/module_data/registered_objects.json}; the thesis attachment includes this source copy, see \autoref{appendix:scene_registry}.
If the registry file is missing or malformed, the module refuses to start.

\minihead{Scene model and registry.}
The module operates on the fiducial-box scene model from \autoref{subsec:ch4_hw_scene}.
The object registry defines the set of detectable objects and their geometry: marker ID, marker size, box dimensions, and a rigid transform between the box reference frame and the marker frame (example in \autoref{fig:ch4_scene_box}).

\minihead{Detection pipeline.}
On \type{READ\_SCENE}, the module waits for one incoming calibrated image set and runs detection using only that single stereo pair.
From this stereo pair, it:
\begin{enumerate}[label=(\roman*)]
    \item detects ArUco markers independently in the left and right images (\type{DICT\_4X4\_100});
    \item estimates an initial marker pose in the world frame using \type{solvePnP} (\type{IPPE\_SQUARE}) per detected marker and per camera;
    \item matches left/right detections by marker ID using a greedy reprojection-error selection:
    it considers only candidate pairs with the same marker ID, computes a reprojection error for each such left--right combination, then repeatedly selects the remaining candidate with the lowest reprojection error.
    The selected pair is accepted only if this minimum error is below a fixed reprojection-error threshold; otherwise matching stops.
    After accepting a pair, the corresponding left and right detections are removed and the procedure repeats until no valid candidates remain or the best remaining error exceeds the reprojection-error threshold; and
    \item refines each matched marker pair pose by a Ceres nonlinear least-squares optimization that minimizes reprojection residuals of the four marker corners in \emph{both} cameras.
\end{enumerate}

Finally, the box pose is obtained by composing the optimized marker pose with the marker-to-box transform from the registry, producing a list of \type{DetectedBox} items.

\minihead{Request/response semantics.}
\type{READ\_REGISTRY} returns the registry content that was loaded at module startup; the file is not reloaded automatically at runtime.
\type{READ\_SCENE} blocks until the next calibrated stereo frame arrives, runs one detection pass on that single frame, and returns the resulting list.
Only one \type{READ\_SCENE} request is processed at a time; concurrent requests are rejected.

\minihead{Visualization outputs.}
For each registry entry, the module registers a simple box resource with the box dimensions from the registry.
After every \type{READ\_SCENE} request, it first removes all previously displayed boxes and then adds the boxes detected in that request at their estimated world poses.
As a result, the visualization always shows the output of the \emph{most recent} scene snapshot and remains unchanged until the next \type{READ\_SCENE} request.

\minihead{Failure modes and known limitations.}
The module detects only fiducial-tagged boxes defined in the registry; it does not detect untagged objects and does not support richer geometry than boxes.
A box is returned only if its marker is detected in \emph{both} left and right images in the same stereo frame; partial detections (marker visible in only one camera) are ignored.
The implementation intentionally uses only a single stereo pair per \type{READ\_SCENE} request, without temporal stabilization (no multi-frame averaging and no tracking across frames), so pose estimates can be noisy when corner detection is unstable (lighting changes, motion blur, partial occlusion).
The response contains only poses and IDs; it does not report confidence values or covariances.

\section{Execution backend module}
\label{sec:ch4_execution_backend}

This section describes the execution stage of the pipeline (\autoref{fig:intro__img_2__usecase_stages}): a robot-side endpoint is exposed by a robot plugin (CBun), and the thesis runtime connects to it through a module that implements the shared \type{robot\_interface}.
This instantiates the robot communication requirement from \reqref{req:robot_communication_interface} while keeping use case logic robot-agnostic.

% ----------------------------
\subsection{Robot communication module}
\label{subsec:ch4_mod_robot}

\minihead{Role in graph.}
\emph{Kassow Robot Module} (\type{robot\_module\_kassow}) provides the shared \type{robot\_interface} used by the calibration module, the UI orchestrator, and the use case modules (\autoref{fig:runtime_full}). It forwards robot-control requests to CBun and publishes robot status and action-finished notifications back into the runtime.

\minihead{Interface-level I/O.}
The module provides the \type{robot\_interface} request/response endpoint and publishes the corresponding status and finished streams (\autoref{sec:ch3_robot_interface}).
Status messages contain (i) the 7 joint positions and (ii) three poses in the world frame: robot base pose, flange pose (TFC; tool flange center), and tool pose (TCP; tool center point, i.e., the pose controlled by Cartesian motions).
The world frame is the robot controller's reference frame and does not have to coincide with the robot base origin.
Finished messages report completion of a previously started action (success/failure) together with its \type{action\_id}.

\minihead{Activation and parameters.}
The module is activable (\autoref{subsec:ch3_activation_interface}). On activation it reads parameters, attempts to connect to the CBun, and starts a polling thread that forwards asynchronous status/finished messages.
The activation parameters are:
\begin{itemize}[nosep]
    \item \type{Host} (default \type{127.0.0.1}) and \type{Port} (default \type{15050}) of the CBun TCP server,
    \item \type{Request Timeout (ms)} (default 200): timeout for waiting for the response after forwarding a request to CBun,
    \item \type{Poll Interval (ms)} (default 10): polling period for receiving CBun-pushed status updates and finished notifications,
    \item \type{Reconnect Wait (ms)} (default 500) between reconnect attempts after disconnect, and
    \item \type{Robot Model} (A810/A1018/A1205/A1410/A1805) used only for visualization.
\end{itemize}

\minihead{Transport and protocol.}
The module uses a single TCP connection to the CBun for all communication.
For each incoming \type{robot\_interface} request, it sends one RPC message to the CBun and waits for the corresponding response (up to \type{Request Timeout (ms)}).
Robot status updates and action-finished notifications are sent by the CBun without a matching request; the module receives them in its polling loop (period \type{Poll Interval (ms)}) and forwards them as publish streams.
If the TCP connection is lost, the module attempts to reconnect repeatedly with delay \type{Reconnect Wait (ms)} between attempts.

\minihead{Implemented actions (robot interface coverage).}
The module implements the \type{RobotControl} feature of the shared \type{robot\_interface}.
The supported actions are: query of robot specs/limits, joint-space motion, Cartesian linear motion, arc motion, trajectory motion, and motion cancellation.

Start-motion requests typically return \type{SUCCESS\_IN\_PROGRESS} together with an \type{action\_id} assigned by CBun.
Completion is then reported through the finished stream using the same \type{action\_id}; cancellation targets the currently running motion identified by this \type{action\_id}.

\minihead{Visualization outputs.}
For operator feedback, the module mirrors robot state into the shared 3D visualization scene (\autoref{sec:ch3_visualization_interface}): a simplified robot model (selected by \type{Robot Model}), RGB axes ($x/y/z$) at the base, flange (TFC), and tool (TCP) poses, and a TCP trajectory polyline showing a fixed-length history of recent TCP positions.

\minihead{Failure modes and known limitations.}
If the module is not connected to the CBun (wrong host/port, CBun not running, or connection drop), incoming \type{robot\_interface} requests fail immediately (the request returns \type{success=false} and no robot command is executed).
If the CBun is connected but does not return a response within the configured \type{Request Timeout (ms)}, the request fails in the same way.
While disconnected, status and finished updates are not forwarded to the runtime.

Tool/gripper control is intentionally not standardized in this thesis (\autoref{subsec:ch2_analysis_robot_interface}); the current use case implementations simulate tool actions (e.g., ``open/close gripper'') by fixed delays rather than a dedicated tool-control interface.

\section{Task-oriented use case modules}
\label{sec:ch4_usecases}

These modules implement the \emph{interpretation} stage of \autoref{fig:intro__img_2__usecase_stages}:
they combine authored spatial intent (poses/trajectories) with a sensed scene snapshot to construct an executable task-specific procedure.
Each use case is implemented as a plugin module wrapped by the shared use case wrapper contract (\autoref{sec:ch3_usecase_wrapper}), which standardizes command creation (command JSON + optional preview payload) and execution control (start/stop and status).

The two primary use cases evaluated in this thesis are \emph{pick-and-place} and \emph{seam welding}. Both reuse the same sensing/authoring infrastructure (pen intents, scene detection) and execute through the common \type{robot\_interface} (\autoref{sec:ch3_robot_interface}), instantiating the extensibility goal of \goalref{goal:2} and the demonstration goal of \goalref{goal:3}.

% ----------------------------
\subsection{Pick-and-place use case}
\label{subsec:ch4_usecase_pick_place}

\minihead{Role in graph.}
\emph{Usecase Pick and Place} (\type{usecase\_pick\_and\_place}) consumes pen intent events, scene detection responses, and the shared \type{robot\_interface} in \autoref{fig:runtime_full}. It implements task logic for selecting a scene object near an authored pick pose, and then executes a fixed pick--lift--transfer--place motion sequence.

\minihead{Interface-level I/O.}
The module is wrapped by the use case wrapper (\autoref{sec:ch3_usecase_wrapper}) and exposes the wrapper request/response endpoint.
It consumes:
(i) the pen intent subscribe stream (only \type{POSE} intents are used),
(ii) the scene detection request/response endpoint (\autoref{sec:ch3_scene_detection_interface}), and
(iii) the robot interface request channel plus status/finished subscribe streams (\autoref{sec:ch3_robot_interface}).
It produces use case wrapper responses, including a command JSON payload and a preview visualization payload.

\minihead{Configuration.}
The module has no activation step. All operator-facing settings are provided through wrapper parameters when creating a command:
three \emph{auto} parameters (scene detection, pick and place positions) and four \emph{advanced} numeric parameters (speed, acceleration, pick detection distance, lift distance).

\minihead{Authoring inputs and parameters.}
\type{Pick Position} and \type{Place Position} are authored as pen \type{POSE} intents (a 6D pose in the world frame).
\type{Scene Request} triggers one \type{READ\_SCENE} call and supplies the detected boxes to command creation.
Advanced parameters are:
\begin{itemize}[nosep]
    \item \type{Movement Speed [mm/s]} (default 100) and \type{Movement Acceleration [mm/s\textsuperscript{2}]} (default 500),
    \item \type{Pick Detection Distance [mm]} (default 20): length of the search segment cast from the pick pose along its local $+z$ axis,
    \item \type{Lift Distance [mm]} (default 200).
\end{itemize}
The scene model is the fiducial-tagged box registry from \autoref{subsec:ch4_hw_scene} and \autoref{appendix:scene_registry}; box dimensions used during pick detection come from the registry snapshot.

\minihead{Command creation.}
Command creation uses the scene snapshot and the authored pick pose to select a target object:
it casts a segment from the pick pose origin in the world frame along the pick pose local $+z$ axis with length \type{Pick Detection Distance}.
For each detected box whose ID exists in the loaded registry, it tests whether the segment intersects the oriented box volume (using registry box dimensions and the detected box pose).
If multiple boxes intersect, it selects the one with the smallest distance from the segment start (first hit).

The created command stores:
\begin{itemize}[nosep]
    \item \type{box\_id}: the selected box identifier (as reported by scene detection),
    \item \type{T\_box\_pick}: the rigid transform $T_{\text{box}\leftarrow\text{pick}}$ (pick expressed in the box frame; stored as quaternion+translation),
    \item \type{place\_pose}: the authored place pose in the world frame (quaternion+translation),
    \item \type{movement\_speed\_m\_s}, \type{movement\_acceleration\_m\_s2}, \type{lift\_distance\_m}.
\end{itemize}
The command is therefore tied to an object ID and a relative pick transform, not to an absolute world pick pose.

\minihead{Preview visualization.}
Along with the command, the module returns a preview payload consisting of:
(i) two 6D poses (the authored pick and place poses),
(ii) one 3D point (the computed first intersection point on the selected box), and
(iii) one 3D polyline with four vertices representing the nominal lift and transfer path:
pick $\rightarrow$ above-pick $\rightarrow$ above-place $\rightarrow$ place.
The preview is derived from the authored inputs and chosen lift distance; it does not incorporate changes in the scene after the command is created.

\minihead{Execution.}
Execution is supported only in non-simulated mode (simulation mode is rejected by this use case).
At the start of execution, the module performs a fresh \type{READ\_SCENE} request and searches the response for a box with the stored \type{box\_id}.
If multiple boxes with the same \type{box\_id} are present in the scene response, it uses the first matching instance.

If a match is found, it reconstructs the world pick pose as:
$T_{\text{world}\leftarrow\text{pick}} = T_{\text{world}\leftarrow\text{box}} \, T_{\text{box}\leftarrow\text{pick}}$,
then converts it to a robot-control pose.

The executed motion sequence is:
\begin{enumerate}[nosep,label=(\arabic*)]
    \item move linearly to \emph{above pick} (pick with world $z$ increased by lift),
    \item move linearly to \emph{pick},
    \item simulate ``close gripper'' by a fixed 1\,s delay,
    \item move linearly back to \emph{above pick},
    \item move linearly to \emph{above place} (place with world $z$ increased by lift),
    \item move linearly to \emph{place},
    \item simulate ``open gripper'' by a fixed 1\,s delay,
    \item move linearly back to \emph{above place}.
\end{enumerate}
All robot moves are executed as linear move requests via \type{RobotWrapper}, using the movement parameters stored in the created use case command JSON (\type{movement\_speed\_m\_s} and \type{movement\_acceleration\_m\_s2}).

\minihead{Stop semantics.}
Pause is not supported in this use case module; stop is supported.
During each robot move, the module monitors stop requests while waiting for the active robot action to finish.
If stop is requested while a move is in progress, it sends a cancel request for that action, waits until the robot reports the action as no longer active, and then terminates the program by processing the stop request at the use case wrapper level.
Stop is also checked at explicit checkpoints: once before the first motion is started, and after each 1\,s gripper-delay step.

\minihead{Failure modes and known limitations.}
If the fresh scene snapshot at execution time does not contain the expected \type{box\_id}, execution fails before motion starts.
Tool control is not implemented; gripper actions are simulated by fixed delays.
No collision checking is performed; the motion sequence assumes that the lifted transfer path is free of obstacles.
Running the command in simulation mode is rejected.

\subsection{Seam welding use case}
\label{subsec:ch4_usecase_weld}

\minihead{Role in graph.}
\emph{Usecase Weld} (\type{usecase\_weld}) consumes pen intent events, scene snapshots, and the shared \type{robot\_interface} in \autoref{fig:runtime_full}. It implements task logic for interpreting an authored trajectory as a weld along a detected seam, and executes a fixed approach--weld--depart sequence.

\minihead{Interface-level I/O.}
The module is wrapped by the use case wrapper (\autoref{sec:ch3_usecase_wrapper}) and exposes the wrapper request/response endpoint.
It consumes:
(i) the pen intent subscribe stream (only \type{TRAJECTORY} intents are used),
(ii) the scene detection request/response endpoint (\autoref{sec:ch3_scene_detection_interface}), and
(iii) the robot interface request channel plus status/finished subscribe streams (\autoref{sec:ch3_robot_interface}).
It produces use case wrapper responses, including a command JSON payload and a preview visualization payload.

\minihead{Configuration.}
The module has no activation step. All operator-facing settings are provided through wrapper parameters when creating a command: two \emph{auto} parameters (scene detection and welding trajectory) and five \emph{advanced} numeric parameters (movement speed, welding speed, acceleration, weld offset, approach distance).

\minihead{Authoring inputs and parameters.}
\type{Welding Trajectory} is authored as a pen \type{TRAJECTORY} intent (a sequence of 6D poses in the world frame). \type{Scene Scan} triggers one \type{READ\_SCENE} call and supplies the detected boxes for seam detection.
Advanced parameters are:
\begin{itemize}[nosep]
    \item \type{Movement Speed [mm/s]} (default 100): speed for non-welding moves (approach, positioning, depart),
    \item \type{Welding Speed [mm/s]} (default 10): speed during the weld move,
    \item \type{Acceleration [mm/s\textsuperscript{2}]} (default 500): acceleration for all moves,
    \item \type{Weld Offset [mm]} (default 10): distance from the seam during welding,
    \item \type{Approach Distance [mm]} (default 70): distance from the seam for approach and depart poses.
\end{itemize}
The scene model is the fiducial-tagged box registry from \autoref{subsec:ch4_hw_scene} and \autoref{appendix:scene_registry}; box dimensions used by seam detection are taken from the registry snapshot.

\minihead{Seam detection.}
From the scene snapshot, the module constructs a set of \emph{seam candidates}. Each seam candidate is a line segment in the world frame (defined by its endpoints), together with a unit \emph{torch direction} that specifies from which side the tool should approach the seam.
In the concrete implementation, seams are generated by geometric tests between pairs of detected boxes using their poses and registry dimensions (edge--edge and edge--face cases).

\minihead{Trajectory-to-seam matching and projection.}
The user trajectory is matched to exactly one seam candidate using a two-stage heuristic:
(i) select a seam that is close to the user trajectory points and whose \emph{torch direction} vector is consistent with the pen's local negative $z$ direction along the trajectory, and
(ii) project the user trajectory onto the selected seam to determine a start and end portion of the seam (the command does not necessarily weld the full seam segment).
The resulting weld start/end poses use an orientation derived from the seam: the tool $x$ axis is aligned with the seam tangent and the tool $z$ axis points towards the seam (derived from the seam torch direction).

\minihead{Command creation.}
Command creation resolves the selected seam and the user trajectory into a fixed robot motion plan at command-creation time.
The created command stores four world-frame poses (quaternion+translation):
\begin{itemize}[nosep]
    \item \type{approach\_pose},
    \item \type{weld\_start\_pose},
    \item \type{weld\_end\_pose},
    \item \type{depart\_pose},
\end{itemize}
and three motion scalars:
\type{movement\_speed\_m\_s}, \type{welding\_speed\_m\_s}, \type{acceleration\_m\_s2}.
The advanced parameters \type{Weld Offset} and \type{Approach Distance} are applied during command creation by shifting the approach/weld/depart positions away from the seam along the local tool axis by the requested distances.

Unlike the pick-and-place use case (\autoref{subsec:ch4_usecase_pick_place}), this use case resolves the seam during command creation and stores a fixed set of world-frame poses to execute (approach, weld start/end, depart).
The seam representation depends on the current relative poses of the detected boxes, and this prototype does not implement a stable seam identifier that could be matched again if the scene changes.

If no seam candidates are detected from the scene snapshot, or if the user trajectory cannot be matched to any candidate seam, command creation fails.

\minihead{Preview visualization.}
Along with the command, the module returns a preview payload consisting of:
(i) one 6D pose (the computed weld start pose) and
(ii) one 3D polyline with four vertices representing the nominal path:
approach $\rightarrow$ weld start $\rightarrow$ weld end $\rightarrow$ depart.
The preview visualizes the stored command poses and therefore corresponds directly to the motion that will be executed.

\minihead{Execution.}
Execution is supported only in non-simulated mode (simulation mode is rejected by this use case).
The module executes the stored poses directly and does not consult scene detection during program execution; the command therefore assumes that the workpiece and fixture remain in the same configuration as during command creation.
The executed sequence is:
\begin{enumerate}[nosep,label=(\arabic*)]
    \item move linearly to \emph{approach} at \type{movement\_speed\_m\_s},
    \item move linearly to \emph{weld start} at \type{movement\_speed\_m\_s},
    \item simulate ``start welding'' by a fixed 1\,s delay,
    \item move linearly to \emph{weld end} at \type{welding\_speed\_m\_s},
    \item simulate ``stop welding'' by a fixed 1\,s delay,
    \item move linearly to \emph{depart} at \type{movement\_speed\_m\_s}.
\end{enumerate}
All moves are issued as linear robot-control requests via \type{RobotWrapper} and are monitored via the finished stream.

\minihead{Pause and stop semantics.}
Pause and stop are supported. During each robot move, the module monitors pause/stop requests while waiting for the active robot action to finish. If pause or stop is requested, it cancels the current robot action and waits until the robot reports that the action is no longer active.
On pause, the module enters the \type{BaseUsecase} pause handling (wait for resume). For the weld move, pause/resume additionally simulates welding state changes: on pause it simulates ``stop welding'', moves the tool to a fixed offset away from the current pose, and on resume it returns to the paused pose and simulates ``start welding'' before continuing.

\minihead{Failure modes and known limitations.}
The command is resolved into world-frame poses at command-creation time; execution does not re-associate the command to an updated scene, and it therefore assumes that the workpiece and fixture remain in the same configuration as during command creation.
Seam detection and trajectory matching are implemented as geometric heuristics with tunable thresholds, validated on the prototype scene objects.
Tool control is not implemented; welding on/off is simulated by fixed delays.
No collision checking is performed; the commanded approach/weld/depart path assumes a clear workspace.
Running the command in simulation mode is rejected.
