\chapter{Hardware Setup and Module Implementations}
\label{chapter:ch4_implementation}

This chapter describes the concrete prototype used in this thesis: the physical setup and the internal implementation of the modules that instantiate the end-to-end configuration introduced in \autoref{fig:runtime_full}.
Unlike \autoref{chapter:ch3_system_design}, which focused on contracts and interface semantics, this chapter focuses on the concrete implementation decisions inside the modules.

\minihead{Accompanying electronic attachment.}
The submitted electronic attachment contains (i) a pruned snapshot of the C++ source tree (\path{diploma-thesis-code/}) and (ii) the hardware and calibration artifacts required to reproduce the prototype setup (CAD, firmware, PDFs, configuration JSON).
A complete overview of the attachment structure, build notes, and artifact file locations is provided in \autoref{appendix:source_code} and \autoref{appendix:artifacts}.

\minihead{Implementation note.}
The system is implemented in C++ and targets Windows and Linux (build workflow and toolchain assumptions are summarized in \autoref{appendix:source_code}).
Most modules are platform-neutral; platform-specific code is currently concentrated in the Windows camera acquisition module (\autoref{subsec:ch4_mod_stereo_camera}).
The web frontend is based on Wt and was used on both Windows and Linux in development.

The pen tracking module utilizes a cross-platform BLE library (SimpleBLE); however, its current device discovery implementation relies on Windows-specific stack behavior where target service UUIDs are successfully exposed during the scanning phase. Running the BLE portion of this module on Linux would require adapting the discovery sequence, as the Linux Bluetooth stack typically requires an explicit connection before it populates the service list.

\minihead{Reading note.}
The UI module is described in Chapter~\ref{chapter:ch3_system_design} at the level of externally observable orchestration behavior (\autoref{sec:ch3_ui_orchestration}) and is not decomposed further here.
This chapter instead focuses on the sensing, authoring, execution, and use-case modules that produce and consume the interface messages defined in Chapter~\ref{chapter:ch3_system_design}.

\minihead{Structure of this chapter.}
We first summarize the physical setup and calibration artifacts, then describe the module implementations in the same logical order as the runtime pipeline shown in \autoref{fig:runtime_full}.

This section summarizes the physical components required to run the system and the calibration artifacts that the operator must provide.
The operational steps for setup and execution are deferred to \autoref{chapter:ch5_user_workflow}; the concrete artifact files shipped with the thesis attachment are listed in \autoref{appendix:artifacts}.

\subsection{Robot platform and execution backend}
\label{subsec:ch4_hw_robot}

The target robot platform is a Kassow collaborative robot.
During development, the prototype was tested primarily in the REMII offline environment, and validated on a physical robot of model KR1205.
As motivated in \autoref{subsec:ch2_analysis_robot_interface}, robot execution is abstracted behind the shared \type{robot\_interface} contract, implemented by the thesis-side module \type{robot\_module\_kassow}.

The prototype assumes a robot-mounted stereo camera rig: the camera is mechanically attached to the robot flange such that its pose is rigid with respect to the flange.
This enables the calibration module to combine robot kinematics (flange pose) with camera observations to place both cameras in a common workspace frame, as discussed in the camera trade-offs in \autoref{subsec:ch2_analysis_camera_tradeoffs}.
The mechanical mount used in the prototype is provided as a CAD artifact in \autoref{appendix:camera_mount_cad} and shown in \autoref{fig:ch4_camera_mount}.

Robot communication is realized through a small robot-side component (CBun) installed on the Kassow controller.
The CBun exposes the robot capabilities to the thesis runtime; the thesis-side \type{robot\_module\_kassow} consumes this endpoint and provides the \type{robot\_interface} protocol to the rest of the runtime.
CBun sources and build notes are provided in \autoref{appendix:cbun}; the operator procedure for installation/enabling is described in \autoref{chapter:ch5_user_workflow}.

\subsection{Stereo camera hardware}
\label{subsec:ch4_hw_camera}

The prototype uses a compact stereo camera module with two synchronized global-shutter sensors (OmniVision OV9281), providing $1280 \times 800$ grayscale images per sensor.
In the prototype configuration, the camera was operated as a single side-by-side stream ($2560 \times 800$) with MJPEG compression at 60\,FPS.

\subsection{Pen input device}
\label{subsec:ch4_hw_pen}

As motivated in \autoref{subsec:ch2_analysis_6D_input_interface}, spatial authoring in the prototype is driven by a hand-held 6D input device.
The prototype uses a custom tracked pen inspired by D-Point, extended for the needs of robot-mounted stereo viewing: a larger marker cube with five marker faces (four sides and a top marker) and two physical buttons for discrete intent events.

\begin{figure}[t]
    \centering
    \includegraphics[width=0.8\linewidth]{img/ch4_implementation/pen_full_bg.png}
    \caption[Tracked pen]{Tracked pen used for spatial authoring (CAD render). The cube carries multiple fiducial markers; buttons on the handle provide discrete intent events.}
    \label{fig:ch4_pen_full}
\end{figure}

The prototype pen design is shown in \autoref{fig:ch4_pen_full}.
Electronics are based on a \type{Seeed Studio XIAO nRF52840 Sense} with a single-cell Li-ion battery (\type{INR10440}), providing BLE connectivity and button reporting.
Although the pen firmware transmits IMU samples, the current tracking module does not fuse IMU data; pose estimation is performed purely from camera observations.
The exact CAD, firmware, and assembly notes (including wiring and screw lengths) are provided in \autoref{appendix:pen_artifacts}, including the license attribution boundary to the upstream D-Point firmware.

\subsection{Workspace scene objects (fiducial boxes)}
\label{subsec:ch4_hw_scene}

To support the scene detection requirements in \autoref{subsec:ch2_analysis_scene_detection}, the prototype scene model consists of simple fiducial-tagged box objects.
Each object carries a single ArUco marker and is described by a registry entry (marker ID and size, box dimensions, and marker-to-box transform).
The registry and an example box CAD model are shipped with the attachment (\autoref{appendix:scene_registry}, \autoref{appendix:scene_cad}).

    \label{fig:ch4_scene_box}
\end{figure}

An example box object used in the prototype is illustrated in \autoref{fig:ch4_scene_box}.

\subsection{Calibration artifacts}
\label{subsec:ch4_hw_calibration_artifacts}

The sensing pipeline relies on a printed Charuco board as a calibration target for intrinsics, stereo calibration, and hand--eye calibration.
The exact calibration PDF and the compile-time board dimensions used by the calibration module are provided in \autoref{appendix:charuco}; the operator workflow for calibration is described in \autoref{chapter:ch5_user_workflow}.

\subsection{Compute platform}
\label{subsec:ch4_hw_compute}

The prototype is intended to run on a standard PC with a modern C++ toolchain.
In development, the system was built and tested on both a laptop-class machine (Ryzen~9~5980HS, 32\,GB RAM) and a desktop-class machine (Ryzen~9800X3D, 64\,GB RAM).
These are representative development machines rather than hard requirements; the system does not rely on specialized compute hardware.
