\chapter*{Introduction}
\addcontentsline{toc}{chapter}{Introduction}

Collaborative robots are attractive for automating tasks that are repetitive but frequently changing, such as small batch production or tasks requiring frequent changeovers. In such cases, the bottleneck is often not the execution time itself, but the time and effort required to re-create robot programs for each new task.

Robot programming is both time-consuming \cite{IFR2023TraditionalVsNew} and skill-intensive. This creates friction whenever tasks need to be re-taught. This thesis explores whether this friction can be reduced by changing how users specify robot tasks.

\section*{Motivation}
\addcontentsline{toc}{section}{Motivation}

Industrial robots are commonly programmed using combinations of the following approaches \cite{VILLANI201866}:
\begin{itemize}
    \item \emph{Offline programming}: The user creates the robot program in a simulated environment, often based on a CAD model of the workspace. This approach allows programming without the physical presence of the robot, therefore avoiding downtime and allowing testing in a simulated environment. However, it requires accurate models of the robot and the environment, and it can be complex to set up and use effectively.
    \item \emph{Teach pendant programming}: The user enters robot motions and commands step-by-step using a handheld device (the teach pendant). This can make teaching poses more intuitive, since you can see the robot's movements in the real world, but programming still involves iterating on trajectories, tool actions, and program structure, which can be time-consuming and typically requires physical presence at the robot.
\end{itemize}

Both of these methods are widely used and capable of producing effective robot programs, but they can become friction-heavy when tasks change frequently. Robot programs usually consist of a sequence of move commands interleaved with tool actions and logic. This brings two main sources of friction:
\begin{itemize}
    \item \emph{Pose definition is tedious}: Specifying precise poses and trajectories through a teach pendant or offline programming can be time-consuming and unintuitive, especially for users without extensive programming experience. This can lead to long iteration loops of editing the program, testing it on the robot, and adjusting it until the desired behavior is achieved.
    \item \emph{Expertise and setup overhead}: Both teach pendant and offline programming require a certain level of expertise in robot programming concepts, and offline programming also requires accurate models and complex setup. This can be a significant barrier for users who are not familiar with programming concepts, and it can lead to reliance on trained personnel or system integrators for programming and deployment.
\end{itemize}

Collaborative robots often partially mitigate the pose definition friction with \emph{hand-guiding} \footnote{Terminology varies. Here, \emph{hand-guiding} means compliant back-driving to teach poses; \emph{lead-through} refers to recording a continuous path for playback.}, where the user can physically back-drive the robot to teach poses \cite{KassowRobotsManuals}. However, hand-guiding may not be suitable for all tasks and it does not solve the expertise and setup overhead.

These limitations motivate the research question of this thesis: can robot programs for frequently changing tasks be authored more efficiently by specifying spatial intent directly in the real workspace, rather than by editing motion commands and tool actions on a pendant or in a simulated environment? We call this approach \emph{in-workspace spatial authoring}, and the next section introduces it at a high level.


\section*{High-level approach}
\addcontentsline{toc}{section}{High-level approach}

We propose \emph{spatial authoring} of cooperative robot motions and tool actions directly in the real workspace using AR-inspired inputs, supported by external sensing. While general-purpose robot programming is expressive, it also exposes many low-level details (motion primitives, tool coordination, error handling) that dominate effort in small programs for frequently changing tasks. Therefore, this thesis focuses on task-oriented \emph{use cases} for common task families. In this context, a use case denotes a self-contained task-specific application that defines how spatial user input and sensed workspace context are interpreted and expanded into robot motion and tool commands.

This approach has four conceptual stages, illustrated in \autoref{fig:intro__img_2__usecase_stages}:
\begin{enumerate*}[label=(\roman*)]
    \item \emph{workspace sensing} to estimate the relevant workspace context (e.g., reference frames and task-relevant objects);
    \item \emph{authoring}, where the user specifies spatial intent in the workspace;
    \item \emph{interpretation}, where the task-oriented logic combines intent with scene context to generate motion and tool actions; and
    \item \emph{execution}, where the resulting steps are run on the robot via a common robot execution interface.
\end{enumerate*}

  \centering

  \caption{Conceptual pipeline of in-workspace spatial authoring on a pick-and-place example; Axes indicate the current tool-center-point (TCP) or pose variables.}
  \label{fig:intro__img_2__usecase_stages}
\end{figure}

We target task-oriented use cases rather than a full robot programming language; our approach does not cover complex branching, loops, exception recovery, or rich sensor-driven logic. This allows us to focus on the core question of whether spatial intent input can reduce friction for frequently changing tasks and simplify the programming process for non-expert users.

To support this, we develop a system that provides workspace sensing and authoring capabilities, together with a shared robot/tool execution layer. Task-specific use cases can then be developed that use these capabilities to implement the interpretation logic for specific tasks. This allows us to evaluate the approach on representative task families while keeping the core system reusable and extensible for other tasks in the future.

We evaluate the approach on two representative task families that stress different requirements on spatial input and program structure. They correspond to the two largest application categories reported in industrial robot installation statistics (handling and welding) \cite{IFRWorldRobotics2023Extended}:
\begin{itemize}
    \item \emph{pick-and-place} as a representative material handling task. It combines scene understanding (object detection and pose estimation) with discrete pose specification (target pose for the object) and tool actions (gripper open/close).
    
    \item \emph{seam welding}: a trajectory-centric task that can require frequent re-teaching due to changing workpieces or fixtures. This allows us to explore how spatial authoring can speed up low volume production and process tasks that require frequent changeovers, which are a relevant application area for collaborative robots.
\end{itemize}


\section*{Goals}
\addcontentsline{toc}{section}{Goals}

Based on the high-level approach described above, we define three goals:
\begin{goals}
  \item \label{goal:1} Explore and validate programming of cooperative robots directly in the real workspace by defining robot motions and tool actions using spatial, AR-inspired inputs, as an alternative to traditional teach pendant, offline programming, and hand-guiding workflows.
  \item \label{goal:2} Provide an extensible system where task-oriented use cases can be implemented as plugins that reuse the same acquisition/execution infrastructure.
  \item \label{goal:3} Demonstrate the approach on a real robot with representative use cases for a material-handling task (pick-and-place) and a trajectory-centric task (seam welding), and evaluate the benefits and limitations of spatial authoring for these tasks.
\end{goals}
